# Jastr Engine and CLI Split V2 Spec

## Intended Outcome

This spec turns the current single-package CLI into `jastr`: a template-centered
Markdown rendering tool with a reusable TypeScript engine and a CLI adapter. The
implemented system gives users three concrete capabilities:

- `jastr run <template-ref> [input flags...]` renders a Markdown template from
  either a named project template or a direct `.md` file path, then prints the
  final Markdown to stdout.
- `jastr generate agent-skill <template-ref> --out <path> [--force]` writes a
  minimal Agent Skill wrapper that invokes `jastr run` and tells the agent to
  follow the rendered output.
- TypeScript callers can consume `@jastr/engine` directly without importing CLI
  syntax, Commander wiring, process exit handling, filesystem discovery, Agent
  Skill concepts, or Bun runtime APIs.

The central domain model is `Template`, not `Skill`; engine API names, types,
and docs must use template vocabulary unless they are specifically inside an
Agent Skill target generator (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P15 and P24). The implementation physically creates workspace packages named
`@jastr/engine` and `@jastr/cli`; older discussion choices about a core/CLI
package split are superseded by the later naming and package decisions (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P7, P17, P27, and P33).

## Architecture And Package Ownership

The repository root becomes a private workspace orchestrator. It owns shared
development tooling and aggregate scripts, but it no longer owns the product
binary or runtime source. The root package must not expose a `bin` entry for
`jastr`.

`packages/engine` is the `@jastr/engine` package. It owns the template engine:
frontmatter parsing, template schema validation, domain input validation,
directive parsing, condition evaluation, interpolation, include orchestration
through injected resolvers, and rendering from explicit source plus explicit
inputs. It must expose a small intentional top-level API and block deep imports
through package `exports` (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P8).

The engine must accept domain-shaped inputs such as
`Record<string, string | boolean>` rather than CLI-shaped flag tokens. CLI
flag parsing may keep a local intermediate representation, but no `RawFlag`,
`CliInputToken`, Commander option object, or `--flag` syntax type may cross into
`@jastr/engine` (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P1 and P3).

`packages/cli` is the `@jastr/cli` package. It owns the `jastr` binary,
Commander program wiring, process entrypoint, command-line argument parsing,
project root discovery, Node filesystem readers and writers, artifact target
commands, stdout/stderr formatting, exit codes, and version output. The CLI may
call process-free engine use cases, but command helpers such as run/generate
wrappers remain CLI adapter code rather than engine exports (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P4).

Template lookup/catalog logic belongs above the engine. Named lookup resolves a
template id to a conventional file path, and direct file lookup reads a specific
file path. Neither lookup mode belongs in the renderer itself, because the
engine must not know about `.jastr`, project roots, cwd, or filesystem layout
(per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P2, P16, P19, P21, and P28).

Artifact target code is separate from the engine. The initial Agent Skill
target may live inside `@jastr/cli` or another repo-internal target module, but
Agent Skill names, `SKILL.md`, Agent Skill frontmatter rules, and wrapper text
must not appear in `@jastr/engine` public vocabulary (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P16, P17, and P24).

## Template Contract

A template is Markdown source with optional YAML frontmatter delimited by `---`.
If frontmatter is present, it must parse to a mapping; empty frontmatter is
treated as an empty mapping. The renderer uses only the recognized root keys
`inputs` and `targets`; unknown root keys are ignored. Recognized structures are
strict: if `inputs` or `targets` is present, malformed data inside those keys
is an error rather than ignored (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P26 and P30).

`inputs` is an optional mapping from input name to input definition. Input names
use the existing lowercase kebab-case convention for input flags and
interpolation names: `^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$`. Supported input types
remain `string`, `boolean`, and `enum`. Every input definition must explicitly
declare `required: true` or `required: false`. Enum inputs must declare a
non-empty string `values` array. Domain validation belongs in the engine, while
CLI wording belongs in the CLI adapter (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P3).

`targets` is an optional mapping of target-specific metadata. This spec
recognizes only `targets.skill`; any other target key is rejected, including
`targets.typescript` (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P30 and P32). The CLI command target is still named `agent-skill`; the
frontmatter metadata key is `targets.skill`.

`targets.skill` is used only by `jastr generate agent-skill`. Rendering a
template with `jastr run` does not require Agent Skill metadata. The initial
`targets.skill` schema is:

```yaml
targets:
  skill:
    name: review-code
    description: Review code with the rendered Jastr template output.
    frontmatter:
      allowed-tools: Read, Grep
      metadata:
        owner: platform
```

`targets.skill.name` and `targets.skill.description` are required for Agent
Skill generation. `name` must satisfy Agent Skill name constraints: 1-64
lowercase letters, numbers, and hyphens, with no leading, trailing, or
consecutive hyphens. `description` must be non-empty and at most 1024
characters. Unknown keys directly inside `targets.skill` are rejected.

`targets.skill.frontmatter` is optional extra Agent Skill frontmatter. If
present, it must be a mapping. The generated Agent Skill frontmatter always
gets `name` and `description` from `targets.skill.name` and
`targets.skill.description`; `targets.skill.frontmatter` may not contain
`name`, `description`, or `inputs`.

The initial Agent Skill passthrough policy is:

- `license`, if present, must be a string.
- `allowed-tools`, if present, must be a string.
- `compatibility`, if present, must be a non-empty string of at most 500
  characters.
- `metadata`, if present, must be a mapping whose values are strings.
- Unknown top-level frontmatter fields are allowed only when the field name is
  kebab-case: `^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$`.
- Unknown top-level frontmatter field values may be any YAML-serializable value.

Invalid `targets.skill.frontmatter` data is a target metadata validation error,
not a rendering error.

Template ids are distinct from Agent Skill names. A named template id matches
`^[a-z0-9][a-z0-9-]*$` and maps to `.jastr/<template-id>/TEMPLATE.md` (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P21, P28, and P29). `targets.skill.name` may differ from the template id because
Agent Skill metadata is target-specific and does not define template identity.

## Run Behavior

`jastr run` accepts exactly one template reference followed by template input
flags:

```text
jastr run <template-ref> [input flags...]
```

Template-reference disambiguation is syntactic, not filesystem-dependent. If
`<template-ref>` ends in `.md`, the CLI treats it as a direct Markdown template
file path. If it matches the template id grammar, the CLI treats it as a named
template id. If it is neither a `.md` path nor a valid template id, the command
fails with `invalid_template_reference`; the CLI must not decide based on
whether a file happens to exist (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P20 and P29).

For named lookup, the CLI discovers the project root by walking upward from
`cwd` until it finds a `.jastr/` directory, then reads
`.jastr/<template-id>/TEMPLATE.md` from that root (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P21 and P28). Named-template includes are resolved relative to the including
file and must remain inside the discovered project root.

For direct file mode, the CLI reads the exact `.md` file path and does not
require a `.jastr/` project root. Direct-file includes are resolved relative to
the including file and must remain inside the directory tree rooted at the
direct template file's containing directory. The CLI must not widen direct-file
include access to `cwd` or to an optional discovered project root (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P22 and P31).

Include safety rules otherwise remain conservative: absolute paths,
home-relative `~` paths, empty paths, `.env` targets, and escapes outside the
applicable containment root are rejected. `include` renders the referenced
fragment with the current input context; `include-raw` inserts the referenced
file literally.

The CLI keeps ownership of argv syntax. It parses input flags after the
template reference, loads the template schema, translates valid flags into
typed domain input values, and calls the engine. Duplicate flags, unknown input
flags, missing required values, invalid boolean values, and unsupported flag
syntax remain CLI adapter errors with CLI-shaped wording. The engine must not
format messages around `--name`, and the CLI must not parse engine error text
to recover structured meaning (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P3 and P14).

Successful `jastr run` writes rendered Markdown to stdout, writes nothing to
stderr, and exits 0. Failed commands print exactly one `Error: <message>` line
to stderr and exit 1. `--help`, `help [command]`, and `--version` are the only
standard zero-exit informational paths.

## Generate Behavior

`generate` always names an explicit target:

```text
jastr generate agent-skill <template-ref> --out <path> [--force]
```

The CLI rejects implicit generation forms such as `jastr generate review --out
out/SKILL.md`, and it rejects unsupported target names such as `typescript`
until a later spec defines and implements them (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P23 and P32).

`generate agent-skill` accepts the same named-template and direct-file
`<template-ref>` forms as `run`. It loads and validates the template enough to
ensure the source is a valid Jastr template and has valid `targets.skill`
metadata. It writes an Agent Skill `SKILL.md` to the explicit `--out` path,
creates parent directories as needed, and refuses to overwrite an existing file
unless `--force` is set.

The generated Agent Skill frontmatter comes from `targets.skill`. The generated
body is a minimal wrapper that tells the agent to run:

```bash
jastr run <template-ref> $ARGUMENTS
```

and then follow the rendered output exactly. For direct-file references, the
wrapper uses the reference supplied to `generate`; callers are responsible for
choosing a path that will be valid from the agent's execution context.

## Engine API Expectations

`@jastr/engine` must expose a small top-level contract sufficient for the CLI
and TypeScript consumers without deep imports. V2 pins the public vocabulary
because package `exports` make top-level names part of the handoff contract.

The `@jastr/engine` top-level export surface must include exactly these
callable exports:

- `parseTemplateSource(source: string): ParsedTemplate`
- `validateTemplateSchema(frontmatter: unknown): TemplateSchema`
- `validateTemplateInputs(schema: TemplateSchema, inputs: TemplateInputValues):
  TemplateInputValues`
- `renderTemplateSource(options: RenderTemplateSourceOptions):
  Promise<RenderTemplateSourceResult>`

The same top-level export surface must include these public types/classes:

- `Template`
- `ParsedTemplate`
- `TemplateSchema`
- `TemplateInputDefinition`
- `TemplateInputValues`
- `TemplateTargets`
- `RenderTemplateSourceOptions`
- `RenderTemplateSourceResult`
- `IncludeResolver`
- `IncludeRequest`
- `IncludeResolution`
- `JastrError`
- `JastrErrorCode`
- `JastrErrorDetails`

The engine may expose only additional supporting types that are necessary to use
the callable exports above. It must not export command helpers, CLI formatting
helpers, filesystem project lookup helpers, package entrypoint helpers, or
artifact target helpers.

`TemplateInputValues` is `Record<string, string | boolean>`. The engine accepts
domain-shaped inputs only; no CLI flag token, Commander option object, or
`--flag` syntax type may appear in the public engine API.

`renderTemplateSource` renders direct template source from explicit source text,
explicit domain inputs, and an injected include resolver. It returns structured
success data instead of writing to stdout. The result must include the rendered
Markdown as `markdown` and the validated template schema as `schema`.

`JastrError` is the shared structured error class. It has a stable `code`, a
human-readable neutral `message`, and optional typed `details`. Engine functions
throw or return `JastrError` instances for expected failures; CLI adapters must
format user-facing messages from `code` plus `details` when adapter-specific
wording is needed, and must not parse error message strings (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P8 and P14).

New public API names, classes, types, generated wrapper text, CLI messages, and
new documentation must use Jastr/template vocabulary. Legacy product vocabulary
is allowed only when immutable historical artifacts or absolute repository
paths must be cited.

Direct source rendering is first-class. A caller with template source in memory
must be able to render without a project root, conventional path, Node
filesystem implementation, or CLI command invocation (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P22).

The engine may define interfaces for include resolution, but the Node
filesystem implementation belongs outside the engine. In-memory tests and
library consumers must be able to supply their own include resolver (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P2).

The engine can parse and preserve `targets` metadata as part of template
schema data, but Agent Skill target generation, Agent Skill wrapper text, and
target-specific validation of `targets.skill.frontmatter` live outside the
engine package.

## Error Code And Message Contract

Every CLI failure prints exactly one `Error: <message>` line to stderr and exits
1. `--help`, `help [command]`, and `--version` are the only standard zero-exit
informational paths.

The implementation must define stable error codes and message templates for new
or changed Jastr surfaces. Existing directive, input, condition, interpolation,
and generic rendering failures remain pinned by the migrated e2e and living-doc
requirements unless this spec explicitly changes them.

The new or changed CLI-facing failure surfaces use these codes and message
templates after the `Error: ` prefix:

- `invalid_template_reference`: `Template reference <template-ref> must be a
  template id or a .md file path.`
- `missing_project_root`: `No .jastr directory found from the current
  directory.`
- `template_not_found`: `Template <template-id> was not found at
  .jastr/<template-id>/TEMPLATE.md.`
- `unsupported_generate_target`: `Unsupported generate target <target>.`
- `missing_target_metadata`: `Template must declare targets.skill metadata for
  generate agent-skill.`
- `invalid_target_metadata`: `Target metadata must be a mapping.`
- `invalid_target_metadata`: `Unsupported target metadata <target>.`
- `invalid_target_metadata`: `targets.skill must be a mapping.`
- `invalid_target_metadata`: `targets.skill.name is required and must be a
  string.`
- `invalid_target_metadata`: `targets.skill.name must be 1-64 lowercase
  letters, numbers, and hyphens with no leading, trailing, or consecutive
  hyphens.`
- `invalid_target_metadata`: `targets.skill.description is required and must be
  a string.`
- `invalid_target_metadata`: `targets.skill.description must be 1-1024
  characters.`
- `invalid_target_metadata`: `Unknown targets.skill field <field>.`
- `invalid_target_metadata`: `targets.skill.frontmatter must be a mapping.`
- `invalid_target_metadata`: `targets.skill.frontmatter must not declare
  <field>.`
- `invalid_target_metadata`: `targets.skill.frontmatter.license must be a
  string.`
- `invalid_target_metadata`: `targets.skill.frontmatter.allowed-tools must be a
  string.`
- `invalid_target_metadata`: `targets.skill.frontmatter.compatibility must be
  1-500 characters.`
- `invalid_target_metadata`: `targets.skill.frontmatter.metadata must be a
  mapping.`
- `invalid_target_metadata`: `targets.skill.frontmatter.metadata field <field>
  must be a string.`
- `invalid_target_metadata`: `targets.skill.frontmatter field <field> must be
  kebab-case.`
- `output_exists`: `Output file <out> already exists. Use --force to overwrite
  it.`
- `include_path_rejected`: either `Include path <include-path> must be
  relative.` or `Include path <include-path> is rejected.`
- `include_outside_root`: for named templates, `Include path <include-path>
  escapes the project root.`; for direct-file templates, `Include path
  <include-path> escapes the template directory.`
- `include_not_found`: `Include file <include-path> was not found.`
- `include_read_error`: `Include file <include-path> could not be read:
  <cause>.`

Engine functions must not emit CLI-only codes such as
`invalid_template_reference`, `missing_project_root`, `template_not_found`,
`unsupported_generate_target`, or `output_exists`. The CLI adapter owns those
codes and may use `JastrError` for consistent formatting.

## Build, Runtime, And Dependency Constraints

Bun remains allowed for package management and build tooling, but neither
`@jastr/engine` nor `@jastr/cli` may require Bun at runtime or expose Bun types
or APIs through public package boundaries (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P5).

Each workspace package declares its own direct runtime dependencies. The engine
owns parser/renderer dependencies such as `yaml`, `remark-directive`,
`remark-parse`, and `unified` if they remain part of engine runtime behavior.
The CLI owns Commander dependencies and depends on `@jastr/engine`. The root
owns shared dev tooling such as TypeScript, Vitest, Biome, tsx, and test harness
dependencies that are not runtime dependencies of a package (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`

Each package builds to its own `dist/`. `@jastr/engine` emits JavaScript and
`.d.ts` files referenced by its package `exports`. `@jastr/cli` emits a
Node-targeted binary and/or JS entrypoint for the `jastr` command. The CLI may
bundle the engine into its executable, but the engine package must still build
as a standalone package contract (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`

`jastr --version` reports the `@jastr/cli` package version plus the existing
git short SHA or dev marker behavior. It must not report the root workspace
orchestrator version (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260605091319Z-core-cli-package-split/discussions/2026-06-05-core-cli-package-split-design-discussion.md`
P13, adapted to the later `@jastr/cli` package name from P27 and P33).
Build-time version metadata must use Jastr vocabulary, for example a
`JASTR_GIT_SHA` define or an equivalent package-local mechanism. New runtime
code, public package boundaries, generated output, and CLI messages must not
expose legacy product-name globals or identifiers.

`packages/engine` and `packages/cli` must each expose a package-local `build`
script. The root `bun run build` command must aggregate both package builds and
is the required package build verification command.

## Unresolved Questions

- Legal trademark clearance and final package publication policy for `jastr`
  remain outside this spec. The name is settled for implementation based on the
  discussion, but this spec does not claim legal availability.
- The future TypeScript generation target needs its own spec before
  `generate typescript` or `targets.typescript` can be accepted.
- A broad root aggregate command such as `bun run verify` can be added later,
  but this spec only requires explicit package-aware root scripts.
- Publication access, npm provenance, release automation, and whether packages
  remain private through the first split are implementation/release management
  decisions, not blockers for the repository migration.

## Acceptance Guidance

A reviewer should consider the implementation complete when all of the
following are true:

- The root `package.json` is a private workspace orchestrator, `packages/engine`
  is named `@jastr/engine`, `packages/cli` is named `@jastr/cli`, and
  `@jastr/cli` owns the `jastr` binary.
- `@jastr/engine` has no runtime imports from `node:fs`, Commander, CLI argv
  parsers, process entrypoints, Bun APIs, `.jastr` lookup code, or Agent Skill
  target code.
- `@jastr/cli` is the only package that owns Commander wiring, cwd/project
  discovery, Node filesystem readers/writers, stdout/stderr formatting, process
  exit behavior, and `jastr --version`.
- Package `exports` prevent CLI and tests from relying on engine deep imports.
- Direct runtime dependencies are declared in the package that imports them,
  and root dependencies are limited to shared dev tooling unless a root script
  directly needs them.
- `@jastr/engine` exposes the pinned top-level exports from this spec, including
  `renderTemplateSource`, `validateTemplateSchema`, `validateTemplateInputs`,
  `parseTemplateSource`, and `JastrError`; it does not expose command helpers,
  CLI formatting helpers, filesystem lookup helpers, or artifact target helpers.
- Engine tests cover direct source rendering, domain input validation,
  recognized/unknown frontmatter behavior, include resolver behavior, structured
  `JastrError` details, and the absence of CLI-shaped input types from public
  APIs.
- CLI e2e cases cover named `jastr run`, direct-file `jastr run`, syntactic
  reference disambiguation, named include containment, direct-file include
  containment, input flag translation, `generate agent-skill`, overwrite
  refusal and `--force`, unsupported generate targets, `--help`, and
  `--version`.
- CLI e2e cases or equivalent focused tests cover the stable error codes and
  message templates for new or changed Jastr failure surfaces listed in this
  spec.
- Agent Skill target tests cover valid and invalid `targets.skill.frontmatter`
  passthrough data, including known fields, kebab-case extension fields,
  rejected `name`/`description`/`inputs`, invalid known-field types, and metadata
  values that are not strings.
- `packages/cli/docs/BEHAVIOR.md` is generated from CLI-owned requirements and
  e2e cases, and the generated document reflects `jastr` commands and `.jastr`
  fixtures.
- `README.md` and `AGENTS.md` no longer describe legacy product names, legacy
  template paths, root behavior-doc ownership, root e2e ownership, or old
  verification commands as current facts.
- The clean verification set exits 0: `bun run check`, `bun run typecheck`,
  `bun run test`, `bun run test:cli:e2e`, `bun run docs:cli:living --check`,
  and `bun run build`.
