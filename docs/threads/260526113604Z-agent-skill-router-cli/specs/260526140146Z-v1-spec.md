# Skillrouter V1 Spec

## Intended Outcome

Skillrouter v1 is a CLI for deterministic AI-agent skill specialization. When a
user invokes a tiny agent-facing router skill, the agent runs Skillrouter with
explicit arguments and receives only the final Markdown instructions relevant to
that invocation.

V1 produces two user-facing capabilities:

- `skillrouter run <skill> [input flags...]` reads
  `.skillrouter/<skill>/SKILL.template.md`, validates the provided input flags,
  evaluates the supported template syntax, resolves selected includes,
  interpolates declared inputs, and prints clean Markdown to stdout.
- `skillrouter generate <skill> --out <path> [--force]` writes a minimal
  agent-facing `SKILL.md` router file at an explicit destination, so users do not
  have to hand-write the small wrapper skill.

Successful `run` output is intended for an AI agent to follow directly. Failed
commands print a concise human-readable error that the agent can relay to the
user and then stop.

The motivating problem is that complex agent skills force the agent to read
validation rules, irrelevant branches, and conditional instructions before it
knows which behavior applies. That wastes context and makes skills harder to
review. Skillrouter moves deterministic routing work into a CLI and leaves the
agent-facing skill as a tiny instruction to run the CLI and follow its output.

The authoring surface stays project-local under `.skillrouter/`. Agent-specific
skill locations, such as `.claude/skills/` or `.agents/skills/`, remain outside
the core rendering contract. V1 includes a narrow `generate` command only because
creating the tiny router file is useful onboarding, and it remains explicit and
agent-agnostic rather than auto-detecting an ecosystem-specific destination (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`

## Scope

V1 includes:

- The canonical commands `skillrouter run <skill> [input flags...]` and
  `skillrouter generate <skill> --out <path> [--force]`; skill input flags belong
  after the skill name for `run` (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P25).
- Project root discovery by walking upward from `cwd` to the nearest ancestor
  containing `.skillrouter/`; commands fail if no such root exists (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P23).
- Skill resolution from a single safe directory name to
  `.skillrouter/<skill>/SKILL.template.md`; slash-separated skill paths are out
  of v1 (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P24).
- Markdown templates with YAML frontmatter input schema and the v1 directive
  set: `if`, `else-if`, `else`, `include`, and `include-raw` (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P3, P5, P6, and P7).
- Input schema validation for `string`, `boolean`, and `enum` inputs declared in
  frontmatter with explicit `required: true` or `required: false` (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P11, P13, P26, and P27).
- Direct input interpolation, include resolution, condition parsing/evaluation,
  and Markdown rendering as independently testable compiler modules.

V1 explicitly does not include:

- General-purpose template engine behavior, arbitrary JavaScript or TypeScript
  execution, helper functions, filters, arbitrary expression interpolation, or
  file-system access from expressions.
- A first-class `step` directive. Template authors own headings, step labels,
  and workflow structure in ordinary Markdown; Skillrouter only selects,
  includes, interpolates, and renders content (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P5 and P32).
- `path`, `path[]`, arrays, objects, numbers as input schema types, or input
  defaults (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P13 and P15).
- Include path aliases such as `@docs/foo.md`, `$PROJECT_ROOT/foo.md`, absolute
  paths, home-relative `~` paths, or root override flags (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P18, P23, and P33).
- Machine-readable run output such as `--format=json`, verbose tracing, router
  skill install/update/uninstall lifecycle management, automatic agent skill
  destination detection, or nested skill groups.

## Template Authoring Contract

A skill source lives at `.skillrouter/<skill>/SKILL.template.md`. The root
template must have YAML frontmatter containing at least `name`, `description`,
and, when the skill accepts inputs, an `inputs` object.

Example frontmatter shape:

```yaml
---
name: analyze-code
description: Analyze code for bugs, security issues, and code quality problems
inputs:
  language:
    type: enum
    values: [typescript, python]
    required: true
  target:
    type: string
    required: false
  fix:
    type: boolean
    required: false
---
```

Every input must use a uniform `type` field. Enum inputs must provide a
non-empty `values` array. Every input must explicitly declare `required: true`
or `required: false`; there is no implicit requiredness default (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P26 and P27).

The template body is Markdown plus Skillrouter directives. The renderer outputs
ordinary Markdown and erases directive syntax from the final output (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P32).

Conditional blocks use sibling block directives with strict adjacency:

```md
:::if{condition="language == 'typescript'"}
Read the TypeScript rules.
:::

:::else-if{condition="language == 'python'"}
Read the Python rules.
:::

:::else
Ask the user for a supported language.
:::
```

`else-if` and `else` must immediately follow the previous branch except for
blank lines. Any non-blank content between branches ends the conditional group,
and a later `else-if` or `else` is a template error (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P7).

`if` and `else-if` require a `condition` attribute. Include directives do not
accept `condition` or `when`; conditional includes must be wrapped in an `if`
block so there is only one conditional mechanism (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P8).

Includes use an explicit `path` attribute:

```md
::include{path="fragments/typescript.md"}

::include-raw{path="examples/raw-prompt.md"}
```

`include` reads the target and processes Skillrouter directives and direct input
interpolation inside that file with the current input context. Included files
are fragments in v1: they do not declare additional input schemas. `include-raw`
reads the target and inserts its contents literally, without evaluating
Skillrouter directives or interpolation inside that file (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P6 and P9).

Direct input interpolation uses `{{inputName}}` only:

```md
Analyze `{{target}}` with language `{{language}}`.
```

Interpolation may reference declared inputs only. It does not support
expressions, helpers, nested paths, function calls, or formatting filters. If a
template interpolates an optional input that was not provided, rendering fails
instead of silently inserting empty text (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P16 and P17).

## Command Behavior

`skillrouter run <skill> [input flags...]` performs this observable sequence:

1. Discover the project root from `cwd`.
2. Validate that `<skill>` is a single safe directory name.
3. Load `.skillrouter/<skill>/SKILL.template.md`.
4. Parse frontmatter, Markdown, directives, include directives, and
   interpolation references.
5. Parse generic command syntax and preserve raw flag shape for schema-aware
   validation.
6. Validate all skill-specific inputs and template semantics in the compiler.
7. Resolve selected includes, detect include cycles, evaluate conditions,
   interpolate declared inputs, and render clean Markdown.
8. Print rendered Markdown to stdout and exit `0`.

The CLI parser owns only generic command syntax, such as recognizing the command
shape, malformed flags, duplicate raw flag appearances, and whether a flag is
written as `--key`, `--key=value`, or another unsupported form. The compiler owns
all skill-specific validation, including required inputs, enum membership,
declared input names, condition references, interpolation references, and
template errors. This keeps the CLI a thin shell around testable compiler logic
(per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P12).

Input flag rules:

- Unknown input flags are validation errors (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P28).
- Duplicate input flags are rejected for all types (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P29).
- Boolean inputs accept `--flag`, `--flag=true`, and `--flag=false`; duplicate
  or conflicting boolean values are rejected, and `--no-flag` is not a v1 form
  (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P14).
- String and enum inputs require `--name=value`. Empty values are invalid unless
  a future version explicitly defines empty-string semantics.
- Missing required inputs are validation errors. Missing optional inputs are
  falsey in conditions and invalid when interpolated directly.

Condition expressions support identifiers, string literals, number literals,
boolean literals, truthiness, `!`, `==`, `!=`, `&&`, `||`, and parentheses only.
They do not support `in`, comparison operators, property access, function calls,
file reads, or JavaScript evaluation (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P10).

Truthiness for v1 conditions is:

- Missing optional input: false.
- Boolean input: its boolean value.
- String or enum input: true when provided with a non-empty value.

Include path rules:

- Paths are resolved relative to the file containing the directive.
- `../` is allowed only if the final resolved path remains inside the discovered
  project root.
- Absolute paths and `~` are rejected.
- `.env` and `.env.*` files are rejected even when they are inside the project
  root.
- `include` tracks the include stack and errors with the include chain when a
  recursive include cycle is detected (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P18 and P19).

`skillrouter generate <skill> --out <path> [--force]` performs this observable
sequence:

1. Discover the project root from `cwd`.
2. Validate and load `.skillrouter/<skill>/SKILL.template.md`.
3. Read `name` and `description` from template frontmatter.
4. Refuse to write if `--out` is missing.
5. Refuse to overwrite an existing file unless `--force` is provided.
6. Write a minimal router skill file to the explicit output path.

`generate` never guesses the correct agent-specific destination. The output path
is mandatory because v1 stays agent-agnostic and avoids surprising writes into
`.claude/`, `.agents/`, or any other ecosystem-specific folder (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P2). Existing files are protected by default and can only be replaced with
`--force` (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P22).

The generated router skill content must contain only frontmatter `name` and
`description`, the command, and failure handling:

````md
---
name: <template name>
description: <template description>
---

Run this command and follow its output exactly:

```bash
skillrouter run <skill> $ARGUMENTS
```

If the command exits non-zero, report the exact error output to the user and
stop.
````

This minimal content keeps the agent-facing file short while making failure
reporting actionable (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P21 and P31).

## Error And Output Contract

Successful `run` output goes to stdout as Markdown only. V1 does not expose
JSON, verbose traces, provenance comments, or generated wrappers in rendered
Markdown (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P20 and P32).

Failures print a single-line `Error: ...` message to stderr and exit non-zero.
Stdout must remain empty on failure so router skills can safely treat stdout as
the rendered instruction channel. Error categories include invalid command
syntax, missing project root, missing skill, invalid frontmatter, malformed
schema, missing required input, unknown input flag, duplicate input flag, invalid
enum value, condition parse error, condition reference to an undeclared input,
invalid interpolation reference, absent optional interpolation, include not
found, include outside project root, rejected absolute or home-relative include,
rejected `.env*` include, and include cycle (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P30).

Diagnostic text should be concise and safe for an AI agent to quote to the user.
Detailed tracing is a future feature, not part of the v1 public output contract.

## Implementation Constraints

Use Bun as the runtime, package manager, and bundler for the project, but keep
runtime code portable by avoiding Bun-specific runtime APIs and Bun's test
runner. Tests should not depend on Bun-only test semantics so the project can
move to Node or another runtime later.

The CLI must remain a thin shell around separately testable modules. At minimum,
the implementation should isolate these concerns:

- command parsing and raw flag collection
- project root discovery
- skill name validation and template path resolution
- frontmatter parsing and schema validation
- Markdown/directive parsing
- condition expression parsing and evaluation
- include resolution and cycle detection
- input validation and coercion
- interpolation
- Markdown rendering
- router skill generation
- structured errors mapped to CLI stderr messages

The Markdown parser foundation should use the `unified` / `remark` ecosystem,
including `remark-directive` or equivalent remark-compatible directive support,
behind internal parser/rendering modules. The user-facing contract is
Skillrouter's documented syntax, not remark internals (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P4).

Template evaluation must be deterministic and side-effect free except for
explicit include file reads. Conditions and interpolation must not execute
JavaScript, TypeScript, shell commands, user-defined functions, or arbitrary file
access.

## Unresolved Questions

These questions do not block v1 because the spec either excludes them or defines
a conservative first behavior:

- Whether later versions need `path`, array, object, number, or default input
  semantics once real skill templates show pressure for them.
- Whether root-relative include syntax, include aliases, or a `--root` option
  are needed for larger repositories.
- Whether authoring ergonomics require a future preprocessor or alternate
  directive syntax if remark directive markers prove awkward.
- Whether JSON, verbose traces, or other debug output formats should be added
  after the Markdown stdout contract is stable.
- Whether nested skill groups or lifecycle commands for install, update, sync,
  and uninstall are useful enough to justify broadening the v1 boundaries.

## Acceptance Guidance

An implementation satisfies this spec when reviewers can verify the behavior
through focused unit tests and CLI-level integration tests:

- `skillrouter run <skill>` discovers the nearest `.skillrouter/` project root
  from nested directories, resolves only single-name skills, reads
  `.skillrouter/<skill>/SKILL.template.md`, and prints rendered Markdown to
  stdout on success.
- Templates with valid `string`, `boolean`, and `enum` frontmatter inputs render
  selected `if` / `else-if` / `else` branches, erase directives from output,
  recursively process `include`, insert literal `include-raw`, and interpolate
  direct input placeholders.
- Validation fails with stderr `Error: ...`, non-zero exit, and empty stdout for
  missing required inputs, unknown flags, duplicate flags, invalid enum values,
  malformed schemas, unsupported input types, unsupported condition operators,
  undeclared condition identifiers, undeclared interpolation placeholders, and
  absent optional interpolation.
- Boolean flags accept `--flag`, `--flag=true`, and `--flag=false`; duplicate
  boolean flags and `--no-flag` are rejected.
- Include tests cover file-relative resolution, `../` within the project root,
  rejection of paths outside the project root, rejection of absolute and `~`
  paths, rejection of `.env` and `.env.*`, include-not-found errors, and include
  cycle diagnostics that include the chain.
- `skillrouter generate <skill> --out <path>` writes the minimal router skill
  content with template `name` and `description`, the exact
  `skillrouter run <skill> $ARGUMENTS` command, and the exact non-zero error
  instruction. It refuses existing files without `--force` and overwrites them
  with `--force`.
- The implementation exposes independently testable modules for parsing,
  validation, condition evaluation, include resolution, interpolation, rendering,
  and generation rather than embedding those behaviors only in CLI command
  handlers.
- Project verification passes with `bun run check`, `bun run typecheck`, and
  `bun test`.
