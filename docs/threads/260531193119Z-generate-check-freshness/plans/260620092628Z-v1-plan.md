# Generate Agent-Skill Freshness Check (`--check`) Implementation Plan

**Goal:** Add a `--check` mode to `jastr generate agent-skill` that rebuilds the
wrapper in memory, byte-compares it against the committed file at `--out`, writes
nothing, and exits `0` (match) or `1` (stale / missing) with stable messages.

**Architecture:** `--check` is entirely a `@jastr/cli` concern. The CLI argv
pre-validator (`packages/cli/src/args.ts`) whitelists the flag and rejects
`--check` + `--force`; the Commander generate command declares the option;
`executeGenerate` (`packages/cli/src/commands.ts`) performs every build step a
normal generate performs and then, instead of writing, calls a new
`checkAgentSkillOutput` helper (`packages/cli/src/targets/agent-skill.ts`) that
reads the committed file and compares bytes. The engine
(`packages/engine/src/errors.ts`) gains only two new string literals in the
`JastrErrorCode` union: `output_stale` and `output_missing`.

**Tech Stack:** TypeScript, Bun (runtime/test/bundler), Vitest, Commander
(`@commander-js/extra-typings`), `yaml`, Biome. Node `fs/promises` for reads.
Functional-requirement YAML + data-driven e2e cases under
`packages/cli/test/e2e/`, plus a living-docs generator that rebuilds
`packages/cli/docs/BEHAVIOR.md`.

## Background you need before starting

You have likely never seen this codebase. Read this section once; it explains
every concept the tasks below assume.

### What `jastr generate agent-skill` already does

`jastr generate agent-skill <template-ref> --out <path>` loads a Markdown
template, validates it, statically renders it once to catch render errors, builds
a small "wrapper" `SKILL.md` from the template's `targets.agent-skill.frontmatter`
metadata, and writes that wrapper to `--out`. The wrapper just tells an agent to
run `jastr run <template-ref>` and follow the output. `--force` overwrites an
existing destination; without `--force`, an existing destination is an error.

`<template-ref>` has four shapes (all already supported): a bare named ref
(`demo`), a grouped ref (`team/review`), a variant ref (`review#deep`), or a
direct file path (`templates/x.md`). You do **not** need to change any of this
shape handling — `--check` reuses it verbatim.

### The exact bytes a wrapper contains (you will hand-author fixtures with these)

`buildAgentSkillContent` (in `packages/cli/src/targets/agent-skill.ts`) produces
deterministic output. For a **no-input** template `demo` with frontmatter
`name: demo`, `description: Demo skill`, it produces exactly (note the single
trailing newline):

````text
---
name: demo
description: Demo skill
---

Run this command and follow its output exactly:

```bash
jastr run demo
```

If the command exits non-zero, report the exact error output to the user and stop.
````

For the variant `review#deep` (base `targets.agent-skill.frontmatter` of
`name: review-base`, `description: Review with the base policy.`,
`allowed-tools: Read`; variant overlay `name: review-deep`,
`description: Review with the deep policy.`; one input `language` left unlocked,
so `$ARGUMENTS` is forwarded) it produces exactly:

````text
---
name: review-deep
description: Review with the deep policy.
allowed-tools: Read
---

Run this command and follow its output exactly:

```bash
jastr run review#deep $ARGUMENTS
```

If the command exits non-zero, report the exact error output to the user and stop.
````

These are copied verbatim from the existing `generate-router` and
`generate-variant-unlocked` e2e fixtures, so they are known-correct.

### How errors surface to the user (uniform contract)

`packages/engine/src/errors.ts` defines `JastrError(code, message, details?)` and
a `JastrErrorCode` string-literal union. Throwing a `JastrError` anywhere in CLI
code is caught in `packages/cli/src/index.ts`, which prints
`Error: <message>\n` to stderr and sets exit code `1`. Success paths write the
returned string to stdout with **no** trailing newline (see
`packages/cli/src/commands/generate.ts`: `process.stdout.write(output)`). Only
`--help`/`help`/`--version` exit `0` as informational paths.

So for this feature:

- Match → return a plain string from `executeGenerate`; it is written to stdout
  with no trailing newline, exit `0`.
- Stale / missing / `--check`+`--force` → throw a `JastrError`; `index.ts` prints
  `Error: <message>\n`, exit `1`.

### The two test layers (you will touch both)

1. **Fast integration tests** (`packages/cli/test/*.test.ts`) run the real CLI
   via `runCli(args, cwd)` (a thin `execa` wrapper in
   `packages/cli/test/support/helpers.ts`). `runCli` uses execa's default
   `stripFinalNewline: true`, so `result.stderr` / `result.stdout` come back with
   the trailing newline **stripped**. Existing generate tests therefore assert
   stderr like `"Error: Output file out/SKILL.md already exists. Use --force to
   overwrite it."` with **no** `\n`. Match this convention.

2. **Data-driven e2e cases** (`packages/cli/test/e2e/cases/<case-id>/case.yml`)
   plus **functional requirements**
   (`packages/cli/requirements/functional/06-generate.yml`). The e2e runner
   (`packages/cli/test/e2e/harness/case-runner.ts`) uses
   `stripFinalNewline: false`, so e2e `stderr` expectations **include** the
   trailing `\n`. A `case.yml`'s `covers:` list ties it to acceptance criteria;
   `validateTraceability` (`harness/traceability.ts`) **fails the suite** if any
   active acceptance criterion has no covering case, or if a case covers a
   nonexistent criterion. This means new requirements and new cases must land
   together.

A case's `fixture/` directory contents are copied into a fresh temp dir that
becomes the project root the CLI runs in. `expect.files` maps an on-disk path to
an expected-content file inside the case dir; the runner reads the on-disk file
**after** the run and asserts equality — this is how we prove `--check` "writes
nothing": the on-disk file still equals the original committed bytes.

```ts
function validateGenerateArgs(rest: string[]): void {
  let sawCheck = false;
  let sawForce = false;

  for (let index = 0; index < rest.length; index += 1) {
    const arg = rest[index];
    if (arg === undefined) continue;

    if (isHelpToken(arg)) {
      return;
    }

    if (arg === "--force") {
      sawForce = true;
      continue;
    }

    if (arg === "--check") {
      sawCheck = true;
      continue;
    }

    if (arg === "--out") {
      validateGenerateOut(rest[index + 1]);
      index += 1;
      continue;
    }

    if (arg.startsWith("--out=")) {
      validateGenerateOut(arg.slice("--out=".length));
      continue;
    }

    if (arg.startsWith("--")) {
      throw new JastrError(
        "invalid_command",
        `Unknown generate option ${arg}.`,
      );
    }

    throw new JastrError(
      "invalid_command",
      `Invalid generate argument ${arg}.`,
    );
  }

  // --force governs overwriting and --check never writes, so the combination is
  // incoherent rather than ignorable. Reject it here, alongside the other
  // generate flag-shape diagnostics.
  if (sawCheck && sawForce) {
    throw new JastrError(
      "invalid_command",
      "--check cannot be combined with --force.",
    );
  }
}
```

runs during the loop, so `--check` without a value for `--out` still produces the
unchanged `Missing value for --out.` / `Missing required --out <path>.` errors.

This task makes `--check` actually run. After whitelisting in Task 3, a plain
`--check` still reaches Commander, which would reject it as an unknown option
until we declare it. We declare the option, add a compare helper that reads the
committed file and compares bytes, and branch `executeGenerate` to compare
instead of write (skipping the output-exists guard).

The comparison is **exact bytes**: we read the committed file as a `Buffer` (no
encoding) and compare with `Buffer.from(content, "utf8")` via `.equals()`. This
honors the spec's "UTF-8 bytes, no normalization" contract — a trailing-newline
or line-ending drift is correctly stale, and there is no chance of two distinct
byte sequences decoding to the same string.

```ts
export async function checkAgentSkillOutput(options: {
  cwd: string;
  out: string;
  templateRef: string;
  content: string;
}): Promise<string> {
  const outputPath = resolveOutputPath(options.cwd, options.out);

  let existing: Buffer;
  try {
    // No encoding => Buffer, so the comparison is exact bytes with no EOL/BOM
    // normalization. Generation is deterministic, so this never flaps.
    existing = await readFile(outputPath);
  } catch (error) {
    if ((error as NodeJS.ErrnoException).code === "ENOENT") {
      throw new JastrError(
        "output_missing",
        `No agent-skill found at ${options.out} to check; generate it with jastr generate agent-skill ${options.templateRef} --out ${options.out}.`,
        { out: options.out },
      );
    }
    throw error;
  }

  if (!existing.equals(Buffer.from(options.content, "utf8"))) {
    throw new JastrError(
      "output_stale",
```
