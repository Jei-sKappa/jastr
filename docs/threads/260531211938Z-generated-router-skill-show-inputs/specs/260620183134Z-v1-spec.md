# Spec: Generated Agent Skill — show available template inputs (v1)

## Status

Active v1 contract for surfacing a template's declared inputs in the generated
Agent Skill body. This spec supersedes, via prose, the specific rules of three
prior specs (enumerated under "Supersession" below). Old thread specs remain
historical documents and are not edited.

Every settled trade-off in this spec is cited inline back to its source
decision in
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
by its heading id. That log uses `P<N>:` headings (one per discussed point); a
citation like `(per <log-path> P3)` points at the `## P3:` record.

## Intended outcome

When `jastr generate agent-skill <template-ref> --out <path>` is run against a
template that declares one or more invocable inputs, the generated wrapper's
**body** carries a human-readable `## Inputs` section listing each available
flag: its name, type (with the enum value set for `type: enum`), required vs.
optional status, the author default when one is declared, and an optional
one-line description. An agent reading the wrapper can therefore translate a
user's natural-language request into a correct `jastr run <template-ref>
--flag=value …` invocation on the first try, instead of guessing flags and
learning the contract only by tripping `run`'s validation errors.

Templates gain a new optional per-input `description` key in `TEMPLATE.md`
frontmatter to feed that section. The change is additive and backward
compatible: every existing template keeps generating successfully, and a
template with no inputs (or a variant with all inputs locked) keeps producing
today's lean wrapper byte-for-byte.

## Context

Today the generated wrapper is intentionally minimal: `name`/`description`
frontmatter (plus passed-through kebab-case fields) and a body that tells the
agent to run `jastr run <template-ref> $ARGUMENTS` and follow the output. The
template's declared `inputs` are deliberately kept out of the artifact, so the
agent has zero signal about which flags exist. For natural-language invocation
this is actively weak: `$ARGUMENTS` is the slash-command substitution token, so
"deploy to prod" expands to `jastr run deploy deploy to prod`, which
`parseRunFlags` (`packages/cli/src/args.ts:68`) rejects as invalid flag syntax.
The passthrough only works when the user already typed valid jastr flags.

This spec is the implementation contract derived from
`docs/threads/260531211938Z-generated-router-skill-show-inputs/proposals/260620171740Z-generated-router-skill-show-inputs-proposal-v2.md`
after its open questions were resolved in
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
(points P1–P6).

## Scope

In scope:

- A new optional `description` key on every input definition (`string`,
  `boolean`, `enum`) in `TEMPLATE.md` frontmatter, validated by `@jastr/engine`.
- Rendering a `## Inputs` section plus a construct-flags instruction in the
  generated Agent Skill body for every input-bearing template, **regardless of
  ref form** — bare named, grouped named, variant (`#<variant-id>`) with at
  least one unlocked input, and direct `.md` (per
  `docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
  P6).
- Removing the `$ARGUMENTS` token from the generator entirely; the command line
  is now always bare `jastr run <template-ref>` (per
  `docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
  P1, P4).

Out of scope (explicitly not built here):

- A placeholder usage skeleton in the command block (e.g. `jastr run deploy
  --env=<...> [--region=<...>]`). Deferred; the command block stays bare prose +
  list. Revisit only on evidence that agents make flag mistakes frequently (per
  `docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
  P4).
- Showing a variant's locked inputs anywhere in the body, even as read-only
  context (per
  `docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
  P5).
- Feeding `description` into `run`'s validation-error output, the `jastr
  validate` report, or any per-template help listing. `description` is
  metadata-only for now; only the generated body consumes it.
- Any new `JastrErrorCode` value. All defects reuse existing codes.
- Putting inputs into generated **frontmatter**. Frontmatter stays
  metadata-only; the inputs view lives in the body only.

## Engine change: the `description` input key

### Schema and type

Add an optional `description?: string` to all three members of
`TemplateInputDefinition` (`packages/engine/src/schema.ts:3`). This is an
additive change to a public exported type (re-exported from
`packages/engine/src/index.ts`); existing consumers are unaffected.

### Validation

In `validateInputDefinition` (`packages/engine/src/schema.ts:68`), validate
`description` once for all three types and attach the result to the returned
definition. Apply these rules **only when the key is present** (use
`Object.hasOwn`), so omitting it stays valid (per
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
P2):

1. Must be a string. Otherwise raise `malformed_schema` with message
   `Description for input <inputName> must be a string.`
2. Must not be empty after trimming (`value.trim() === ""`). Otherwise raise
   `malformed_schema` with `Description for input <inputName> cannot be empty.`
3. Must be single-line: it must not contain `\n` or `\r`. Otherwise raise
   `malformed_schema` with `Description for input <inputName> must be a single
   line.`

No maximum length is imposed (per
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
P2). The stored value is the original string (not trimmed), echoed verbatim by
the generator.

Recommended structuring (to keep DRY and apply to all three type branches): a
small `validateOptionalDescription(inputName, rawDefinition): string |
undefined` helper called once, with its result assigned onto whichever
`definition` object the existing per-type branch builds. Messages above use the
same phrasing pattern as the existing `Default for input <name> …` messages
(`packages/engine/src/schema.ts:170-201`).

### Non-interference invariant

`description` is metadata-only. It must not change the behavior of
`validateTemplateInputs`, `renderTemplateSource`, `coerceRunFlags`
(`packages/cli/src/flags.ts`), or `sampleInputsForStaticRender`
(`packages/cli/src/variants.ts:41`) — all of which key off
`type`/`required`/`values`/`default`. Adding the property must leave their
outputs identical for any template.

## Generator change: the `## Inputs` body

### Two body shapes

The generator emits exactly one of two body shapes. The frontmatter block
(`---` … `---`, built unchanged from `target.name`, `target.description`, and
passed-through fields) and the trailing failure line are identical in both.

The `$ARGUMENTS` token is removed from the generator in all cases; the command
line is always bare `jastr run <template-ref>` (per
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
P1, P4).

**Shape 1 — no rendered inputs.** Used when the template declares no inputs, or
when a selected variant locks all declared inputs. This is **byte-identical to
today's bare wrapper** (`packages/cli/src/targets/agent-skill.ts:197-208`):

```text
---
{frontmatter}
---

Run this command and follow its output exactly:

```bash
jastr run {template-ref}
```

If the command exits non-zero, report the exact error output to the user and stop.
```

**Shape 2 — one or more rendered inputs.** Used when at least one input is
rendered (all declared inputs for non-variant refs; the unlocked inputs for a
variant ref):

```text
---
{frontmatter}
---

## Inputs

{bullets}

Map the user's request to the inputs above and append them as `--flag=value` arguments, including every required input. Then run this command and follow its output exactly:

```bash
jastr run {template-ref}
```

If the command exits non-zero, report the exact error output to the user and stop.
```

The construct-flags instruction sentence is **pinned verbatim** for byte-stable
`--check`:

> `Map the user's request to the inputs above and append them as `--flag=value` arguments, including every required input. Then run this command and follow its output exactly:`

(The backticks around `--flag=value` are literal in the emitted Markdown.)

`{template-ref}` is the requested ref as supplied to `generate`
(`template.requestedTemplateRef`, as used at
`packages/cli/src/commands.ts:160`), so variant refs render as
`jastr run <ref>#<variant-id>`.

### Per-input bullet anatomy

`{bullets}` is the rendered inputs joined by single newlines, one bullet per
input, in **template frontmatter declaration order** (`Object.entries` order of
`schema.inputs`) — pinned for `--check` stability (per
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
P3).

Each bullet is built as the exact concatenation:

```text
- `--{name}` ({type-token}, {req-token}{default-seg}){desc-seg}
```

Where:

- `{name}` is the input name; the flag is backticked with a leading `--`.
- `{type-token}`:
  - `string` → `string`
  - `boolean` → `boolean`
  - `enum` → `enum: ` followed by the declared values joined with `|` (pipe, in
    declared order), e.g. `enum: dev|prod`. The pipe is chosen because the comma
    is already the field separator inside the parentheses (per
    `docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
    P3).
- `{req-token}`: `required` when `required: true`, else `optional`.
- `{default-seg}`: empty unless the input is optional **and** declares a
  `default`, in which case it is `, default: {default-value}`. The engine
  guarantees `required` inputs never carry a default
  (`packages/engine/src/schema.ts:89`), so this segment never co-occurs with
  `required`. `{default-value}` renders as:
  - boolean → `true` or `false`
  - string / enum → the literal default string
- `{desc-seg}`: empty unless the input declares a `description`, in which case
  it is ` — {description}` (a space, an em dash U+2014, a space, then the
  verbatim description). `--check` compares UTF-8 bytes, so the non-ASCII em
  dash is safe and unambiguous against the leading bullet `-` (per
  `docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
  P3). The description is echoed verbatim with no escaping; the single-line
  validation rule already prevents the only value (a newline) that would break
  the bullet.

Worked examples (all three type-tokens, the three required/optional/default
shapes, with and without description):

```text
## Inputs

- `--env` (enum: dev|prod, required) — target environment
- `--region` (string, optional, default: us-east-1) — deployment region
- `--dry-run` (boolean, optional) — preview without applying
- `--tag` (string, required)
- `--verbose` (boolean, optional, default: false)
```

### Variant locked-input filtering

For a `<ref>#<variant-id>` wrapper, the section lists only the **unlocked**
inputs; locked inputs are completely invisible — not listed, not annotated, not
mentioned (per
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
P5). The "open input" predicate is the existing one: an input is open when
`!Object.hasOwn(lockedInputs, name)` (`packages/cli/src/variants.ts:60-67`). A
fully-locked variant has zero open inputs and therefore renders Shape 1
unchanged from today.

### Signature and call-site changes

- `buildAgentSkillContent` (`packages/cli/src/targets/agent-skill.ts:182`):
  replace the `hasInputs: boolean` option with the ordered list of inputs to
  render, e.g. `inputs: ReadonlyArray<{ name: string; definition:
  TemplateInputDefinition }>`. Render Shape 2 when `inputs.length > 0`, else
  Shape 1. Remove the `$ARGUMENTS` branch from the command construction.
- `executeGenerate` (`packages/cli/src/commands.ts:159-166`): compute the
  ordered open-input list and pass it in:
  - non-variant (`selectedVariant === undefined`): all entries of
    `schema.inputs` in order.
  - variant: entries of `schema.inputs` filtered to those not present in
    `selectedVariant.lockedInputs`, preserving declaration order.
- `packages/cli/src/variants.ts`: add an ordered-list helper (e.g.
  `listUnlockedTemplateInputs(schema, lockedInputs)`) returning the open entries
  in declaration order, so the generator and the existing
  `hasUnlockedTemplateInputs` gate agree. `hasUnlockedTemplateInputs` may be
  kept or re-expressed in terms of the new helper.

## Expected behavior (observable)

1. `generate agent-skill` against a no-input template writes today's lean
   wrapper byte-for-byte (Shape 1).
2. `generate agent-skill` against an input-bearing template writes Shape 2 with
   a `## Inputs` bullet per declared input in declaration order and the bare
   `jastr run <ref>` command.
3. `generate agent-skill <ref>#<variant>` with at least one unlocked input
   writes Shape 2 listing only the unlocked inputs; the locked inputs appear
   nowhere in the file.
4. `generate agent-skill <ref>#<variant>` with all inputs locked writes Shape 1
   (no `## Inputs`, bare `jastr run <ref>#<variant>`), unchanged from today.
5. No generated wrapper contains the literal token `$ARGUMENTS` anymore.
6. A template declaring a valid per-input `description` validates and renders it
   verbatim after the em dash. A template omitting `description` validates and
   renders the bullet with no description segment.
7. A template with a non-string, empty/whitespace, or multi-line `description`
   fails `validateTemplateSchema` with the corresponding `malformed_schema`
   message, surfaced identically by `run`, `validate`, and `generate` (uniform
   `Error: <message>` to stderr, exit 1).
8. `generate agent-skill --check` over an input-bearing template byte-compares
   the committed file against the new Shape-2 content: exit 0 with `agent-skill
   at <out> is up to date.` on an exact match, `output_stale` when bytes differ,
   `output_missing` when absent — mechanism unchanged
   (`packages/cli/src/targets/agent-skill.ts:241-274`).

## Constraints

- `@jastr/engine` must remain free of Node FS, Commander, CLI argv parsers,
  process entrypoints, Bun runtime APIs, `.jastr` lookup, and Agent Skill target
  code. The `description` work is pure schema validation and stays inside the
  engine; all body rendering stays in `@jastr/cli`.
- No new `JastrErrorCode`. `description` defects reuse `malformed_schema`; the
  generate/check paths reuse `output_stale`/`output_missing`/`output_exists`.
- Determinism: generated output must be a pure function of template + selected
  variant. The bullet order, separators (`|`, `, `), em-dash description
  separator, and the pinned instruction sentence are all fixed so `--check`'s
  exact-byte comparison never flaps.
- Frontmatter pass-through still excludes template-owned keys via
  `RESERVED_FRONTMATTER_FIELDS` (`packages/cli/src/targets/agent-skill.ts:10`);
  `inputs` never leaks into frontmatter. The inputs view is body-only.
- Node-compatible source only (no Bun-specific runtime APIs) in both packages.

## Supersession

This spec supersedes, by prose, only the following rules of prior specs; the
prior spec files are not edited (per
`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
P6):

- **package-split v2**
  (`docs/threads/260605091319Z-core-cli-package-split/specs/260606182808Z-v2-spec.md`,
  "Generate Behavior", ~lines 323-332): the generated body is no longer `jastr
  run <template-ref> $ARGUMENTS`. The command is always bare `jastr run
  <template-ref>`, and input-bearing wrappers carry a `## Inputs` section plus
  the construct-flags instruction. The engine input-schema rules are extended
  with the optional `description` key.
- **generated-skill-variants v2**
  (`docs/threads/260614103530Z-generated-skill-variants/specs/260614165823Z-v2-spec.md`,
  lines 219-227): "Generated wrapper bodies stay minimal… They do not list
  unlocked inputs" no longer holds — input-bearing wrapper bodies now list
  unlocked inputs. The `$ARGUMENTS`-gating logic is reinterpreted: the same
  unlocked-input gate now decides whether the `## Inputs` section + construct
  prose appear (Shape 2 vs. Shape 1), since `$ARGUMENTS` is gone. Locked inputs
  remain absent from the body.
- **input-defaults v2**
  (`docs/threads/260612215058Z-template-input-defaults/specs/260613125936Z-v2-spec.md`):
  the recognized per-input keys are extended with the optional `description`
  key, subject to the validation rules above.

## Unresolved questions (non-blocking)

- The placeholder usage skeleton (deferred option from P4) may be revisited if
  agents are observed constructing flags incorrectly in practice. No trigger
  metric is defined; this is a judgment call left for a future thread.
- Future reuse of `description` (in `run` errors, the `validate` report, or a
  per-template help listing) is noted but unspecified; out of scope here.

## Acceptance guidance

The implementation is correct when:

- All of `bun run check`, `bun run typecheck`, `bun run test`, `bun run
  test:cli:e2e`, `bun run docs:cli:living --check`, and `bun run build` exit 0.
- A no-input template and a fully-locked variant generate byte-identical output
  to before this change (verified by the unchanged regression-anchor expected
  files).
- An input-bearing template generates Shape 2: the `## Inputs` section in
  declaration order, correct type-tokens (with `|`-joined enum values),
  correct `required`/`optional`/`default:` segments, em-dash descriptions only
  where declared, the verbatim construct-flags sentence, a bare `jastr run
  <ref>` command, and no `$ARGUMENTS` anywhere.
- A variant wrapper with an unlocked input lists only that input and never names
  the locked input(s).
- A valid `description` round-trips verbatim into the bullet; a non-string,
  empty/whitespace, or multi-line `description` fails with the specified
  `malformed_schema` message and exit 1, via `run`, `validate`, and `generate`
  alike.
- `generate agent-skill --check` reports up-to-date / stale / missing correctly
  against the new body, with no new error codes introduced.
- `README.md`, `AGENTS.md`, and `BEHAVIOR.md` no longer assert `$ARGUMENTS`
  forwarding and accurately describe the inputs view and the `description` key.
