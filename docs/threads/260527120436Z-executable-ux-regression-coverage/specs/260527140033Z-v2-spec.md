# User Documentation Site With Executable Examples V2 Spec

## Intended Outcome

This spec replaces the earlier `skillrouter-case`-block UX regression approach
with a user-facing documentation site backed by executable examples.

When implemented, Skillrouter will have a clear, descriptive, extensive docs
site intended for final users. The site will explain what Skillrouter is, how
to install and use it, how to author templates, how `run` and `generate` behave,
and how flags, conditionals, includes, generated router skills, and errors work.

The examples shown to users must not be stale hand-written snippets. Commands,
outputs, and generated files shown in the docs must be backed by executable
example fixtures that run during documentation checks and builds. If an example
no longer works, the docs check or docs build fails.

The guiding rule is:

> Any command, output, generated file, or user-facing behavior shown in the
> documentation must be backed by an executable example or generated from one.

## Context

The previous spec in this thread,
`docs/threads/260527120436Z-executable-ux-regression-coverage/specs/260527121344Z-v1-spec.md`,
specified Markdown `skillrouter-case` fenced blocks as executable UX cases. An
implementation of that approach proved the regression idea, but the result was
too fixture-shaped for the project owner's actual goal. The case blocks were
machine-readable, but hard for a human product owner or final user to read as
documentation.

The clarified goal is broader and more user-facing:

- The docs site is the documentation final users will read.
- It must be clear, descriptive, and extensive, not an internal test dashboard.
- The executable machinery should sit behind the docs, mostly invisible to the
  reader.
- The project still needs the regression protection that made the v1 idea
  valuable.

This v2 spec preserves the useful insight from v1: user-facing behavior should
be protected by executable examples. It changes the source shape: normal docs
pages become primary, and structured example fixtures/manifests become the
machine-checkable backing layer.

Skillrouter remains a private, early-stage CLI. Its core user-facing commands
are:

```bash
skillrouter run <skill> [input flags...]
skillrouter generate <skill> --out <path> [--force]
```

The existing Vitest unit and integration tests remain valuable and should not be
replaced. The docs executable-example layer is a product-documentation and
user-facing regression layer, not a substitute for low-level parser, compiler,
and rendering tests.

## Scope

V2 includes:

- A final-user documentation site under a dedicated docs-site source directory.
- Clear Markdown documentation pages with normal prose, commands, templates,
  rendered output, generated router skill examples, explanations, and reference
  material.
- A hidden executable examples system based on example directories and manifest
  files, not visible `skillrouter-case` blocks inside user docs.
- A docs check command that executes examples through the real Skillrouter CLI,
  verifies expected stdout, stderr, exit code, and expected generated files, and
  fails on drift.
- A docs build command that runs the executable example checks before building
  the static documentation site.
- Integration into normal project verification so stale docs examples are not
  optional.
- Contributor guidance that any user-facing docs example must be executable or
  generated from an executable example.
- Documentation coverage for the important v1 CLI surface: overview, getting
  started, `run`, `generate`, inputs and flags, conditionals, includes and
  include safety, errors, CLI reference, and complete examples.

V2 does not include:

- The `skillrouter-case` fenced-code-block format from the v1 spec.
- A public docs page that exposes fixture YAML or test harness details to final
  users.
- Browser automation or visual regression testing.
- Search indexing beyond what the selected static docs generator provides by
  default.
- Versioned public docs for multiple Skillrouter releases.
- API reference generation from TypeScript source.
- Interactive in-browser execution of Skillrouter.
- Arbitrary shell execution from docs examples.
- Environment-variable fixtures, stdin fixtures, permission fixtures, symlink
  fixtures, or multi-command scripts in v1 of the docs-example system.
- A new Skillrouter product command for docs.
- Changes to Skillrouter's CLI behavior except when executable docs expose an
  existing bug that must be fixed to match the current v1 CLI contract.

## Site Technology And Repository Layout

Use a Markdown-first static documentation site generator. VitePress is the
preferred v1 implementation choice because it is lightweight, Markdown-oriented,
and suitable for a CLI documentation site. If implementation discovers a concrete
compatibility problem with VitePress and the repo's Bun/TypeScript setup, pause
and open a follow-up discussion before substituting another generator.

The documentation source should live under:

```text
docs/site/
```

The executable example fixtures should live under:

```text
docs/examples/
```

The docs tooling should live under:

```text
test/docs/
```

or, if a non-test script is cleaner:

```text
scripts/docs/
```

The preferred file layout is:

```text
docs/site/
  index.md
  guide/
    getting-started.md
    authoring-templates.md
    running-skills.md
    generating-router-skills.md
    inputs-and-flags.md
    conditionals.md
    includes.md
    errors.md
  reference/
    cli.md
    template-syntax.md
    frontmatter-schema.md
  examples/
    complete-language-router.md
  .vitepress/
    config.ts

docs/examples/
  run-basic/
    example.yml
    project/
      .skillrouter/demo/SKILL.template.md
    expected/
      stdout.md
      stderr.txt
  generate-router/
    example.yml
    project/
      .skillrouter/demo/SKILL.template.md
    expected/
      stdout.txt
      files/
        out/SKILL.md

test/docs/
  example-loader.ts
  docs-examples.test.ts
```

The exact page filenames may change if the implementer finds a clearer
information architecture, but the implementation must keep a clear split between
final-user docs pages and hidden executable example fixtures.

## User-Facing Documentation Behavior

The final docs site must read like product documentation, not a test suite.

The home page must explain:

- What Skillrouter is.
- The problem it solves for AI-agent skill specialization.
- The core idea: author rich project-local templates, expose tiny agent-facing
  router skills, and let the CLI render only the instructions relevant to a
  task.
- The two canonical commands: `run` and `generate`.
- A short end-to-end example that links to the getting started guide.

The getting started guide must show:

- A minimal `.skillrouter/<skill>/SKILL.template.md`.
- The command to run it.
- The rendered Markdown output.
- A short explanation of why that output appeared.

The authoring guide must explain:

- Template location: `.skillrouter/<skill>/SKILL.template.md`.
- Root frontmatter purpose.
- `name` and `description`.
- `inputs`.
- The relationship between input names, CLI flags, condition references, and
  interpolation placeholders.

The `run` guide must explain:

- Project root discovery from `cwd`.
- Skill resolution.
- Input flag parsing.
- Rendering selected branches.
- stdout as the rendered Markdown channel.
- stderr as the error channel.

The `generate` guide must explain:

- Why router skills exist.
- Why output paths are explicit.
- How `--out` and `--force` work.
- What generated router skill content looks like.
- How the generated router skill handles command failure.

The inputs and flags guide must explain:

- Supported input types: `string`, `boolean`, and `enum`.
- Required vs optional inputs.
- Accepted flag forms.
- Rejected forms such as `--no-flag`, duplicate flags, unknown flags, bare
  string/enum flags, invalid enum values, and empty values.

The conditionals guide must explain:

- `if`, `else-if`, and `else`.
- Condition references such as `${language}`.
- Truthiness for missing optional inputs, booleans, strings, and enums.
- The need for longer outer fences when nesting conditionals.

The includes guide must explain:

- `include`.
- `include-raw`.
- File-relative include resolution.
- Why absolute paths, `~`, paths outside the project root, `.env`, and `.env.*`
  are rejected.

The errors guide must explain:

- The uniform `Error: <message>` stderr shape.
- Empty stdout on failure.
- Why concise errors are important for agent-facing router skills.
- Common failure examples.

The CLI reference must list:

- `skillrouter run <skill> [input flags...]`.
- `skillrouter generate <skill> --out <path> [--force]`.
- `skillrouter --help`.
- `skillrouter help [command]`.
- `skillrouter --version`.

The complete examples section must show larger realistic workflows that combine
inputs, conditionals, includes, raw includes, and generation.

## Authoring Executable Examples

User-facing docs pages should contain normal prose and example references, not
inline test-fixture blocks.

Use a small, explicit example reference in docs pages. The exact rendering
mechanism may be a VitePress component, Markdown transform, or prebuild
replacement, but the source should stay much cleaner than the v1
`skillrouter-case` block format. A preferred shape is:

```md
<Example id="run-basic" />
```

The rendered site must replace that reference with user-friendly documentation,
such as:

- Example title.
- Brief description.
- Relevant template file or generated file snippets.
- Command.
- Rendered stdout.
- stderr when explaining errors.
- Explanation text written in the docs page around the example.

Example fixture directories are the executable source of truth:

```text
docs/examples/<example-id>/
  example.yml
  project/
  expected/
```

Example IDs must use lowercase kebab-case:

```text
^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$
```

Each `example.yml` must define exactly one command execution. The v1 docs
example system deliberately does not support multi-command scripts. If a docs
page needs to show a sequence, define multiple example fixtures and explain the
sequence in prose.

The recommended manifest shape is:

```yaml
id: run-basic
title: Render a TypeScript branch
description: Shows how `run` selects one branch from an enum input.
cwd: project
command: ["run", "demo", "--language=typescript"]
expect:
  exitCode: 0
  stdoutFile: expected/stdout.md
  stderr: ""
render:
  show:
    - kind: file
      path: project/.skillrouter/demo/SKILL.template.md
      label: Template
    - kind: command
    - kind: stdout
```

The exact TypeScript type may be refined during implementation, but it must
support these concepts:

- `id`
- `title`
- `description`
- `cwd`
- `command`
- expected exit code
- expected stdout, either inline or from a file
- expected stderr, either inline or from a file
- expected generated or modified files
- render instructions controlling what the final docs page displays

The manifest must not support shell snippets. `command` is an argv array passed
after the Skillrouter executable.

## Executable Example Validation

The docs examples check must:

1. Discover example directories under `docs/examples/`.
2. Read each `example.yml`.
3. Validate the manifest schema.
4. Validate safe paths.
5. Create an isolated temporary project for each example.
6. Copy the example's `project/` fixture into that temporary project.
7. Resolve `cwd` relative to the temporary project.
8. Run the real `src/cli/index.ts` through the same runtime used by existing
   CLI integration tests.
9. Compare exit code exactly.
10. Compare stdout exactly.
11. Compare stderr exactly.
12. Compare expected generated files exactly or through explicit
    contains/not-contains checks.
13. Normalize or template only documented dynamic path placeholders.
14. Fail with an error that includes the example ID and the mismatched field.

The runner must not invoke a shell. It must run:

```ts
execa("bun", [cliPath, ...command], {
  cwd,
  reject: false,
  stripFinalNewline: false,
});
```

or an equivalent non-shell invocation.

Expected stdout and stderr must be exact after documented placeholder expansion.
The supported placeholders for v1 are:

```text
{{projectRoot}}
{{cwd}}
```

Do not add ad hoc magic such as "empty stdout means derive the expected output."
If a dynamic path is needed, it must appear explicitly in the expected output
using a documented placeholder.

Path safety rules:

- Manifest paths must be relative.
- Manifest paths must not be empty.
- Manifest paths must use forward slashes.
- Manifest paths must not contain `..` segments.
- Resolved paths must stay inside the example directory or temporary project,
  depending on the field.

The docs examples check must also verify documentation references:

- Every `<Example id="...">` reference in `docs/site/` resolves to an existing
  example fixture.
- Every example fixture is referenced by at least one docs page, unless the
  manifest explicitly marks it as `hidden: true` for shared or transitional
  examples.
- Duplicate example IDs are errors.

## Site Build Behavior

Add package scripts:

```json
{
  "scripts": {
    "docs:check": "...",
    "docs:build": "...",
    "docs:dev": "..."
  }
}
```

The exact command strings depend on the selected generator and script layout,
but the behavior is fixed:

- `bun run docs:check` runs the executable examples and documentation reference
  validation.
- `bun run docs:build` runs `docs:check` first, then builds the static site.
- `bun run docs:dev` starts the local documentation development server.

`bun run test` must include the executable docs checks, either by importing them
as Vitest tests or by making the test suite execute the same validation logic.
The docs checks must not be optional in normal local verification.

The static site build output must go to a gitignored directory such as:

```text
docs/site/.vitepress/dist/
```

or another generator-default output directory that is not committed.

## Initial Documentation Content

The initial docs site must be useful to final users even before Skillrouter is
public. It should be small enough to maintain but complete enough to explain the
current v1 CLI.

The initial page set must include at least:

```text
docs/site/index.md
docs/site/guide/getting-started.md
docs/site/guide/authoring-templates.md
docs/site/guide/running-skills.md
docs/site/guide/generating-router-skills.md
docs/site/guide/inputs-and-flags.md
docs/site/guide/conditionals.md
docs/site/guide/includes.md
docs/site/guide/errors.md
docs/site/reference/cli.md
docs/site/reference/template-syntax.md
docs/site/reference/frontmatter-schema.md
docs/site/examples/complete-language-router.md
```

The initial executable example set must cover at least:

- Minimal `run` success.
- Branch selection by enum input.
- String interpolation.
- Boolean flags: bare, `true`, and `false`.
- Missing required input failure.
- Unknown flag failure.
- Duplicate flag failure.
- Invalid enum value failure.
- Missing optional input in a falsey condition.
- Missing optional input interpolation failure.
- `include`.
- `include-raw`.
- Rejected include path escaping the project root.
- Rejected `.env` include.
- `generate` success.
- `generate` missing `--out`.
- `generate` overwrite refusal.
- `generate --force`.
- `--help`.
- `help run`.
- `--version`.

The docs pages do not need to display every failure example in full. Some
examples may be used only to protect reference-page snippets or to ensure the
docs harness covers behavior the site states in prose.

## Relationship To Existing Tests

Do not remove existing unit or CLI integration tests as part of this feature.

The docs example system has a different purpose:

- Existing unit tests protect implementation details and low-level permutations.
- Existing CLI integration tests protect command plumbing and focused CLI
  behavior.
- Executable docs examples protect final-user documentation from going stale.

If a docs example reveals a missing low-level test, add the low-level test in
the appropriate existing `test/*.test.ts` file. Do not use docs examples as the
only coverage for complex parser or compiler edge cases.

Update `AGENTS.md` to record this rule:

> Any user-facing command, output, generated file, or behavior shown in docs
> must be backed by an executable docs example or generated from one.

Also update `README.md` to mention:

- The docs site exists under `docs/site/`.
- Executable examples live under `docs/examples/`.
- `bun run docs:check` validates examples.
- `bun run docs:build` builds the docs site.
- `bun run test` includes executable docs checks.

The README should not become a full duplicate of the docs site. It should point
readers to the docs site source and explain how to verify it.

## Constraints

Use Bun as the package manager and script runner, matching the project.

Keep production Skillrouter runtime code portable. Adding docs tooling may add
docs-only dependencies, but it must not introduce Bun-specific APIs into the
CLI runtime source.

Prefer TypeScript for docs tooling so validation logic is type-checked with the
rest of the project.

Use existing dependencies where reasonable. The project already has `yaml`,
`execa`, Vitest, TypeScript, and Node standard-library filesystem APIs.

Do not add a BDD framework. The earlier discussion considered BDD/Gherkin and
rejected it as too much ceremony for deterministic CLI I/O. The v2 direction is
docs-first with executable fixtures, not BDD syntax.

Do not expose fixture manifests or harness-specific YAML in final rendered docs
unless a page is intentionally documenting the docs system for contributors.

Expected command output in examples must be reviewable. Avoid broad snapshots
that are updated blindly. Prefer explicit expected files and explicit manifest
fields.

Generated docs output must not be committed unless the project later decides to
publish built artifacts from the repository.

The docs check and docs build must be deterministic on a clean checkout.

New files should use ASCII unless there is a clear reason to use non-ASCII.

Project verification must pass:

```bash
bun run check
bun run typecheck
bun run test
bun run docs:check
bun run docs:build
```

If `bun run test` already includes `docs:check`, it is still acceptable to list
and run `docs:check` separately for reviewer clarity.

## Settled Trade-Offs

The final-user docs site is the primary artifact. Executable examples exist to
protect the docs, not to become the docs.

The v1 `skillrouter-case` format is superseded. It is useful thread history,
but it should not be carried forward into final-user documentation.

Examples are referenced from docs pages and backed by fixture directories. This
keeps user-facing pages readable while keeping command execution and expected
outputs precise.

The executable examples run at docs-check/build time, not in the browser. This
is simpler, deterministic, and appropriate for a CLI project.

The initial site can use a standard static docs generator instead of a custom
site renderer. The project should spend effort on documentation content and
example correctness, not on building a bespoke website engine.

## Unresolved Questions

These questions do not block v2:

- Whether VitePress remains the long-term docs generator if the project later
  needs advanced search, versioned docs, or richer interactive components.
- Whether example rendering should use a VitePress component, a Markdown
  transform, or a prebuild script that writes generated Markdown into a
  temporary docs source directory.
- Whether every executable example must be shown on a page, or whether some
  hidden examples should be allowed for statements in reference prose.
- Whether docs examples should eventually support multi-step workflows, such as
  `generate` followed by a second command, or whether those should remain
  separate examples linked by prose.
- Whether the project should publish the docs site automatically once it becomes
  public.
- Whether future docs should include "design notes" pages for AI-agent authors
  or keep all design rationale in thread artifacts.

## Acceptance Guidance

An implementation satisfies this spec when a reviewer can verify all of the
following.

Documentation site artifacts exist:

- `docs/site/index.md`
- Guide pages for getting started, authoring templates, running skills,
  generating router skills, inputs and flags, conditionals, includes, and errors.
- Reference pages for CLI, template syntax, and frontmatter schema.
- At least one complete example page.
- Static docs generator configuration under `docs/site/`.

Executable example artifacts exist:

- `docs/examples/` contains example directories.
- Each example directory contains `example.yml`.
- Example directories contain project fixtures under `project/`.
- Expected outputs and expected generated files are stored under `expected/` or
  declared explicitly in the manifest.

Docs tooling exists:

- Example loading and validation code exists in TypeScript.
- Example execution uses the real `src/cli/index.ts` entry point.
- Example execution invokes Bun/Skillrouter without a shell.
- Final newlines in stdout and stderr are preserved during comparison.
- Dynamic path placeholders are explicit and documented.
- Unsafe paths are rejected.
- Duplicate example IDs are rejected.
- Docs page references to examples are validated.

Package scripts exist:

- `bun run docs:check`
- `bun run docs:build`
- `bun run docs:dev`

Verification behavior is correct:

- `bun run docs:check` fails if an example command exits with an unexpected code.
- `bun run docs:check` fails if stdout differs from expected stdout.
- `bun run docs:check` fails if stderr differs from expected stderr.
- `bun run docs:check` fails if an expected generated file is missing or wrong.
- `bun run docs:check` fails if a docs page references a missing example.
- `bun run docs:check` fails if an example is malformed.
- `bun run docs:build` runs `docs:check` before building the site.
- `bun run test` includes executable docs validation.

Content coverage is sufficient:

- A final user can read the docs site and understand what Skillrouter does.
- The getting started guide shows a complete minimal working example.
- The `run` docs show a command and rendered Markdown output from an executable
  example.
- The `generate` docs show a generated router skill from an executable example.
- The inputs/flags docs cover accepted and rejected forms.
- The conditionals docs explain truthiness and branch selection.
- The includes docs explain `include`, `include-raw`, and rejected include
  paths.
- The errors docs explain single-line stderr errors and empty stdout on failure.
- The CLI reference lists the canonical commands and options.

Project guidance is updated:

- `README.md` points to the docs site source and docs verification commands.
- `AGENTS.md` records that documented user-facing behavior requires executable
  docs examples.

Final verification passes:

```bash
bun run check
bun run typecheck
bun run test
bun run docs:check
bun run docs:build
```

Manual review of the rendered docs site should show readable final-user
documentation, not fixture YAML or internal test-case formatting.
