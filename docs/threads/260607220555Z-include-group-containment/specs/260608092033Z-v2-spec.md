# Include Group Containment v2 Spec

## Intended Outcome

When this spec is implemented, the `::include` and `::include-raw` directives
use one containment model across direct-file rendering (`jastr run <path>.md`)
and named-template rendering (`jastr run <id>` or
`jastr run <group>/<template-id>`). Includes can resolve from the top-level
template directory, the current source file directory, or an explicit group
root, but the final resolved realpath must remain inside the allowed include
boundary.

The boundary model supports shared partials without returning to project-root
include access. A template is either standalone, where the allowed boundary is
the top-level template's own directory, or grouped, where the allowed boundary is
the exact group root marked by `.jastrgroup`.

Concretely, an author gets:

- `::include{path="fragments/a.md"}`: the default, resolved from the top-level
  template's own directory.
- `::include{root="group", path="shared/preamble.md"}`: resolved from the group
  root, but only when the top-level template is classified as grouped.
- `::include{root="file", path="sibling.md"}`: resolved relative to the file
  where the directive physically appears.
- A boundary-only safety guarantee: includes cannot read files outside the
  allowed standalone-template or group boundary. Files intentionally placed
  inside that boundary, including `.env`-style files, are includable and remain
  the author's responsibility.

## Context

This spec is forward-designed from a settled design review of jastr's include
containment model. The trigger is an inconsistency in current behavior and a
hard product requirement that the previous model could not satisfy.

Current state (as implemented in
`packages/cli/src/templates/template-ref.ts` and
`packages/cli/src/templates/includes.ts`): the two run modes have different
include boundaries. A direct-file template is contained to the template file's
own directory (`includeRoot = path.dirname(templatePath)`, label
`"template directory"`); a named template is contained to the discovered project
root that holds `.jastr/` (`includeRoot = projectRoot`, label
`"project root"`). On top of that, includes are governed by a denylist:
absolute paths and `~` paths are rejected as non-relative, `.env`/`.env.*` is
rejected by name (both on the literal input and the resolved basename), and
containment is checked lexically (`path.resolve` + `path.normalize`, which do
not follow symlinks).

Two problems drove this redesign. First, the same `::include` directive behaves
differently depending on how the template was invoked, which violates least
astonishment. Second, shared partials are a hard requirement: authors must be
able to share a common fragment across multiple templates, and neither a
project-root boundary (inconsistent, and a porous denylist) nor a strict
single-template boundary (no sharing at all) is acceptable.

Two alternatives were evaluated and rejected:

- Strict self-contained: unify both modes on the single template's own
  directory. This is simplest and tightest, but it cannot reach shared partials
  at all, which fails the hard requirement.
- Capability / marker model: drop path boundaries; each includable file
  self-certifies via a filename convention such as `name.jastr.md`, matched on
  the resolved realpath. This is secure by default and gives sharing for free,
  but it forces a marker extension onto every includable file type, which was
  judged too intrusive.

The settled direction keeps the self-contained spirit but makes the unit of
self-containment either a standalone template directory or an explicitly marked
group root. Grouped templates use an exact layout rather than arbitrary upward
marker discovery, so the security boundary can be audited from the top-level
template path.

## Scope

In scope:

- A single include containment model shared by direct-file templates, existing
  one-segment named templates, and grouped two-segment named templates.
- Three resolution roots selected by an optional `root` attribute on `include`
  and `include-raw`: `template` (default), `group`, and `file`.
- Strict grouped-template classification:
  - Direct access is grouped iff the top-level template realpath is exactly
    `<group>/templates/<template-id>/template.md` and `<group>/.jastrgroup`
    exists as a file; otherwise the direct template is standalone.
  - Named access is grouped iff invoked as `jastr run <group>/<template-id>`,
    both segments are valid template-id segments, `<group>/.jastrgroup` exists
    as a file, and `<group>/templates/<template-id>/template.md` exists.
  - One-segment `jastr run <template-id>` remains the existing
    `.jastr/<template-id>/template.md` named-template path and is standalone.
- Group declaration via a `.jastrgroup` marker file at the exact group root.
  The marker's contents are ignored; only its existence as a file matters.
- Grouped named lookup using exactly one slash:
  `<group>/<template-id>` maps to
  `<group>/templates/<template-id>/template.md`. More than one slash, an empty
  segment, or a segment outside the existing template-id grammar is an invalid
  template reference.
- Realpath-based include containment: every include, for any `root`, resolves to
  a realpath, and that realpath must stay inside the allowed boundary.
- Replacing standalone absolute-path / `~` / `..` rejection rules with
  containment alone, so portability is the author's responsibility rather than a
  jastr-enforced denylist.
- Updating the include/template error contract, the affected e2e cases
  (`direct-file-include-escape`, `include-escape-rejected`, plus new
  group/`root` cases), `packages/cli/docs/BEHAVIOR.md`, and `AGENTS.md` so they
  describe the unified model.
- Implementing the grouped-vs-standalone classification in explicit, readable
  code. Future maintainers should be able to re-audit the decision without
  inferring behavior from incidental path manipulation.

Explicitly out of scope:

- Per-file marker extensions such as `name.jastr.md` for includable files.
- Magic path prefixes such as `::include{path="@group/shared.txt"}`. Root
  selection is handled by the separate `root` attribute, and `path` stays a
  normal filesystem path.
- Arbitrary upward `.jastrgroup` discovery. Markers outside the exact group-root
  position for the top-level template are irrelevant to the run.
- Recursive or sideways scans for misplaced `.jastrgroup` files. The CLI does
  not search unrelated directories to decide whether the active template's group
  marker is misplaced.
- Rejecting a `template.md` file directly beside `.jastrgroup`. Such a file is
  not a grouped template; if directly run, it is standalone.
- New directives or any change to the directive set beyond adding the `root`
  attribute to existing `include` and `include-raw`.
- A secrets denylist for files inside the allowed boundary. Within-boundary
  files are includable by design.

## Expected Behavior

### Template Classification

The CLI determines the top-level template first, then classifies it as
standalone or grouped. The group classification is determined once for the
top-level template and is not recomputed while processing included files.

Direct file access:

```bash
jastr run my-single-template/template.md
jastr run my-templates-group/templates/my-template-1/template.md
```

For direct access, the CLI resolves the top-level template file to its realpath.
If that realpath has the exact shape
`<group>/templates/<template-id>/template.md` and `<group>/.jastrgroup` exists
as a file, the template is grouped and `<group>` is the include boundary.
Otherwise the template is standalone and the top-level template directory is the
include boundary.

Named access:

```bash
jastr run my-single-template
jastr run my-templates-group/my-template-1
```

One-segment named access keeps the existing catalog:
`.jastr/<template-id>/template.md`. Such templates are standalone for include
containment.

Two-segment named access has exactly one slash and maps to a grouped template:
`<group>/<template-id>` loads
`<group>/templates/<template-id>/template.md` and requires
`<group>/.jastrgroup` to exist as a file. Grouped named access never falls back
to standalone lookup.

### Group Marker

A group is declared by a marker file named `.jastrgroup` at the exact group
root:

```text
my-templates-group/
  .jastrgroup
  shared/
    hello.txt
  shared-file-in-group-root.txt
  templates/
    my-template-1/
      fragments/
        something.md
      my-file-1.txt
      template.md
    my-template-2/
      template.md
```

The marker file is existence-only. Its contents are ignored; non-empty marker
files do not fail validation.

A directory named `templates/` is not sufficient on its own to create a group.
A `.jastrgroup` file in any position other than the exact group root for the
active top-level template does not define the group for that template.

### Standalone Templates

A standalone direct-file template has no active group:

```text
my-single-template/
  fragments/
    something.md
  template.md
```

A direct file at `<some-dir>/template.md`, including one directly beside a
`.jastrgroup` marker, is standalone unless its realpath is exactly
`<group>/templates/<template-id>/template.md` and `<group>/.jastrgroup` exists
as a file.

### Include Syntax

Template-local includes use the default `template` root and resolve from the
top-level template directory:

```md
::include{path="fragments/something.md"}
::include{root="template", path="fragments/something.md"}
```

Group-root includes resolve from the group root:

```md
::include{root="group", path="shared/hello.txt"}
::include{root="group", path="shared-file-in-group-root.txt"}
```

`root="group"` is valid only when the top-level template was classified as
grouped. In a standalone template, `root="group"` is an error.

Current-file-relative includes resolve from the directory of the source file
where the directive appears:

```md
::include{root="file", path="sibling-of-current-file.md"}
```

`root="file"` changes the starting directory for path resolution. It does not
change the security boundary.

Any `root` value other than `template`, `group`, or `file` is an error.

### Security Boundary

Every include, regardless of `root`, resolves to a realpath, and the realpath is
then checked for containment. The allowed boundary is:

- Standalone template: the top-level template's own directory.
- Grouped template: the exact `<group>` directory containing `.jastrgroup`.

Resolution starting points differ by root, but the boundary check is the same
for all three:

- `root="template"` resolves from the top-level template directory.
- `root="group"` resolves from the group directory.
- `root="file"` resolves from the current source file's directory.
- In every case the final realpath must stay inside the allowed boundary.

For a grouped template, `root="file"` can reach nearby partials elsewhere in
the group but cannot escape the group. For a standalone template, `root="file"`
cannot escape the top-level template directory.

Because containment is enforced on the resolved realpath, absolute paths, `~`,
and `..` need no special denylist semantics: any of them that resolves outside
the boundary is rejected by the single containment check, and any that resolves
inside the boundary is allowed. This is a deliberate change from the current
lexical check plus absolute/`~`/`.env` denylist.

## Error Code And Message Contract

Every CLI failure prints exactly one `Error: <message>` line to stderr and exits
1. `--help`, `help [command]`, and `--version` are the only zero-exit
informational paths.

The include/group changes use these stable codes and message templates after
the `Error: ` prefix:

- `invalid_template_reference`: `Template reference <template-ref> must be a
  template id, a group/template id, or a .md file path.`
- `template_not_found` for one-segment named access: `Template <template-id> was
  not found at .jastr/<template-id>/template.md.`
- `template_not_found` for grouped named access when either the group marker or
  template file is absent: `Template <group>/<template-id> was not found at
  <group>/templates/<template-id>/template.md.`
- `invalid_include_root`: `Include root <root> must be template, group, or
  file.`
- `include_group_missing`: `Include root group requires the template to be
  inside a .jastrgroup.`
- `include_outside_root`: `Include path <include-path> escapes the allowed
  include boundary.`
- `include_not_found`: `Include file <include-path> was not found.`
- `include_read_error`: `Include file <include-path> could not be read:
  <cause>.`

The standalone absolute-path / `~` / `.env` rejection messages are removed.
Those path shapes now succeed or fail through normal realpath containment.

## Constraints

- Engine purity: `@jastr/engine` must not learn about `.jastrgroup`, groups,
  project roots, `cwd`, or filesystem layout. Group classification, root
  selection, realpath resolution, and containment all live in the CLI's injected
  include resolver and template-reference loading.
- Include request shape: the `root` attribute is parsed from the directive and
  passed through the engine's include request to the resolver as an additive
  field. The engine must not interpret `root`.
- Node compatibility: no Bun-specific runtime APIs in engine or CLI source.
  Realpath resolution uses Node filesystem APIs.
- Realpath is load-bearing: containment must be checked after resolving
  symlinks, not lexically. Resolving the realpath is what prevents a symlink
  inside the boundary from pointing out of it.
- Direct-access classification uses the realpath of the top-level template file,
  not the spelling of a symlinked path supplied on the command line.
- Threat model: the boundary protects against a template reaching unintended
  files outside its group/template. It does not defend against an author who
  deliberately places files, symlinks, or hardlinks to secrets inside the
  allowed boundary.
- Uniform error UX: every failure prints exactly one `Error: <message>` line to
  stderr and exits 1.
- Privacy / churn: the project is private and pre-release, so boundary,
  lookup, and error-message changes may be breaking; no legacy include behavior
  needs to be preserved.

## Explicit Decisions

- Grouped named access is in scope for this spec.
- The only grouped named-reference grammar is `<group>/<template-id>`.
- More than one slash in a non-`.md` template reference is invalid.
- Grouped named access maps to `<group>/templates/<template-id>/template.md`.
- Existing one-segment named access remains `.jastr/<template-id>/template.md`
  and is standalone.
- Strict group classification supersedes upward nearest-marker discovery.
- Direct access is grouped only when the top-level template realpath has the
  exact grouped layout and the exact group-root marker exists.
- Named grouped access never falls back to standalone lookup.
- `root="group"` on standalone templates is an error.
- `.jastrgroup` marker contents are ignored.
- A `template.md` directly beside `.jastrgroup` is allowed and is standalone if
  directly run.
- Include security is boundary-only; there is no within-boundary secrets
  denylist.
- The grouped-vs-standalone code path must stay explicit and easy to audit.

## Acceptance Guidance

A reviewer should consider the implementation correct when all of the following
hold:

- `::include` and `::include-raw` use identical containment behavior for direct
  `.md` templates, one-segment named templates, and grouped two-segment named
  templates; there is no longer a project-root include boundary for named
  templates.
- One-segment named templates still load from `.jastr/<template-id>/template.md`
  and are standalone for include containment.
- `jastr run <group>/<template-id>` validates exactly one slash, validates both
  segments with the existing template-id grammar, requires
  `<group>/.jastrgroup`, and loads
  `<group>/templates/<template-id>/template.md` without falling back to
  standalone lookup.
- Direct-file rendering classifies a template as grouped only when the
  top-level template realpath is exactly
  `<group>/templates/<template-id>/template.md` and `<group>/.jastrgroup`
  exists as a file; otherwise it is standalone.
- `.jastrgroup` is treated as an existence-only marker. Its contents are not
  parsed, and a non-empty marker does not fail.
- A `template.md` directly beside `.jastrgroup` is not rejected merely for being
  beside the marker; if directly run, it is standalone.
- No upward marker search, nearest-marker discovery, recursive scan, or
  misplaced-marker validation participates in group classification.
- `::include{path=...}` resolves from the top-level template directory by
  default; `::include{root="group", path=...}` resolves from the group root for
  grouped templates; `::include{root="file", path=...}` resolves from the
  directory of the file containing the directive.
- `root="group"` in a standalone template fails with
  `include_group_missing`; an unknown `root` value fails with
  `invalid_include_root`.
- For all three valid roots, an include whose resolved realpath escapes the
  boundary fails with `include_outside_root` and
  `Include path <include-path> escapes the allowed include boundary.` An include
  resolving inside the boundary succeeds, including absolute paths and paths
  containing `..` that net back inside the boundary.
- A symlink inside the boundary that resolves to a target outside the boundary
  is rejected because containment is enforced on the realpath, not lexically.
- The standalone absolute-path / `~` / `.env` denylist messages are gone,
  replaced by the single containment rejection.
- `@jastr/engine` still has no knowledge of `.jastrgroup`, groups, project
  roots, `cwd`, or filesystem layout; the engine only passes the parsed include
  `root` value through to the injected resolver.
- e2e coverage exists for: existing one-segment named template containment,
  grouped named template loading, direct grouped template classification,
  standalone direct templates, default template-root include, `root="group"`
  include, `root="file"` include, invalid root, `root="group"` on standalone,
  an escape rejection for each valid root, within-boundary absolute/`..`
  includes, and a symlink-based escape rejection.
- The existing `direct-file-include-escape`, `include-escape-rejected`,
  `include-absolute-rejected`, `include-tilde-rejected`, and any `.env`
  rejection cases are updated or removed to match the unified containment model.
- `packages/cli/docs/BEHAVIOR.md` regenerates cleanly
  (`bun run docs:cli:living --check` exits 0) and reflects the group/`root`
  behavior.
- `AGENTS.md` no longer asserts or implies the superseded include model without
  also pointing at this spec as the active include-containment contract.
- The clean verification set exits 0: `bun run check`, `bun run typecheck`,
  `bun run test`, `bun run test:cli:e2e`, `bun run docs:cli:living --check`,
  and `bun run build`.
