# Template Input Defaults and Project Input Config v2

## Intended Outcome

When this spec is implemented, Jastr templates can declare template-author
fallback values for optional inputs, and named project templates can receive
standing user/project input values from `.jastr/config.yml`. Users can run a
multi-input named template such as `jastr run review` without repeating stable
preferences every time, while template authors can make optional input behavior
explicit in the template schema.

The final effective input precedence is:

1. CLI input flags.
2. `.jastr/config.yml` values for the selected named template.
3. Template-author `default:` values declared in frontmatter.

For direct `.md` invocations, the effective precedence is only CLI input flags
over template-author defaults. Direct `.md` runs do not discover or consume
`.jastr/config.yml` in this spec (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P3).

## Context

Today, template inputs support `string`, `boolean`, and `enum`, and each input
must explicitly declare `required: true` or `required: false`. There is no
schema-level fallback value and no CLI project config file. This makes practical
templates noisy to run: a review template with inputs like `depth` and `output`
forces the same flags to be repeated on every `jastr run`.

This spec elaborates the proposal in
`docs/threads/260612215058Z-template-input-defaults/proposals/260612215058Z-input-defaults-proposal.md`
using the settled discussion in
`docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`.
The core design keeps two concepts separate:

- Template-author defaults are part of the template schema and live in
  frontmatter (per
  `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
  P7).
- Project config values are user/project-supplied standing input values and
  live in `.jastr/config.yml` (per
  `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
  P4).

## Scope

In scope:

- Add `default:` support to template input definitions in frontmatter.
- Validate defaults as part of engine schema validation.
- Apply defaults in the engine input validation/rendering path.
- Add optional project-local `.jastr/config.yml` support for named template
  runs.
- Merge selected named-template config values with CLI flags before engine
  validation.
- Preserve direct `.md` execution without requiring `.jastr`.
- Add CLI functional requirements, e2e cases, and living documentation coverage
  for the new behavior.
- Update `AGENTS.md`, and update `README.md` if the public description or usage
  examples are affected.

Out of scope:

- No explicit `--config`, `--input-config`, profiles, user-home config, or other
  direct config override mechanism in this spec (per
  `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
  P3).
- No direct `.md` config keying, path identity, realpath identity, or config
  lookup for direct `.md` invocations.
- No template-local sidecar defaults file. Author defaults remain in
  frontmatter.
- No warning UX. Jastr continues using the existing success/failure model:
  failures print one `Error: <message>` line to stderr with exit code 1.
- No change to the directive set.
- No change to Agent Skill wrapper argument pass-through behavior.
- No config layering. There is only the automatic named-template project config
  source in this spec.

## Template-Author Defaults

An input definition may declare `default: <value>` only when it also declares
`required: false`. Declaring `default:` on `required: true` is a schema error
because an author default would make requiredness misleading (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P8).

Valid examples:

```yaml
inputs:
  dry-run:
    type: boolean
    required: false
    default: true
  language:
    type: enum
    values: [typescript, python]
    required: false
    default: typescript
  target-file:
    type: string
    required: false
    default: src/index.ts
```

Default values use the same domain rules as runtime input values (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P9):

- Boolean input defaults must be YAML booleans.
- String input defaults must be non-empty strings.
- Enum input defaults must be non-empty strings and must be one of the declared
  `values`.
- Empty string defaults are invalid.
- Quoted boolean-like strings such as `default: "true"` are invalid for boolean
  inputs.

Optional inputs without defaults keep today's absent-input behavior (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P18). `required: false` does not imply a synthetic empty string, `false`, or
other implicit value. Only an explicit `default:` supplies an omitted optional
input. As a result:

- An omitted optional boolean with no default remains absent and falsey in
  conditions.
- Interpolating an omitted optional input with no default still fails with the
  existing absent optional interpolation error.
- A defaulted optional input is present in the effective value map, so
  conditions and interpolation use the default value.

Schema validation should continue using `malformed_schema` for malformed input
definitions. New message templates should be stable and covered by tests:

- `Input <input-name> cannot declare default when required is true.`
- `Default for input <input-name> must be a boolean.`
- `Default for input <input-name> must be a string.`
- `Default for input <input-name> cannot be empty.`
- `Default for input <input-name> must be one of: <values>.`

## Engine Behavior and API Constraints

`@jastr/engine` owns template schema semantics. It must parse, validate, and
apply frontmatter defaults in the engine path, not in the CLI (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P10).

`validateTemplateInputs(schema, suppliedInputs)` must return the effective
validated values after applying template-author defaults. `renderTemplateSource`
must render using those effective values. Direct TypeScript consumers of
`@jastr/engine` therefore get the same default behavior as the CLI.

The public top-level API remains the same set of functions:
`parseTemplateSource`, `validateTemplateSchema`, `validateTemplateInputs`,
`renderTemplateSource`, and `JastrError`. Do not add a new top-level default
application helper. It is acceptable to broaden input parameter typing as needed
so raw maps from YAML/config can be validated by the engine while returning the
validated `TemplateInputValues` domain type.

The engine still must not import Node filesystem APIs, Commander, process
entrypoints, Bun runtime APIs, `.jastr` lookup code, or Agent Skill generation
code. Project config discovery and parsing remain CLI responsibilities.

## Project Config File

Named template runs may use a project-local `.jastr/config.yml`. The file is
optional. It is discovered from the same project root used for named template
lookup. Direct `.md` template runs do not search for, read, or apply this file
(per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P3).

The config shape is:

```yaml
inputs:
  review:
    depth: standard
    output: chat
  team/review:
    depth: deep
```

The recognized top-level section is `inputs`, not `defaults`, because config
values are supplied inputs, not schema defaults (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P4).

Template keys under `inputs` are exact named template refs. Standalone templates
use keys such as `review`; grouped templates use keys such as `team/review`,
matching the user-facing `jastr run team/review` command (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P13).

Project config values are user/project-supplied input values and can satisfy
either `required: true` or `required: false` inputs. The config file does not
care about requiredness; requiredness is evaluated only after CLI flags and
config values are merged and template defaults are applied (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P15).

Config values use strict YAML/domain types, not CLI flag coercion (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P16):

- Boolean inputs use YAML booleans, for example `dry-run: true`.
- String and enum inputs use non-empty YAML strings.
- Enum values must be in the declared set after precedence is applied.
- Quoted boolean strings such as `dry-run: "true"` are strings, not booleans,
  and are invalid for boolean inputs unless overridden by a CLI flag before
  validation.

Unknown top-level keys in `.jastr/config.yml` are ignored for forward
compatibility. The recognized `inputs` top-level section must be a mapping when
present, but validation below `inputs` is selected-entry-only for a run (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P5 and P17). This behavior needs a dedicated CLI functional requirement and e2e
case.

Absent or empty `.jastr/config.yml` means no project config. A whitespace-only
file also means no project config. Invalid YAML, a non-mapping top-level value,
a non-mapping `inputs` value, or a non-mapping selected template entry is an
error. A non-mapping entry under `inputs` for a different template ref is not
validated and does not affect the selected run (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P5 and P12).

Config parse and shape failures must use the stable CLI error code
`invalid_config` and these exact message templates:

- `.jastr/config.yml could not be parsed.`
- `.jastr/config.yml must be a mapping.`
- `.jastr/config.yml inputs must be a mapping.`
- `.jastr/config.yml inputs.<template-ref> must be a mapping.`

In the selected-entry message, `<template-ref>` is replaced with the exact
selected named template ref. For example, `jastr run review` prints
`.jastr/config.yml inputs.review must be a mapping.`, and
`jastr run team/review` prints
`.jastr/config.yml inputs.team/review must be a mapping.`

The final CLI still prints them as `Error: <message>` with exit code 1.

## CLI Merge and Validation Behavior

The CLI owns filesystem config discovery, YAML parsing, config shape validation,
selection of `inputs.<template-ref>`, CLI flag parsing/coercion, and merging.
The engine owns final effective input validation and default application (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P11).

For `jastr run <named-ref> [input flags...]`, the CLI should:

1. Load the named template using existing project root discovery.
2. Parse and validate the template schema.
3. Read `.jastr/config.yml` from the discovered project root if it exists.
4. Select only `inputs.<named-ref>` from config.
5. Parse/coerce CLI flags.
6. Build the supplied input map as selected config values overwritten by CLI
   flag values.
7. Pass the supplied input map to `renderTemplateSource`.
8. Let the engine validate unknown names, value types, enum membership,
   requiredness, and defaults on the effective map.

For `jastr run <path>.md [input flags...]`, the CLI should:

1. Load the direct file using existing direct mode.
2. Parse/coerce CLI flags.
3. Pass only CLI flag values to `renderTemplateSource`.
4. Let the engine validate and apply template-author defaults.

The CLI flag parser must no longer reject missing required inputs before config
and defaults can participate. Requiredness is enforced only in the engine final
validation path after merge/default application.

Unknown configured input names and invalid winning config values are strict only
within the selected template's config entry after CLI precedence is applied.
Entries for other template refs are not validated during this run, including
entries whose value is not a mapping (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P5).

Invalid config values overridden by a CLI flag for the same input must not fail
the run. The CLI must merge first and validate only the effective supplied
values after precedence. This behavior must have a dedicated functional
requirement, an e2e case, and a focused code comment at the merge/validation
point so future maintainers do not accidentally pre-validate overridden config
values (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P6).

Example:

```yaml
# .jastr/config.yml
inputs:
  review:
    depth: typo
```

```bash
jastr run review --depth=deep
```

If `depth` is an enum with values `[quick, standard, deep]`, this command
succeeds and uses `deep`.

If the same config value wins because no CLI flag overrides it, final engine
validation fails.

## Agent Skill Generation

`jastr generate agent-skill` keeps the current wrapper command rule. If the
template declares any inputs, the generated wrapper uses:

```bash
jastr run <template-ref> $ARGUMENTS
```

If the template declares no inputs, the generated wrapper uses:

```bash
jastr run <template-ref>
```

Defaults and config can make an input-bearing template runnable without flags,
but generation must not inspect `.jastr/config.yml` to decide wrapper content,
and it must not remove `$ARGUMENTS` for input-bearing templates (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260612215058Z-template-input-defaults/discussions/2026-06-13-template-input-defaults-design-discussion.md`
P14).

## Implementation Constraints

- Use Bun as the workspace runtime/package manager/bundler, while keeping
  `@jastr/engine` and `@jastr/cli` source Node-compatible.
- Keep project config discovery and YAML reading in `@jastr/cli`.
- Keep template defaults, schema validation, and effective input validation in
  `@jastr/engine`.
- Do not introduce Commander options after `<template-ref>` for this feature.
  `run` input flags remain template-specific pass-through arguments.
- Keep the uniform CLI error UX: one `Error: <message>` line to stderr and exit
  code 1.
- Preserve existing include containment and direct-file behavior.
- Update public API tests if the public type signatures are broadened, but do
  not add a new public top-level helper for defaults.
- Update `packages/cli/requirements/functional/` with requirement IDs and
  acceptance criteria for the new config/default behaviors.
- Add or update e2e cases under `packages/cli/test/e2e/cases/` and regenerate
  `packages/cli/docs/BEHAVIOR.md`.
- Update `AGENTS.md` because it currently states templates have no config
  concept and documents Agent Skill wrapper behavior. Update `README.md` if
  public usage or current status changes.

## Acceptance Guidance

A reviewer should expect focused engine tests for:

- Schema parsing preserves valid `default:` metadata.
- `default:` on `required: true` fails schema validation.
- Invalid default types fail schema validation for boolean, string, and enum
  inputs.
- `validateTemplateInputs` returns effective values that include defaults for
  omitted defaulted optional inputs.
- CLI/supplied values override frontmatter defaults.
- Optional inputs without defaults remain absent, falsey in conditions, and
  still error when interpolated without a value.
- `renderTemplateSource` applies defaults for direct engine consumers.
- Public API export tests still pass with the intended top-level API.

A reviewer should expect focused CLI/unit/e2e coverage for:

- Named `jastr run review` uses `.jastr/config.yml` values.
- Config values satisfy required inputs.
- Precedence is CLI flag over config over frontmatter default.
- Direct `.md` runs ignore `.jastr/config.yml` but still apply frontmatter
  defaults.
- Grouped named templates use exact keys such as `team/review`.
- Absent, empty, and whitespace-only `.jastr/config.yml` behave as no config.
- Invalid YAML and invalid config shapes fail with the exact `invalid_config`
  code and exact message templates listed in this spec.
- A malformed non-selected template entry such as `inputs.other-template: true`
  does not affect `jastr run review`.
- Unknown top-level config keys are ignored, with a dedicated functional
  requirement and e2e case.
- Unknown input names under the selected template entry fail, while entries for
  other template refs do not affect the run.
- Invalid selected config values fail when they win precedence.
- Invalid selected config values do not fail when a CLI flag overrides the same
  input, with a dedicated functional requirement, e2e case, and implementation
  comment.
- Config values use strict YAML/domain types; quoted boolean strings fail for
  boolean inputs unless overridden before validation.
- Agent Skill wrapper output remains unchanged for input-bearing templates.
- `packages/cli/docs/BEHAVIOR.md` is regenerated and checked.

Before considering the implementation clean, run:

```bash
bun run check
bun run typecheck
bun run test
bun run test:cli:e2e
bun run docs:cli:living --check
bun run build
```

All commands should exit 0.
