# Generated Skill Variants v2 Spec

## Intended Outcome

Jastr supports project-config-defined variants for named templates. A user can
define a variant over one existing `.jastr/<template-id>/TEMPLATE.md` or grouped
template and run it with `<template-ref>#<variant-id>`. The variant can lock
selected template inputs so they are always supplied by the selected variant,
hidden from generated Agent Skill invocation where possible, and protected from
conflicting invocation-time CLI flags.

The feature lets projects generate multiple purpose-specific Agent Skill
wrappers from one base template without copying the template or creating a new
`.jastr/<variant>/TEMPLATE.md`. A generated variant wrapper points back to the
base template through the variant ref and relies on CLI/runtime enforcement, not
wrapper convention, to preserve locked input behavior.

## Context

The upstream proposal is
`docs/threads/260614103530Z-generated-skill-variants/proposals/260614103530Z-locked-input-skill-variants-proposal.md`.
It follows the template input defaults and project config work: defaults and
`inputs.<template-ref>` reduce repeated flags, but both are intentionally
overridable. This spec adds a stronger specialization mechanism for generated
skills where selected inputs are locked.

The pre-spec discussion at
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md`

identifiers rather than decision-log `D<N>` headings; citations below reference
those operative points inline.

This spec does not reopen richer input domains. Locked values apply to the
current Jastr input model: `string`, `boolean`, and `enum`.

This v2 spec supersedes
`docs/threads/260614103530Z-generated-skill-variants/specs/260614164514Z-v1-spec.md`
only to clarify the CLI help-text contract. The product behavior remains the
same as v1: help must make variant refs discoverable, but the exact help copy is
not a stable message contract.

## Scope And Non-Scope

In scope:

- Named-only variant refs with syntax `<template-ref>#<variant-id>`, including
  grouped refs such as `team/review#deep`
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P1, P5).
- A new `.jastr/config.yml` `variants` surface using exact template-ref keys
  and nested variant-id keys, with `locked-inputs` for locked values
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P2, P5).
- `jastr run <template-ref>#<variant-id>` resolution, config lookup, locked
  input precedence, locked CLI flag conflict handling, stable errors, and error
  codes.
- `jastr generate agent-skill <template-ref>#<variant-id>` generation,
  including optional variant Agent Skill frontmatter overrides, variant-aware
  wrapper command shape, and locked-value static render validation.
- CLI help, README, AGENTS.md, functional requirements, e2e cases, and living
  CLI docs updates needed to make the feature discoverable and traceable
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P12, P13, P14).

Out of scope:

- Direct `.md` variants. `jastr run path/to/template.md#variant` and
  `jastr generate agent-skill path/to/template.md#variant` are not supported
  because variants depend on project config, and direct `.md` runs deliberately
  do not read `.jastr/config.yml`
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P1, P5).
- Template-declared variants, generate-time `--lock` flags, or another external
  variant artifact. Variant definitions live in `.jastr/config.yml` for this
  version.
- Duplicating templates to specialize a few inputs.
- Treating locking as a generated wrapper convention without CLI enforcement.
- Listing or describing remaining unlocked inputs in generated wrapper bodies;
  wrappers stay minimal
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P21).
- An implicit default variant. `variants.demo.default` is just a variant named
  `default` and is selected only by `demo#default`
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P22).

## Expected Behavior

### Variant Ref Grammar

A variant ref is a named template ref followed by `#` and a variant id:

```text
<template-id>#<variant-id>
<group>/<template-id>#<variant-id>
```

The base template ref keeps the existing named template grammar. The variant id
uses the same lowercase kebab-case grammar as template id segments. Config uses
the exact base template ref as the key under `variants`, including grouped keys
such as `team/review`
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P5).

Malformed refs such as `demo#`, `demo#deep#extra`, `demo#Deep`,
`bad/ref/shape#deep`, and direct `.md` variant refs fail before template
loading with the invalid template reference error message specified below.

### Config Shape

`.jastr/config.yml` may contain `variants` alongside the existing overridable
`inputs` surface:

```yaml
inputs:
  demo:
    depth: quick

variants:
  demo:
    custom:
      locked-inputs:
        depth: deep
        dry-run: true
      agent-skill:
        frontmatter:
          name: custom-review
          description: Review TypeScript changes with the custom policy.
```

`variants` must be a mapping when the command path reads and validates project
config for variants. For a selected variant, `variants.<template-ref>` and
`variants.<template-ref>.<variant-id>` must both be mappings. `locked-inputs`
and `agent-skill` are optional. Empty selected variants are valid aliases over
the base named template. Unknown fields in the selected variant entry are
rejected so misspellings such as `locked-input` cannot silently produce an
empty alias
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P6, P16).

Deep validation is selected-only: malformed unrelated templates or variants
under `variants` do not break a run or generation command for another selected
variant. The selected `locked-inputs` and selected `agent-skill.frontmatter`
shape are strict
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P6, P17).

`agent-skill` override shape mirrors template target metadata:

```yaml
variants:
  demo:
    custom:
      agent-skill:
        frontmatter:
          name: custom-review
```

The selected `agent-skill` value must be a mapping whose only supported field is
`frontmatter`; `frontmatter` must be a mapping when present
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P17).

### Run Behavior And Input Precedence

`jastr run <template-ref>` remains the existing bare named run behavior and
never selects a variant. It uses CLI flags over `inputs.<template-ref>` over
template-author frontmatter defaults. A variant named `default` has no reserved
meaning unless explicitly selected with `#default`
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P22).

`jastr run <template-ref>#<variant-id>`:

1. Resolves the base named template ref exactly as `jastr run <template-ref>`
   would.
2. Loads `.jastr/config.yml` from the discovered project root.
3. Looks up `variants.<template-ref>.<variant-id>`.
4. Fails if the selected variant is missing.
5. Collects selected `locked-inputs`, defaulting to an empty mapping when
   omitted.
6. Rejects any CLI flag whose input name is locked by the selected variant.
7. Builds the supplied input map as `inputs.<template-ref>` overwritten by CLI
   flags, then overwritten by selected `locked-inputs`.
8. Lets the engine apply template-author defaults and validate final effective
   inputs.

The effective precedence for variant runs is locked inputs over CLI flags over
baseline config inputs over template-author defaults. The implementation must
still perform the CLI conflict pre-check: invocation-time flags for locked keys
fail instead of being silently overridden. Baseline config duplicates under
`inputs.<template-ref>` are allowed and are overridden by the locked value
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P2, P3, P18).

Every selected `locked-inputs` key must correspond to a declared template input.
The implementation may enforce this through the existing engine validation path
after merging selected locked inputs, rather than duplicating input-domain
validation in the CLI
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P7, P11).

### Generate Agent Skill Behavior

Normal `jastr generate agent-skill <template-ref>` keeps existing behavior and
requires valid base `targets.agent-skill.frontmatter` metadata in the template.
It does not inspect `.jastr/config.yml` for wrapper command decisions.

`jastr generate agent-skill <template-ref>#<variant-id> --out <path> [--force]`
is variant-aware:

- It resolves and validates the selected variant before writing.
- It constructs final Agent Skill frontmatter by starting from base
  `targets.agent-skill.frontmatter` when present, or an empty object when absent
  for selected variant generation only. It then shallowly overlays selected
  `variants.<template-ref>.<variant-id>.agent-skill.frontmatter`, if present,
  and validates the merged result with the normal Agent Skill metadata and
  reserved-field rules. A variant may override only `name`, only `description`,
  only extra fields, or any combination. Nested fields are replaced as whole
  values, not deep-merged
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P4, P15, P19).
- If the merged frontmatter lacks required Agent Skill metadata, generation
  fails through the existing target metadata validation path. This means a base
  template without `targets.agent-skill` can still support variant generation
  only when the selected variant provides enough frontmatter to make the merged
  result valid
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P19).
- Static render validation uses real selected `locked-inputs` values for locked
  keys and synthetic sample values only for unlocked inputs, so generation
  validates branches the variant can actually render
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P10).
- The generated wrapper command is `jastr run <template-ref>#<variant-id>
  $ARGUMENTS` only when the base template declares at least one input that is
  not locked by the selected variant. If all base-template inputs are locked,
  the wrapper command is `jastr run <template-ref>#<variant-id>` with no
  `$ARGUMENTS`
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P9).
- Generated wrapper bodies stay minimal: frontmatter, the command, and the
  existing failure instruction. They do not list unlocked inputs
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P21).

### Help Text And Stable Errors

Run and generate help must mention variant refs concisely. Help acceptance is
semantic, not exact-copy: tests and docs should verify that help describes named
template refs, variant refs using `#` syntax, and direct `.md` file paths, but
they must not pin the exact help sentence unless a later spec explicitly makes
help copy stable.

The intended argument-description wording is:

```text
Template id, template variant (<id>#<variant>), or .md file path
```

Equivalent concise wording is acceptable when it preserves those three concepts
and includes a visible `#` variant example. Detailed examples belong in README
and living docs, not CLI help
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P12).

All failures continue to print `Error: <message>` to stderr with exit code 1.
Variant-specific public messages are:

- Invalid template or variant reference:
  `Template reference <ref> must be a template id, a group/template id, a template id#variant id, a group/template id#variant id, or a .md file path.`
- Missing selected variant:
  `Variant <template-ref>#<variant-id> was not found in .jastr/config.yml.`
- Non-mapping `variants`:
  `.jastr/config.yml variants must be a mapping.`
- Non-mapping selected template variants entry:
  `.jastr/config.yml variants.<template-ref> must be a mapping.`
- Non-mapping selected variant:
  `.jastr/config.yml variants.<template-ref>.<variant-id> must be a mapping.`
- Non-mapping selected locked inputs:
  `.jastr/config.yml variants.<template-ref>.<variant-id>.locked-inputs must be a mapping.`
- CLI locked conflict:
  `Input --<input-name> is locked by variant <template-ref>#<variant-id>.`
- Unknown selected variant field:
  `.jastr/config.yml variants.<template-ref>.<variant-id> field <field> is not supported.`
- Non-mapping selected `agent-skill`:
  `.jastr/config.yml variants.<template-ref>.<variant-id>.agent-skill must be a mapping.`
- Unknown selected `agent-skill` field:
  `.jastr/config.yml variants.<template-ref>.<variant-id>.agent-skill field <field> is not supported.`
- Non-mapping selected variant frontmatter:
  `.jastr/config.yml variants.<template-ref>.<variant-id>.agent-skill.frontmatter must be a mapping.`

These messages are exact contracts for tests and docs
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P8, P20).

Error code mapping:

- Invalid variant ref grammar uses `invalid_template_reference`.
- Malformed `variants` structures and unknown selected config fields use
  `invalid_config`.
- Merged Agent Skill frontmatter validation failures use the existing
  `invalid_target_metadata` or `missing_target_metadata` categories, as
  applicable.
- CLI flag conflicts with locked inputs use a new `locked_input_flag` code.
- Missing selected variants use a new `variant_not_found` code
  (per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P23).

## Constraints And Active Contract Updates

Package boundaries remain unchanged. `@jastr/engine` continues to own template
schema validation, input validation, effective defaults, rendering, and public
error types. It must not learn about `.jastr/config.yml`, variant lookup,
filesystem project discovery, Commander, Agent Skill output paths, or generated
wrapper construction. `@jastr/cli` owns variant ref parsing, project config
loading, selected-only variant config validation, locked input merge/conflict
logic, help text, stable CLI errors, and Agent Skill generation.

The implementation should reuse existing config/default mechanics where they
fit, but it must not model locked values as ordinary defaults. Locked inputs are
an explicit selected variant layer with conflict checks against CLI flags.

This spec updates earlier active contracts as follows
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P14):

- Canonical named refs now include an optional `#<variant-id>` suffix only for
  named templates. Direct `.md` refs remain non-variant.
- Named variant runs may merge baseline `inputs.<template-ref>` with selected
  `variants.<template-ref>.<variant-id>.locked-inputs`.
- Variant generated wrappers are allowed to inspect `.jastr/config.yml` and
  selected locked inputs to decide whether `$ARGUMENTS` belongs in the wrapper.
  This is a variant-specific exception to the previous generation rule that
  wrapper content does not inspect project config.
- Generated Agent Skill wrappers still forward `$ARGUMENTS` for normal
  input-bearing templates, but variant wrappers forward `$ARGUMENTS` only when
  at least one base-template input remains unlocked.

Requirement ownership should follow the mixed organization from the discussion:
add `packages/cli/requirements/functional/12-variants.yml` for variant-only
behavior, update `06-generate.yml` for variant generation effects, and update
`07-help.yml` for help text. Avoid duplicating one acceptance criterion across
multiple requirement files
(per `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/discussions/2026-06-14-generated-skill-variants-design-discussion.md` P13).

Implementation must update `AGENTS.md` whenever it changes behavior described
there, especially claims about canonical commands, config precedence, and
generated wrapper `$ARGUMENTS` behavior. It must also update `README.md` for the
new user-facing config and command syntax, regenerate
`packages/cli/docs/BEHAVIOR.md`, and keep public docs aligned with e2e cases.

## Unresolved Questions

No product-shape question blocks this v2 contract. Internal implementation
details remain intentionally open: helper names, exact parser return types,
test case slugs, and whether validation helpers are shared between run and
generate. Those choices must not alter the public behavior, error messages,
error codes, docs, or acceptance criteria defined here.

## Acceptance Guidance

The implementation is correct when a reviewer can verify all of the following:

- `jastr run demo#custom` resolves `demo`, selects
  `variants.demo.custom`, merges inputs with locked values at highest effective
  precedence, and rejects a CLI flag for any locked key with the exact locked
  input error.
- `inputs.demo` values that duplicate selected `locked-inputs` do not error and
  are overridden by locked values.
- `jastr run demo` never selects a variant, even when `variants.demo.default`
  exists.
- Empty selected variants behave as explicit aliases over the base named
  template.
- Grouped refs such as `team/review#deep` work with exact config keys
  `variants["team/review"].deep`.
- Malformed variant refs, direct `.md` variant refs, missing variants,
  malformed selected config shape, unknown selected fields, malformed selected
  `agent-skill`, and locked CLI conflicts all produce the exact messages and
  error codes in this spec.
- Selected `locked-inputs` values are validated through the engine against the
  base template's declared inputs, types, enum values, and required/default
  behavior; unknown locked keys cannot succeed.
- Malformed unrelated variant entries do not break commands for another
  selected variant.
- `jastr generate agent-skill demo#custom` writes a wrapper that calls
  `jastr run demo#custom`, uses merged variant/base Agent Skill frontmatter,
  omits `$ARGUMENTS` when all declared inputs are locked, includes `$ARGUMENTS`
  when any declared input remains unlocked, and uses locked values during static
  render validation.
- Normal non-variant `run` and `generate agent-skill` behavior remains covered
  and unchanged except for help/reference wording that now acknowledges variant
  syntax.
- Run and generate help make variant refs discoverable by mentioning `#` syntax
  and the three accepted reference families: named template refs, named variant
  refs, and direct `.md` file paths. Help tests do not require exact copy.
- Functional requirements are updated in `12-variants.yml`, `06-generate.yml`,
  and `07-help.yml` as described above, with e2e cases covering each new public
  behavior.
- `README.md`, `AGENTS.md`, and `packages/cli/docs/BEHAVIOR.md` describe the
  same active contract.

Before the feature is considered clean, the standard project checks should pass:

```bash
bun run check
bun run typecheck
bun run test
bun run test:cli:e2e
bun run docs:cli:living --check
bun run build
```
