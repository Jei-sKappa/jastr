# Decision log — home-directory (global) `.jastr` support (the seed)

Thread: docs/threads/260621200326Z-home-global-jastr-dir/
Target: the seed (seed/260621200326Z-home-global-jastr-dir-seed.md)
Subject: resolving how a home (global) `.jastr` and a project-local `.jastr` coexist — lookup order, precedence, and how templates/config/includes compose across the two.

## P1: Coexistence model — layered, modeled on agent-skill discovery

Point: The fork that drives everything downstream — are home (global) and project-local `.jastr` *either/or* (global is only a fallback when no local project is found), or *layered* (both active at once, local extends/overrides global)?

What you need to know: Discovery today (`packages/cli/src/fs/project-root.ts`) walks up from cwd and returns the first directory containing `.jastr/`; there is exactly one project root, ever. Named/grouped templates resolve under `<projectRoot>/.jastr/...` (`templates/template-ref.ts`); `config.yml` is read from that one root (`config.ts`); includes are bounded by realpath to a template/group root, so each template is already self-contained.

The either/or (fallback) model is far simpler — still one active root, home consulted only as a last resort, no merging or precedence. The layered (compose) model is more powerful but pulls in precedence rules at the template-id and config-key level, plus the include-across-boundary question.

Decision: Layered (compose), modeled on how agent-skill discovery works, with three confirmed mechanical points: (1) both `~/.jastr` and the project-local `.jastr` are active on every invocation — the template namespace is the union of the two; (2) local shadows global on a template-id/group collision (project is "closer," same as a project `.claude/skills/x` overriding a user-level one); (3) roots stay self-contained — a global template's includes resolve inside the global root, a local template's inside the local root, with no cross-boundary include (the current containment model is untouched).

Rationale: The user's mental model is explicitly "act just like agent-skills do," which is the layered model. It cleanly settles the template namespace (union, local-wins). Trade-off accepted: the analogy does NOT settle `config.yml`/variant composition or the home-is-the-project edge case (discovery is an upward walk that can land on `~/.jastr` and conflate global with local) — those were flagged as the next decision points to resolve, since agent-skills have no config-merge analog and use a fixed user-config location rather than an upward walk.

## P2: Discovery mechanics — locating the global root and reconciling it with the upward walk

Point: How are the two roots actually found, and how do global (home) and the existing upward-walk local discovery reconcile? This is foundational — config composition can't be settled until we know how the roots are located.

What you need to know: Today there's one root from the upward walk (`packages/cli/src/fs/project-root.ts`), and that walk throwing `missing_project_root` at the filesystem root is the only "not found" path. The seed explicitly says `~/.jastr` (not XDG, not `~/.config`), so the location is settled. What's left is coexistence with the walk. Proposed mechanism: Global := `os.homedir()/.jastr` located by absolute path (never by walking), active whenever it exists; Local := the upward walk from cwd, exactly as today; same-realpath collapse — if the walk lands on the home directory itself (project literally at `~`, or cwd under `~` with no nearer `.jastr`), local and global resolve to the same realpath and are treated as ONE root applied once (no self-shadowing, no self-merge). Two genuine forks rode on top. Fork A: does "global-only" become a valid run context? Today an out-of-project cwd with no `.jastr` up to `/` gives `missing_project_root`; the seed's pitch ("available from any project, not just a local one") implies a run with only `~/.jastr` should now succeed global-only, relaxing `missing_project_root` to fire only when NEITHER local nor global exists. Fork B: add a `JASTR_HOME` env override (global := `$JASTR_HOME/.jastr`, defaulting to `os.homedir()`), motivated primarily by testability — the e2e harness copies fixtures into temp roots and there's no clean way to fake `os.homedir()` otherwise — with agent-skills' `CLAUDE_CONFIG_DIR` as precedent; an explicit override only relocates the base and changes none of the other rules.

Decision: Adopt the proposed mechanism as stated, plus both forks: Fork A — global-only is a valid run context, so `missing_project_root` fires only when neither a local project `.jastr` nor the global `~/.jastr` (or `$JASTR_HOME/.jastr`) exists. Fork B — add the `JASTR_HOME` env override relocating the home base, defaulting to `os.homedir()`.

Rationale: The core mechanism has no defensible alternative once P1 is fixed (global must be located absolutely, not by the walk, or it would be conflated with local; same-realpath collapse is the only sane handling of project-at-home). Fork A is required by the seed's stated intent rather than contested. Fork B is justified by Design-for-Testability (a stated project principle) — there is no other clean way to test global behavior without writing to the real `~` — with the power-user escape hatch as a secondary benefit. Trade-off accepted for B: one additional config surface.

## P3: How `config.yml` composes across global and local

Point: A template body resolves from local-or-global by ref (P1), but config is keyed by the same ref string in potentially two `config.yml` files (`~/.jastr/config.yml` and the local one). How do the two configs compose? Agent-skills give no guidance — skills have no merging config file.

What you need to know: Inputs already compose per-key today — `executeRun` (`packages/cli/src/commands.ts`) builds `{ ...configInputs, ...flagInputs }`, so the chain is CLI flags > config values > template defaults, per input key, and the engine then validates the final effective map. The engine rejects any supplied input key the template does not declare (`unknown_input`, `packages/engine/src/inputs.ts:16`). Variants are selected as a unit (`variants.<ref>.<id>` → `{ locked-inputs, agent-skill.frontmatter }`), with `locked-inputs` applied on top of everything.

Two shapes were weighed. Option A (two-layer per-key overlay / full compose): both `config.yml` files always consulted for any ref, independent of where the template body resolved; inputs merge per-key CLI flags > local config > global config > template defaults; variants resolve as a UNIT — local `variants.<ref>.<id>` shadows global's wholesale if both exist, else whichever exists is used, then the selected variant's `locked-inputs` apply as today. Pros: unlocks global-template-with-per-project-default-overrides (the feature that makes a global install worth more than symlinking a folder), natural continuation of the existing per-key chain, "local over global" consistent with P1. Cons: the `unknown_input` edge — a local template shadowing a global one with a different input schema can pull a stale global `inputs.<ref>` key into the effective map and error against the local schema (same failure mode that already exists for stale config, new only in that two files can contribute); two files to reason about. Option B (config travels with the template's resolved root, no cross-layer merge): global template → global config only, local template → local config only. Pros: simplest mental model, most faithful "self-contained bundle" reading of the skills analogy, no cross-root `unknown_input` surprise. Cons: kills the per-project-override feature (you'd have to copy the whole template locally to retune its defaults); reduces "config available from any project" to just "global config for global templates."

Decision: Option A — the two-layer per-key overlay. Both config files are always consulted for any ref; inputs merge CLI flags > local config > global config > template defaults; variants shadow as a unit (local wholesale over global), not by per-key merge of `locked-inputs` across the two configs.

Rationale: The per-project-override capability is the main reason a layered global install beats symlinking a template folder around, and Option B cannot express it. The `unknown_input` edge is real but narrow (requires same-ref-different-schema shadowing, an unusual and self-inflicted setup) and fails loudly with an existing accurate error rather than doing something silently wrong. Option A keeps one consistent precedence story with P1. Variant unit-shadowing (part of A as accepted) avoids confusing partial-variant states from cross-config `locked-inputs` merges and matches "local shadows global" at the natural unit of a named variant.

## P4: How global resolution surfaces in CLI output and errors

Point: The CLI renders template paths relative to cwd and `template_not_found` names a single searched location. Both assumptions break for a global template (it lives outside cwd, and "not found" now means absent from both roots). How should a globally-resolved template be displayed in output and errors?

What you need to know: The CLI renders paths relative to cwd — `path.relative(template.cwd, template.templatePath)` for the render `sourceId` and the `generate` success message (`packages/cli/src/commands.ts:93,186`) — and `template_not_found` hardcodes one location, e.g. `Template foo was not found at .jastr/foo/TEMPLATE.md` (`packages/cli/src/templates/template-ref.ts:162`). For a global template, relative-to-cwd renders as a fragile depth-dependent `../../../.jastr/foo/TEMPLATE.md` that also leaks directory depth.

An initial proposal to render global paths with a `~/` prefix was rejected on two correct objections from the owner: `~` is not a home shortcut on Windows (`os.homedir()` returns `C:\Users\<name>`; cmd.exe never expands `~`, PowerShell only sometimes), and once `JASTR_HOME` overrides the base, `~/` is an outright lie. A follow-up logical-tag idea (e.g. `<global>/.jastr/...`) was then weighed but set aside: its motivations (avoid leaking the home path, keep test output reproducible) do not apply here — the generated wrapper embeds the ref string `template.requestedTemplateRef`, never the filesystem path, so paths appear only in transient stdout/stderr (and this is a private single-owner tool); and test reproducibility is already solved by the harness `substitute:` mechanism, which has a `projectRoot` token for absolute-path cases. The real absolute path is already computed — `loadTemplateReference` runs `realpath()`, so `template.templatePath` is the canonical symlink-resolved path.

Decision: Display a globally-resolved template's real absolute (realpath-resolved) path in CLI output and errors. Local-resolved templates stay cwd-relative, unchanged. The asymmetry is intentional: local is "right here" (relative, matching the existing "paths relative to where you stand" convention), global is "over there" (the full path says exactly where), and in a message listing both the leading `/` or drive letter distinguishes them. `template_not_found` (absent from both roots) names both the local relative path and the global absolute path. Tests gain one new `substitute` token for the global root, mirroring the existing `projectRoot` token, so e2e output stays machine-independent.

Rationale: The owner pushed for honesty, and the real path is the honest, portable, actionable, lowest-magic answer (KISS): the OS yields a correct native path on every platform, it is exactly right under `JASTR_HOME`, it needs no invented notation, and it is already in hand. The recommendation was updated from the logical-tag lean to the absolute-path choice on the merits — the tag's motivating costs were shown not to apply to this tool. Trade-offs accepted: more verbose output than a relative path or tag, and a local-relative/global-absolute mix within a single message (judged clarifying rather than confusing, since the path shape itself signals which root); plus one new closed-set `substitute` token to keep tests deterministic.
