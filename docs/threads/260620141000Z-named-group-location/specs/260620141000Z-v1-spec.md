# Named group location realignment — v1 spec

## Intended outcome

After this change, a named template group lives **inside** the project's
`.jastr/` directory at `.jastr/<group>/templates/<template-id>/TEMPLATE.md`,
alongside the standalone templates at `.jastr/<template-id>/TEMPLATE.md`. Running
`jastr run <group>/<template-id>` (and `generate` / `validate` against the same
ref) resolves the template from that location. The user-visible command surface
does not change at all — only where the group's files must live on disk.

The point is a single, predictable home for everything named: a user who has
learned that `jastr run demo` maps to `.jastr/demo/…` can now correctly predict
that `jastr run team/demo` maps to `.jastr/team/…`. The current split — where
standalone named templates live *inside* `.jastr/` but grouped named templates
live *beside* it at the project root — is removed.

## Context

Jastr resolves a *named* reference by walking up from the cwd to find the
project root (the nearest ancestor containing a `.jastr/` directory;
`packages/cli/src/fs/project-root.ts`). Today the two named families then diverge
(`packages/cli/src/templates/template-ref.ts`):

- **Standalone** `jastr run <id>` resolves to
  `<project-root>/.jastr/<id>/TEMPLATE.md` — inside `.jastr/`.
- **Grouped** `jastr run <group>/<id>` resolves to
  `<project-root>/<group>/templates/<id>/TEMPLATE.md`, guarded by
  `<project-root>/<group>/.jastrgroup` — **beside** `.jastr/`, at the project
  root, with `.jastr/` used only as the discovery anchor.

This asymmetry was not a deliberate design choice; it surfaced while implementing
the `validate` command. The `validate-grouped` e2e case had to ship an otherwise
inert `.jastr/config.yml` purely to materialize a `.jastr/` anchor, even though
none of the grouped template's files live under `.jastr/`. That phantom anchor is
a direct symptom of the location split. Because the project is private and
pre-release with no external users, the location can be corrected directly rather
than carrying a compatibility shim.

Two design decisions were settled in the originating session and are inlined into
the operative sections below: (a) named groups move under `.jastr/`; (b) the
`.jastrgroup` marker is retained and **required** in both named and direct access
even though it is logically redundant for named access, because direct access has
no invocation-level signal of group intent — the marker is its only intent
carrier — and a uniform requirement keeps a group a self-contained, relocatable
unit that runs identically by name or by direct path.

## Scope

In scope:

- Changing **named grouped** lookup so the group root is
  `<project-root>/.jastr/<group>/`, i.e. the template resolves to
  `<project-root>/.jastr/<group>/templates/<template-id>/TEMPLATE.md` and the
  marker is checked at `<project-root>/.jastr/<group>/.jastrgroup`.
- Updating the `template_not_found` message for named grouped misses to report
  the new `.jastr/<group>/templates/<id>/TEMPLATE.md` location.
- Realigning the affected **named** grouped e2e fixtures and integration tests to
  the new on-disk layout, and removing the now-unnecessary phantom anchors they
  carried.
- Updating the docs and requirements that assert the old grouped layout:
  `AGENTS.md`, `README.md`, `packages/cli/requirements/functional/01-run.yml`,
  the regenerated `packages/cli/docs/BEHAVIOR.md`, and a supersession note on the
  include-group-containment v2 spec's named-grouped-lookup rule.

Explicitly out of scope:

- **Direct `.md` access.** `jastr run <path>.md` and its grouped classification
  (`classifyDirectTemplate`) are unchanged. A direct grouped template is still any
  realpath of the exact form `<group>/templates/<id>/TEMPLATE.md` with
  `<group>/.jastrgroup` present, located **anywhere**, requiring no `.jastr/`.
  Leaving the direct grouped e2e cases structurally untouched is itself the proof
  that direct classification is unaffected.
- **The reference grammar and CLI surface.** `jastr run <group>/<id>`, the
  `#<variant-id>` suffix, one-segment standalone refs, and the invalid-reference
  rules all stay exactly as specified. Only the on-disk resolution target moves.
- **Project config keys.** Because the user-facing ref string `<group>/<id>` is
  unchanged, `inputs.<group>/<id>`, `variants.<group>/<id>`, and their selection
  precedence are unchanged. No config migration is implied.
- **The include containment model.** The realpath-containment rules, the three
  `root` selectors (`template`, `group`, `file`), and the group-root boundary
  semantics are unchanged; only the *filesystem location* of a named group's root
  moves under `.jastr/`. `root="group"` for a named grouped template continues to
  resolve from, and be contained by, the group root — now `.jastr/<group>/`.
- **The `.jastrgroup` marker's role and contents.** It remains an existence-only
  marker (contents ignored) and remains required for every group.
- **Standalone named resolution.** `.jastr/<id>/TEMPLATE.md` is untouched.

## Expected behavior

1. **Named grouped resolution.** For `jastr run <group>/<id>` (and `generate` /
   `validate` against the same ref, with or without a `#<variant-id>` suffix),
   the loader finds the project root by the existing upward `.jastr/` walk, then
   resolves the template at
   `<project-root>/.jastr/<group>/templates/<id>/TEMPLATE.md` and requires the
   marker at `<project-root>/.jastr/<group>/.jastrgroup`. Both segments must
   satisfy the existing template-id grammar; reference-shape validation is
   unchanged.

2. **Group root and includes.** The resolved group root for a named grouped
   template is `<project-root>/.jastr/<group>/` (its realpath). Includes with
   `root="group"` resolve from there and every include's resolved realpath must
   remain within that boundary, exactly as the containment v2 spec requires — the
   only change is that the boundary now sits under `.jastr/`.

3. **Not-found message.** When the marker or the template file is missing at the
   new location, the existing `template_not_found` failure reports the new path:
   `Template <group>/<id> was not found at .jastr/<group>/templates/<id>/TEMPLATE.md.`
   The standalone not-found message
   (`Template <id> was not found at .jastr/<id>/TEMPLATE.md.`) is unchanged.

4. **Marker required, both modes.** A named group with no
   `.jastr/<group>/.jastrgroup` is not found (the message above). This mirrors
   direct access, where a `<group>/templates/<id>/TEMPLATE.md` layout without a
   sibling-root `.jastrgroup` is classified standalone. The marker stays required
   even though, for named access, the ref shape plus the fixed layout already
   identify the group; the redundancy is accepted for cross-mode consistency and
   group portability.

5. **A name may be both a standalone template and a group.** Because both
   families now live under `.jastr/`, the same directory name can host both:
   `jastr run foo` resolves the standalone `.jastr/foo/TEMPLATE.md`, while
   `jastr run foo/bar` resolves the grouped
   `.jastr/foo/templates/bar/TEMPLATE.md` (with `.jastr/foo/.jastrgroup`). The two
   are disambiguated by reference arity (one segment vs two), consistent with the
   existing containment rule that a `TEMPLATE.md` beside a `.jastrgroup` is itself
   standalone. No new error is introduced for this coexistence.

6. **Old location no longer resolves.** A group at the old project-root location
   `<project-root>/<group>/templates/<id>/TEMPLATE.md` (not under `.jastr/`) is no
   longer found by a named ref and fails with the not-found message in (3).

## Constraints

- **Single code touchpoint for the path.** The only place that constructs the
  named group root is `packages/cli/src/templates/template-ref.ts:191`
  (`loadGroupedNamedTemplate`). The realignment is inserting the `.jastr` segment
  into that join plus updating the corresponding not-found message string.
  `classifyDirectTemplate` derives the group root from the supplied path's own
  ancestry and must remain location-agnostic and unchanged.
- **Engine purity.** `@jastr/engine` must not learn about `.jastr`, groups,
  project roots, markers, or filesystem layout. Group discovery, root selection,
  and classification stay in `@jastr/cli` (per the package-split v2 contract and
  the include-group-containment v2 spec).
- **Supersedes one prior rule.** This spec supersedes only the
  *named*-grouped-lookup location rule of the include-group-containment v2 spec at
  `docs/threads/260607220555Z-include-group-containment/specs/260608092033Z-v2-spec.md`
  (its "Named access is grouped iff invoked as `jastr run <group>/<template-id>`
  … `<group>/.jastrgroup` exists … `<group>/templates/<template-id>/TEMPLATE.md`
  exists" clause now resolves `<group>` as `.jastr/<group>` relative to the
  project root). Every other clause of that spec — direct classification, marker
  semantics, `root` selectors, containment, error contract — stands.
- **Runtime portability.** Use Bun for tooling, but avoid Bun-specific runtime
  APIs in CLI source so the package stays Node-compatible.
- **Error UX unchanged.** Failures still print `Error: <message>` to stderr with
  exit code 1; no new `JastrErrorCode` values are introduced — the move reuses the
  existing `template_not_found` code with the updated path text.
- **Thread workflow.** Work proceeds inside this thread
  (`docs/threads/260620141000Z-named-group-location/`); the living CLI behavior
  doc `packages/cli/docs/BEHAVIOR.md` must be regenerated, not hand-edited.

## Unresolved questions

These do not block implementation:

- **Deeper nesting under `.jastr/`.** This spec keeps a group at exactly one
  segment below `.jastr/` (`.jastr/<group>/`), matching the single-slash named
  grammar. Whether a future version permits multi-segment group paths under
  `.jastr/` is left open.
- **Direct-access unification.** Whether direct `.md` access should eventually
  also resolve relative to a discovered project root (rather than the supplied
  path) is a separate, larger question and is not addressed here.
- **Marker requirement for named access.** Recorded as settled (required in both
  modes) for consistency and portability; a future spec could revisit making it
  optional-but-honored for named refs without affecting direct access. Flagged
  only so the decision is visible, not because it is open for this version.

## Acceptance guidance

A reviewer should confirm:

- `jastr run <group>/<id>`, `jastr generate agent-skill <group>/<id> …`, and
  `jastr validate <group>/<id>` all resolve a template at
  `.jastr/<group>/templates/<id>/TEMPLATE.md` with `.jastr/<group>/.jastrgroup`,
  including the `#<variant-id>` variants of those refs.
- A named grouped template placed at the **old** project-root location (not under
  `.jastr/`) is not found and fails with
  `Template <group>/<id> was not found at .jastr/<group>/templates/<id>/TEMPLATE.md.`
  — there is at least one regression case asserting this.
- A missing marker at `.jastr/<group>/.jastrgroup` produces the same not-found
  message.
- `root="group"` includes for a named grouped template resolve from, and are
  contained by, `.jastr/<group>/`; an include escaping that root is still
  rejected by the existing containment error.
- Direct grouped access is demonstrably unchanged: the direct grouped e2e cases
  (`direct-grouped-template-include` and the `include-*` escape/standalone cases)
  pass without structural fixture changes.
- The five **named** grouped e2e cases (`grouped-named-run`,
  `variant-run-grouped`, `validate-grouped`, `validate-grouped-variant`,
  `config-grouped-template-key`) are realigned to `.jastr/<group>/…`, and the
  phantom anchors are gone — `grouped-named-run` no longer ships an unrelated
  `.jastr/demo/TEMPLATE.md`, and `validate-grouped` no longer ships an inert
  `.jastr/config.yml` (the group directory now materializes `.jastr/` on its own).
- `AGENTS.md`, `README.md`, and `packages/cli/requirements/functional/01-run.yml`
  describe the new layout, and `packages/cli/docs/BEHAVIOR.md` has been
  regenerated to match.
- The full local gauntlet exits 0: `bun run check`, `bun run typecheck`,
  `bun run test`, `bun run test:cli:e2e`, `bun run docs:cli:living --check`, and
  `bun run build`.
