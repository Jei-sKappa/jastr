# Functional Requirements Registry With E2E Case Coverage V3 Spec

## Intended Outcome

This spec replaces the VitePress documentation-site-backed executable examples
approach from v2 with a cleaner product-contract and test architecture:

- a stable functional requirements registry that names what Skillrouter promises;
- executable e2e cases that prove those requirements through the real CLI;
- traceability checks that link active requirements to e2e coverage.

When implemented, the project owner can ask an AI agent to add, change, remove,
or audit a functional requirement by stable ID. The test suite can then verify
whether that requirement is covered by executable behavior. The immediate goal
is not final-user documentation. The immediate goal is a reliable, readable,
agent-friendly product contract with executable regression evidence.

## Context

The original goal in this thread was executable UX regression coverage for
Skillrouter, a private early-stage CLI for deterministic AI-agent skill
specialization. The v1 spec proposed executable Markdown case blocks. The v2
spec replaced that with a final-user VitePress documentation site backed by
executable examples.

That v2 implementation proved the value of executable examples, but it also
introduced the wrong ownership boundary for the current phase of the project.
The example artifacts became tightly coupled to the documentation site: they
contained presentation concerns, site rendering metadata, and validation rules
about whether each example appeared in `docs/site`. The user has clarified that
this coupling is not acceptable yet. The project does not need a documentation
site before the behavioral contract is mature.

The settled v3 direction is:

- remove the documentation-site layer for now;
- stop calling the executable artifacts "examples";
- treat them as e2e cases under `test/`;
- add a first-class functional requirements registry with stable IDs;
- link e2e cases to functional requirement IDs.

This is a deliberate change from v2: the site is not the source of safety. The
functional requirements registry plus e2e cases are the source of safety.

## Scope

V3 includes:

- Removing the current VitePress documentation-site implementation and its
  docs-example coupling.
- Introducing a functional requirements registry at
  `requirements/functional-requirements.yml`.
- Defining stable functional requirement IDs for the existing Skillrouter v1
  CLI behavior.
- Migrating existing executable docs examples into e2e cases under
  `test/e2e/cases/`.
- Removing site-specific fields from executable case manifests, including
  `render`, `hidden`, and any docs-reference requirement.
- Adding an e2e runner that executes every case through the real Skillrouter CLI
  from an isolated temporary project fixture.
- Adding traceability validation that verifies e2e cases reference real
  functional requirements and active functional requirements are covered.
- Adding `bun run test:e2e` as a focused command for the e2e suite.
- Keeping the existing Vitest unit and integration tests intact.
- Updating `AGENTS.md` and `README.md` so future agents understand the new
  requirement/e2e model and do not recreate the docs-site coupling.

V3 does not include:

- Rebuilding or preserving the VitePress documentation site.
- Generating a final-user documentation site from the requirements registry.
- Rendering e2e cases into Markdown pages.
- Site-reference validation such as "every case must be referenced by docs".
- BDD/Gherkin/Cucumber.
- Browser automation or visual regression tests.
- A database or external requirements-management tool.
- A public multi-version requirements portal.
- A new Skillrouter product command.
- Refactoring core Skillrouter runtime behavior unless an existing e2e case
  exposes behavior that contradicts the current v1 CLI contract.

## Architecture

The new architecture has three separate layers:

```text
requirements/functional-requirements.yml
  Stable product contract. Names what Skillrouter promises.

test/e2e/cases/<case-id>/case.yml
  Executable behavior cases. Prove requirements through the real CLI.

test/e2e/*.ts
  Loader, schema validation, runner, and traceability tests.
```

The dependency direction is one-way:

```text
e2e cases reference requirement IDs
test harness reads requirements and cases
requirements do not reference test files
cases do not reference any documentation site
```

Functional requirements and e2e cases are intentionally separate. A requirement
is the stable product promise; an e2e case is executable evidence. Tests and
fixtures can be renamed, split, or reorganized without changing the requirement
ID. Requirement IDs must be stable enough for the project owner to use them in
agent prompts such as "remove FR-CLI-GEN-003" or "add a new requirement for
optional enum inputs."

## Functional Requirements Registry

Create `requirements/functional-requirements.yml` as the canonical registry.
The file must be plain YAML with a top-level sequence of requirement records.

Each requirement record must support:

```yaml
- id: FR-CLI-RUN-001
  title: Run renders a skill template
  status: active
  description: >
    The CLI renders `.skillrouter/<skill>/SKILL.template.md` when the user runs
    `skillrouter run <skill>` from inside a project.
  acceptance:
    - Exits with code 0 when the skill exists and required inputs are valid.
    - Prints rendered Markdown to stdout.
    - Prints nothing to stderr.
```

Required fields:

- `id`: stable functional requirement ID.
- `title`: short human-readable requirement name.
- `status`: one of `active`, `deferred`, or `removed`.
- `description`: product-level behavior statement.
- `acceptance`: non-empty list of observable acceptance statements.

Optional fields:

- `notes`: additional product context.
- `replacedBy`: requirement ID that supersedes a `removed` requirement.
- `removedReason`: explanation for a `removed` requirement.
- `coverage`: for `deferred` requirements, a short reason coverage is deferred.

IDs must use this format:

```text
FR-<AREA>-<FEATURE>-<NNN>
```

Initial areas should match Skillrouter's v1 surface:

- `FR-CLI-RUN-*`
- `FR-CLI-GEN-*`
- `FR-CLI-HELP-*`
- `FR-CLI-VERSION-*`
- `FR-CLI-FLAGS-*`
- `FR-TPL-INPUT-*`
- `FR-TPL-COND-*`
- `FR-TPL-INCLUDE-*`
- `FR-ERR-*`

IDs must never be renumbered. If a requirement is removed, keep the record and
set `status: removed` with `removedReason`. If a requirement is intentionally
not covered yet, set `status: deferred` with `coverage`. Only `active`
requirements are required to have e2e coverage.

The initial registry must cover the user-facing v1 CLI behavior currently
represented by the executable docs examples: successful `run`, string input
interpolation, boolean flags, enum branch selection, missing optional inputs,
missing required inputs, unknown flags, duplicate flags, invalid enum values,
include and include-raw behavior, rejected unsafe includes, `generate`,
overwrite refusal, `--force`, help, and version output.

## E2E Case Layout

Move executable cases under:

```text
test/e2e/cases/
```

Each case directory should contain:

```text
test/e2e/cases/<case-id>/
  case.yml
  project/
  expected/
```

The `project/` directory is copied into a temporary directory before execution.
The e2e runner must execute the real CLI entry point from the temp project,
not call internal rendering functions directly.

Case IDs should be stable enough for test diagnostics but do not need to be as
stable as functional requirement IDs. A good case ID includes the primary
requirement ID and a short descriptor:

```text
FR-CLI-RUN-001-basic-run
FR-TPL-INPUT-002-string-interpolation
FR-CLI-GEN-003-force-overwrite
```

## E2E Case Manifest

Each `case.yml` must contain only behavior and traceability information. It
must not contain site rendering metadata.

Required shape:

```yaml
id: FR-TPL-INPUT-002-string-interpolation
requirementIds:
  - FR-TPL-INPUT-002
title: Interpolate a string input
description: Expands a declared string input in rendered template output.
cwd: project
command: ["run", "demo", "--target-file=src/index.ts"]
expect:
  exitCode: 0
  stdout: "Analyze `src/index.ts`.\n"
  stderr: ""
```

Required fields:

- `id`: unique e2e case ID.
- `requirementIds`: non-empty list of functional requirement IDs covered by the
  case.
- `title`: short human-readable case name.
- `description`: concise explanation of the behavior under test.
- `cwd`: safe relative path inside the copied temp project.
- `command`: non-empty array of Skillrouter CLI arguments, excluding the
  executable name.
- `expect.exitCode`: expected process exit code.
- `expect.stdout` or `expect.stdoutFile`.
- `expect.stderr` or `expect.stderrFile`.

Optional `expect` fields:

- `files`: mapping of generated file path to expected file path.
- `fileContains`: mapping of generated file path to required substrings.
- `fileNotContains`: mapping of generated file path to forbidden substrings.

The manifest schema must reject:

- unknown top-level fields;
- unknown `expect` fields;
- empty IDs, titles, descriptions, commands, or requirement IDs;
- duplicate case IDs;
- duplicate requirement IDs inside one case;
- absolute paths;
- `..` path segments;
- backslash path separators;
- non-string command items;
- setting both `stdout` and `stdoutFile`;
- setting both `stderr` and `stderrFile`;
- missing stdout expectation;
- missing stderr expectation.

The manifest schema must not include `render`, `hidden`, site labels, page
references, or any documentation-specific field.

## Traceability Behavior

Add validation that reads both `requirements/functional-requirements.yml` and
all `test/e2e/cases/*/case.yml` files.

The validation must fail when:

- a requirement ID is duplicated;
- a requirement has an invalid ID format;
- a requirement has an invalid status;
- an `active` requirement has no covering e2e case;
- an e2e case references a requirement ID that does not exist;
- an e2e case references a `removed` requirement;
- an e2e case has duplicate `requirementIds`;
- a `removed` requirement lacks `removedReason`;
- a `deferred` requirement lacks a `coverage` explanation.

The validation may allow an e2e case to cover multiple requirements when a
single CLI workflow genuinely proves more than one product promise. It should
not require one case per requirement.

The validation should produce error messages that name the affected requirement
ID or case ID so AI agents can repair the right record.

## E2E Execution Behavior

Add a Vitest e2e suite under `test/e2e/`.

The runner must:

1. Load and validate the functional requirements registry.
2. Load and validate every case manifest.
3. Validate requirement/case traceability.
4. For each case, create an isolated temp directory.
5. Copy the case's `project/` fixture into that temp directory.
6. Resolve `cwd` inside the temp project, creating it when needed.
7. Execute the real CLI entry point with Bun:

   ```text
   bun <repoRoot>/src/cli/index.ts ...case.command
   ```

8. Compare actual exit code, stdout, stderr, generated files, required
   substrings, and forbidden substrings against the manifest.
9. Clean up the temp directory after execution.

The runner may preserve the current placeholder support for expected output:

```text
{{projectRoot}}
{{cwd}}
```

These placeholders are test harness placeholders only. They are not related to
Skillrouter template interpolation and must be expanded only in expected stdout,
stderr, expected files, and substring assertions.

## Project Scripts And Verification

Add or update package scripts:

```json
{
  "test:e2e": "vitest run test/e2e",
  "test": "vitest run"
}
```

Remove package scripts that exist only for the v2 docs site:

```json
{
  "docs:dev": "...",
  "docs:check": "...",
  "docs:build": "..."
}
```

Remove `vitepress` from `devDependencies` if it is no longer used anywhere.

The normal full verification set after this change should be:

```bash
bun run check
bun run typecheck
bun run test
bun run test:e2e
```

If `bun run test` already includes the e2e suite because it runs all Vitest
tests, `test:e2e` still remains useful as a focused command.

## Removal And Migration Requirements

Remove these v2 documentation-site artifacts unless another active part of the
repo depends on them:

```text
docs/site/
docs/examples/
test/docs/
```

Migrate the existing executable example coverage into e2e cases. Preserve the
behavioral coverage even though the directories and manifests are renamed.

The migration must keep or recreate coverage for at least:

- root help;
- run help;
- version output shape;
- successful basic run;
- string interpolation;
- boolean true, false, and bare boolean flags;
- enum branch selection;
- missing optional condition behavior;
- missing optional interpolation behavior;
- missing required input error;
- unknown flag error;
- duplicate flag error;
- invalid enum value error;
- include fragment behavior;
- include-raw behavior;
- rejected include path escaping;
- rejected `.env` include;
- generate router skill success;
- generate missing `--out`;
- generate overwrite refusal;
- generate `--force` overwrite.

The migration must update `AGENTS.md` to remove claims that docs examples live
under `docs/examples` and docs pages live under `docs/site`. Replace them with
the new functional requirements and e2e case contract.

The migration must update `README.md` if it currently describes the docs site
or docs commands as part of the project workflow.

## Constraints

Use the repo's existing stack:

- Bun for package scripts and CLI execution.
- Vitest for tests.
- TypeScript for harness code.
- YAML for requirement and case manifests, using the existing `yaml`
  dependency if suitable.

Keep this change focused on boundaries and traceability. Do not introduce a
docs generator, Gherkin, Cucumber, a database, a web UI, or a requirements
management service.

The implementation must preserve Skillrouter's current CLI contract unless an
e2e migration reveals a real mismatch with the existing v1 spec. In that case,
the implementer should fix the bug or stop and surface the contradiction rather
than silently changing expected outputs.

The e2e cases must be deterministic and isolated. They must not depend on the
developer's current working directory, global configuration, network access, or
existing files outside the repo and temp directories.

The registry and case files are intended to be edited by AI agents. Prefer
simple schemas, clear error messages, stable IDs, and file names that make
failure output actionable.

## Unresolved Questions

The first implementation can proceed without resolving these:

- Whether the functional requirements registry should eventually move under
  `docs/requirements/` or stay at root `requirements/`.
- Whether future final-user documentation should consume the requirements and
  e2e cases directly or use a separate curated docs source.
- Whether every active requirement must always have e2e coverage, or whether
  some active requirements should support a `coverage: unit` or
  `coverage: integration` exception later.
- Whether removed requirements should remain forever or be archived after a
  period of time.

For v3, use root `requirements/functional-requirements.yml`, require e2e
coverage for every `active` requirement, and keep removed requirements in the
registry.

## Acceptance Guidance

A reviewer should consider the implementation complete when all of the
following are true:

- `requirements/functional-requirements.yml` exists and contains stable
  functional requirement records for the current user-facing v1 CLI behavior.
- E2E cases live under `test/e2e/cases/`, not `docs/examples/`.
- E2E case manifests use `case.yml`, contain `requirementIds`, and contain no
  `render`, `hidden`, or docs-site fields.
- The current executable behavior coverage from v2 has been migrated into e2e
  cases.
- The e2e runner executes cases through the real CLI entry point from isolated
  temp project fixtures.
- Traceability validation fails for missing requirements, invalid requirement
  references, duplicate IDs, uncovered active requirements, removed requirements
  still referenced by tests, and invalid manifest schemas.
- `docs/site/`, `docs/examples/`, and `test/docs/` are removed unless a
  concrete non-site dependency remains and is documented.
- `package.json` no longer exposes `docs:dev`, `docs:check`, or `docs:build`
  for the removed site.
- `vitepress` is removed from `devDependencies` if unused.
- `AGENTS.md` accurately describes the functional requirements and e2e case
  workflow.
- `README.md` does not advertise removed docs-site commands or directories.
- These commands exit 0:

  ```bash
  bun run check
  bun run typecheck
  bun run test
  bun run test:e2e
  ```

- A reviewer can intentionally break a case expectation and see the e2e suite
  fail with the case ID.
- A reviewer can intentionally add an active requirement with no case and see
  the traceability suite fail with the requirement ID.
