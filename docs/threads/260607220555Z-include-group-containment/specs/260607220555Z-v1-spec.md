# Include Group Containment v1 Spec

## Intended Outcome

When this spec is implemented, the `::include` directive resolves the same way
no matter how a template was invoked — there is one include containment model
shared by direct-file rendering (`jastr run <path>.md`) and named-template
rendering (`jastr run <id>`). The model is self-contained and secure by default:
an include can only reach files inside an explicit boundary, and any path whose
final resolved location escapes that boundary is rejected.

The boundary is widened from "a single template's directory" to "a group of
templates" so that shared partials become a first-class, supported capability
without resorting to per-file marker extensions or magic path namespaces.

Concretely, an author gets:

- `::include{path="fragments/a.md"}` — the default, resolved from the template's
  own directory.
- `::include{root="group", path="shared/preamble.md"}` — reaches shared files
  anywhere inside the surrounding group.
- `::include{root="file", path="sibling.md"}` — resolved relative to the file
  where the directive physically appears.
- A guarantee that none of these can read arbitrary files from the project root,
  home directory, or filesystem (`.env`, SSH keys, etc.): every include must
  resolve to a realpath that stays inside the allowed boundary.

## Context

This spec is forward-designed from a settled design review of jastr's include
containment model. The trigger is an inconsistency in current behavior and a
hard product requirement that the previous model could not satisfy.

Current state (as implemented in
`packages/cli/src/templates/template-ref.ts` and
`packages/cli/src/templates/includes.ts`): the two run modes have *different*
include boundaries. A direct-file template is contained to the template file's
own directory (`includeRoot = path.dirname(templatePath)`, label
`"template directory"`); a named template is contained to the discovered project
root that holds `.jastr/` (`includeRoot = projectRoot`, label
`"project root"`). On top of that, includes are governed by a denylist:
absolute paths and `~` paths are rejected as non-relative, `.env`/`.env.*` is
rejected by name (both on the literal input and the resolved basename), and
containment is checked *lexically* (`path.resolve` + `path.normalize`, which do
not follow symlinks).

Two problems drove this redesign. First, the same `::include` directive behaves
differently depending on how the template was invoked, which violates least
astonishment. Second, **shared partials are a hard requirement** — authors must
be able to share a common fragment across multiple templates — and neither a
project-root boundary (inconsistent, and a porous denylist) nor a strict
single-template boundary (no sharing at all) is acceptable.

Two alternatives were evaluated and rejected:

- **Strict self-contained** (unify both modes on the single template's own
  directory): simplest and tightest, but it cannot reach shared partials at all,
  which fails the hard requirement.
- **Capability / marker model** (drop path boundaries; each includable file
  self-certifies via a filename convention such as `name.jastr.md`, matched on
  the resolved realpath): secure-by-default and gives sharing for free, but it
  forces a marker extension onto *every* includable file type, which was judged
  too intrusive.

The settled direction keeps the self-contained spirit but makes the unit of
self-containment a **group** rather than a single template, declared by an empty
`.jastrgroup` marker file, with explicit root selection on the directive.

This spec supersedes the include-containment rules in the v2 spec at
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260605091319Z-core-cli-package-split/specs/260606182808Z-v2-spec.md`.
Specifically it replaces that spec's divergent "project root" vs "template
directory" boundaries (its Run Behavior section), its two-variant
`include_outside_root` message contract and its `include_path_rejected`
absolute/`.env` rules (its Error Code And Message Contract section), and it
reopens that spec's out-of-scope exclusion of "nested template groups" and
"slash-separated template ids" for the group concern only. Where this spec and
v2 disagree about include resolution, **this spec wins**; v2 remains authoritative
for everything else (package split, engine API, `generate agent-skill`,
template/targets schema, version behavior).

## Scope

In scope:

- A single include containment model shared by both direct-file and named
  templates, replacing the current two-boundary behavior.
- Three resolution roots selected by an optional `root` attribute on the include
  directive: `template` (default), `group`, and `file`.
- Group declaration via an empty `.jastrgroup` marker file; the directory that
  contains it is the group root and becomes the containment boundary for every
  template beneath it.
- Realpath-based containment: every include — for any `root` — resolves to a
  realpath, and that realpath must stay inside the allowed boundary.
- Replacing the standalone absolute-path / `~` / `..` rejection rules with
  containment alone, so portability is the author's responsibility rather than a
  jastr-enforced denylist.
- Updating the include error contract, the affected e2e cases
  (`direct-file-include-escape`, `include-escape-rejected`, plus new
  group/`root` cases), `packages/cli/docs/BEHAVIOR.md`, and `AGENTS.md` so they
  describe the unified model.

Explicitly out of scope:

- **Per-file marker extensions** (e.g. `name.jastr.md` for includable files).
  Rejected as too intrusive; this spec deliberately does not adopt it.
- **Magic path prefixes** such as `::include{path="@group/shared.txt"}`.
  Rejected because `@group/` is itself a valid filesystem directory name, which
  makes the syntax ambiguous and would force jastr to permanently reserve a path
  segment authors may legitimately want. Root selection is handled by the
  separate `root` attribute, and `path` stays a normal relative filesystem path.
- **Nested groups** — the behavior of a `.jastrgroup` declared inside another
  group's tree is deferred (see Unresolved Questions).
- **New directives** or any change to the directive set beyond adding the `root`
  attribute to the existing `include` and `include-raw`.
- **A final grammar for named-access addressing of a template that lives inside
  a group** (see Unresolved Questions); this spec settles containment, not the
  named-lookup catalog layout.

## Containment Model

There are three relevant directories ("roots") for any include:

- `template` — the directory containing the top-level `template.md` being run.
- `group` — the directory marked by an empty `.jastrgroup` file, if one exists at
  or above the template.
- `file` — the directory containing the source file in which the current
  `::include` directive physically appears (the top-level template, or a
  fragment pulled in by an earlier include).

The default root is always `template`. These two directives are equivalent:

```md
::include{path="fragments/a.md"}
::include{root="template", path="fragments/a.md"}
```

### Group marker

A group is declared by an empty marker file named `.jastrgroup`. The directory
containing that file is the group root. A directory named `templates/` is **not**
sufficient on its own to create a group — this is a settled decision, so that the
security boundary never changes accidentally just because a folder happens to be
named `templates`.

A grouped template has this shape:

```text
my-templates-group/
  .jastrgroup
  shared/
    hello.txt
  shared-file-in-group-root.txt
  templates/
    my-template-1/
      fragments/
        something.md
      my-file-1.txt
      template.md
    my-template-2/
      template.md
```

A standalone template has no `.jastrgroup` anywhere above it, so it has no group:

```text
my-single-template/
  fragments/
    something.md
  template.md
```

Having `.jastrgroup` with no `templates/` folder is an error.
Having `.jastrgroup` with `template.md` is an error

### Include syntax

Template-local includes (default root) resolve from the template's own
directory:

```md
::include{path="fragments/something.md"}
::include{path="my-file-1.txt"}
```

Group-root includes resolve from the group root:

```md
::include{root="group", path="shared/hello.txt"}
::include{root="group", path="shared-file-in-group-root.txt"}
```

Current-file-relative includes resolve from the directory of the file where the
directive appears:

```md
::include{root="file", path="sibling-of-current-file.md"}
```

`root="file"` changes the *starting directory* for path resolution. It does not
change the *security boundary* (see below).

### Security boundary

Every include — regardless of `root` — resolves to a realpath, and the realpath
is then checked for containment. The allowed boundary is:

- **Standalone template**: the top-level template's own directory.
- **Grouped template**: the directory containing `.jastrgroup`.

Resolution starting points differ by root, but the boundary check is the same
for all three:

- `root="template"` resolves from the template directory.
- `root="group"` resolves from the group directory.
- `root="file"` resolves from the current source file's directory.
- In every case the final realpath must stay inside the allowed boundary.

So for a grouped template, `root="file"` can reach nearby partials elsewhere in
the group but cannot escape the group; for a standalone template, `root="file"`
cannot escape the template directory.

Because containment is enforced on the resolved realpath, absolute paths, `~`,
and `..` need no special denylist semantics: any of them that resolves outside
the boundary is rejected by the single containment check, and any that resolves
*inside* the boundary is allowed. This is a deliberate change from the current
lexical check plus absolute/`~` denylist — portability of a template is the
author's responsibility, not something jastr enforces by banning path shapes.

### Command shapes

Direct access names the template file path:

```bash
jastr run my-single-template/template.md
jastr run my-templates-group/templates/my-template-1/template.md
```

Named access (recommended shape) addresses a template, and the group name must
participate in the address rather than being silently dropped:

```bash
jastr run my-single-template
jastr run my-templates-group/my-template-1
```

The group name must not be silently ignored during named lookup, because a bare
`my-template-1` becomes ambiguous if more than one group contains a template
with that name. (The exact named-lookup grammar and how it reconciles with the
existing `.jastr/<id>/template.md` catalog is an open item — see Unresolved
Questions.)

## Constraints

- **Engine purity.** `@jastr/engine` must not learn about `.jastrgroup`, groups,
  project roots, `cwd`, or filesystem layout. Group discovery, root selection,
  realpath resolution, and containment all live in the CLI's injected include
  resolver (`createFileIncludeResolver` and the loaded template reference), which
  is the only component that touches the filesystem. This preserves the v2
  engine/CLI boundary: the engine still receives an `IncludeResolver` and an
  `IncludeRequest` carrying `path`, `from`, `raw`, and `stack`. The `root`
  attribute is parsed from the directive, so the engine must pass it through to
  the resolver (an additive change to the include request), but the engine must
  not interpret it.
- **Node compatibility.** No Bun-specific runtime APIs in engine or CLI source;
  realpath resolution uses Node `fs` (`fs.realpath`/`realpathSync` family).
- **Realpath is load-bearing.** Containment must be checked after resolving
  symlinks, not lexically. Resolving the realpath is what prevents a symlink
  inside the boundary from pointing out of it.
- **Threat model.** The boundary protects against a template reaching *unintended*
  files outside its group/template; it does not defend against an author who
  deliberately places symlinks or hardlinks to secrets inside the boundary.
  Author-side filesystem manipulation is accepted as out of scope, consistent
  with the design review.
- **Uniform error UX.** Every failure prints exactly one `Error: <message>` line
  to stderr and exits 1, consistent with v2. Only `--help`, `help [command]`,
  and `--version` exit 0.
- **Privacy / churn.** The project is private and pre-release, so the boundary
  and error-message changes may be breaking; no legacy include behavior needs to
  be preserved.

## Questions

### Resolved

- **Named-access grammar vs. the `.jastr/` catalog.** The recommended named shape
  is `jastr run <group>/<template>`, and the group example lays templates out as
  `<group>/templates/<id>/template.md` under a `.jastrgroup` root. This does not
  match the current `.jastr/<template-id>/template.md` named-lookup convention,
  and v2 explicitly excluded slash-separated template ids. How group-aware named
  lookup coexists with (or replaces) the `.jastr/` catalog — and the exact
  reference grammar and id rules — must be settled before named access against a
  group is implemented.
  Answer: given that `/` is not a valid template id character, if there is a `/` it's treaded as `group/template-id`. More than one `/` is an error.
- **Within-boundary secrets.** Under pure realpath containment, a `.env` (or SSH
  key, etc.) that physically lives *inside* the group/template boundary becomes
  includable, whereas today `.env`/`.env.*` is rejected by name regardless of
  location. Decide whether to keep a within-boundary secrets denylist (a small
  belt-and-suspenders rule) or rely solely on containment and on authors not
  placing secrets inside template/group trees:
  Answer: just use the boundary as a security boundary, if a user puts a secret inside the boundary it's their responsibility not ours.
- **`root="group"` for a standalone template.** When a template has no
  `.jastrgroup` above it, `root="group"` has no group to resolve against. Decide
  whether this is an error (`group` requested but no group exists) or falls back
  to the template root:
  Answer: it's an error.
- **Nested groups.** Behavior when a `.jastrgroup` is declared inside another
  group's subtree is undefined: does the nearest `.jastrgroup` win, the outermost,
  or is nesting rejected outright?
  Answer: starting from the template directory and it stops when it finds the first `.jastrgroup`. In theory we should never even have the chance to verify if there are nested groups (I think).
- **Group-marker discovery direction and limits.** Group discovery presumably
  walks upward from the template looking for `.jastrgroup`; confirm the search
  origin (template directory vs. `cwd`) and any stop conditions, and whether more
  than one `.jastrgroup` on the path is an error.
  Answer: from the template directory and it stops when it finds the first `.jastrgroup`.
- **Symlink/hardlink residual.** A hardlink inside the boundary to an outside
  secret cannot be detected by realpath; confirm this residual is accepted (it
  requires the same author write access already deemed out of scope).
  Answer: ok.

## Acceptance Guidance

A reviewer should consider the implementation correct when all of the following
hold:

- `::include` produces identical containment behavior for `jastr run <path>.md`
  and `jastr run <id>`; there is no longer a "project root" vs "template
  directory" distinction in the include resolver, and the
  `template-ref`/`includes` code no longer carries two `includeRootLabel`
  variants for the two run modes.
- A `.jastrgroup` file at a directory establishes that directory as the
  containment boundary for every template beneath it; a `templates/` directory
  with no `.jastrgroup` does **not** establish a group.
- `::include{path=...}` resolves from the template directory by default;
  `::include{root="group", path=...}` resolves from the group root;
  `::include{root="file", path=...}` resolves from the directory of the file
  containing the directive.
- For all three roots, an include whose resolved realpath escapes the boundary is
  rejected with a stable `Error: <message>` and exit 1, and an include resolving
  inside the boundary succeeds — including absolute paths and paths containing
  `..` that net back inside the boundary.
- A symlink inside the boundary that resolves to a target outside the boundary is
  rejected (containment is enforced on the realpath, not lexically).
- The standalone absolute-path / `~` / `..` denylist messages are gone, replaced
  by the single containment rejection; the new `root` attribute is parsed by the
  CLI and passed through the engine's include request without the engine
  interpreting it.
- `@jastr/engine` still has no knowledge of `.jastrgroup`, groups, project roots,
  or filesystem layout; all group discovery and containment live in the CLI
  include resolver.
- e2e coverage exists for: default template-root include, `root="group"` include,
  `root="file"` include, an escape rejection for each root, a standalone template
  vs. a grouped template, and a symlink-based escape rejection. The existing
  `direct-file-include-escape` and `include-escape-rejected` cases are updated to
  the unified model.
- `packages/cli/docs/BEHAVIOR.md` regenerates cleanly
  (`bun run docs:cli:living --check` exits 0) and reflects the group/`root`
  behavior; `AGENTS.md` no longer asserts the two-boundary include model and the
  superseded include claims in the v2 spec are reconciled.
- The clean verification set exits 0: `bun run check`, `bun run typecheck`,
  `bun run test`, `bun run test:cli:e2e`, `bun run docs:cli:living --check`, and
  `bun run build`.
