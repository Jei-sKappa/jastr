# Executable UX Regression Coverage V1 Plan

## Goal

Implement the executable UX regression layer described by
`docs/threads/260527120436Z-executable-ux-regression-coverage/specs/260527121344Z-v1-spec.md`.
The finished change should give reviewers readable Markdown UX cases under
`docs/ux/`, execute those cases through Vitest against the real Skillrouter CLI,
and document the workflow for future agents.

## Tasks

### Task 1: Add the UX documentation inventory and initial case corpus

**Objective:** Create the human-readable UX contract that lists the initial
user-facing behaviors and embeds executable `skillrouter-case` examples.

**Input / context:** Use the upstream spec
`docs/threads/260527120436Z-executable-ux-regression-coverage/specs/260527121344Z-v1-spec.md`,
especially `Feature Inventory`, `Markdown Case Files`, `Case Schema`, and
`Initial UX Cases`. Use existing CLI behavior from `README.md`,
`test/cli-integration.test.ts`, `test/render.test.ts`, `test/flags.test.ts`,
`test/generate.test.ts`, and `test/includes.test.ts` as the source for expected
outputs.

**Steps:**
1. Create `docs/ux/feature-inventory.md`.
2. In `docs/ux/feature-inventory.md`, add a short explanation of executable UX
   coverage and the rule that new user-facing CLI behavior requires a UX case.
3. Add a feature matrix with columns `Area`, `User-facing behavior`, `Case IDs`,
   and `Notes`.
4. Create `docs/ux/cases/run.md`.
5. Add executable cases for `RUN-001`, `RUN-002`, and the missing optional input
   behavior described by the spec. If splitting the missing optional input
   behavior is clearer, use `RUN-003` and `RUN-004` and record both IDs in the
   inventory.
6. Create `docs/ux/cases/generate.md`.
7. Add executable cases for `GENERATE-001` and `GENERATE-002`, using generated
   file assertions for router skill content and exact stderr/stdout assertions
   wherever output is stable.
8. Create `docs/ux/cases/errors-and-flags.md`.
9. Add executable cases for accepted flag forms, representative rejected flag
   forms, missing project root behavior, unsafe include path behavior, and
   help/version behavior. Split broad areas into multiple case blocks when one
   command per case makes failures easier to diagnose.
10. Ensure every implemented case ID appears exactly once in
    `docs/ux/feature-inventory.md`.
11. Keep all Markdown and YAML ASCII-only.

**Files modified:**

- `docs/ux/cases/run.md` (NEW)

- `docs/ux/cases/errors-and-flags.md` (NEW)

**Acceptance criteria:**
- `docs/ux/feature-inventory.md` explains the executable UX workflow and maps
  implemented case IDs to user-facing behavior.
- `docs/ux/cases/run.md` contains executable `run` UX cases.
- `docs/ux/cases/generate.md` contains executable `generate` UX cases.
- `docs/ux/cases/errors-and-flags.md` contains executable flag, error, include
  edge, help, and version UX cases.
- Every implemented case block has a stable ID matching
  `^[A-Z][A-Z0-9-]*-[0-9]{3}$`.
- Every case block uses a `skillrouter-case` fenced code block with YAML shaped
  according to the spec.

### Task 2: Implement and unit-test the UX case loader

**Objective:** Provide a small reusable loader that discovers Markdown case
files, extracts `skillrouter-case` blocks, parses YAML, validates the case
schema, and rejects unsafe paths before any CLI command runs.

**Input / context:** Start from the docs created in Task 1. Use the upstream
spec sections `Case Schema`, `Runner Behavior`, and `Constraints`. Use the
existing `yaml` package; do not add a new parser dependency.

**Steps:**
1. Create `test/ux/case-loader.ts`.
2. Define and export `SkillrouterCase`, `LoadedUxCase`, and any internal helper
   types needed for source context.
3. Add `loadUxCases(root: string): Promise<LoadedUxCase[]>` that recursively
   finds Markdown files under `docs/ux/cases/**/*.md`, sorts paths
   lexicographically, extracts case blocks in file order, validates duplicate
   IDs, and returns loaded cases.
4. Add a block extractor that finds fenced code blocks whose info string is
   exactly `skillrouter-case`.
5. Capture the nearest preceding Markdown heading for each block when practical.
6. Parse each block body as YAML using the existing `yaml` package.
7. Add validation for required top-level fields, required `expect` fields, field
   types, unknown top-level fields, and unknown `expect` fields.
8. Validate case IDs against `^[A-Z][A-Z0-9-]*-[0-9]{3}$`.
9. Validate `command` as a non-empty string array.
10. Validate `cwd`, setup file paths, and expected file paths as relative,
    non-empty, forward-slash-style paths that do not contain `..` path
    segments and do not resolve outside the temporary project root.
11. Validate `expect.fileContains` and `expect.fileNotContains` as maps from
    safe paths to string arrays.
12. Create `test/ux/case-loader.test.ts`.
13. Add unit tests covering a valid case, duplicate IDs, malformed YAML, unknown
    fields, missing required fields, invalid IDs, invalid command arrays, unsafe
    `cwd`, unsafe setup file paths, unsafe expected file paths, and heading
    extraction.
14. Keep error messages concise and include source file context or case ID when
    available.

**Files modified:**
- `test/ux/case-loader.ts` (NEW)
- `test/ux/case-loader.test.ts` (NEW)

**Verification:**
```bash
bun run test test/ux/case-loader.test.ts
```

**Acceptance criteria:**
- `loadUxCases` is exported from `test/ux/case-loader.ts`.
- The loader discovers Markdown case files deterministically.
- The loader extracts only `skillrouter-case` fenced blocks.
- The loader parses YAML with the existing `yaml` package.
- The loader rejects malformed or unsupported case shapes before command
  execution.
- The loader rejects duplicate case IDs across all discovered case files.
- The loader rejects unsafe `cwd`, setup file, and expected file paths.
- `bun run test test/ux/case-loader.test.ts` exits with code 0.

### Task 3: Implement the Vitest UX case runner

**Objective:** Execute every loaded UX case against the real Skillrouter CLI and
assert the process and filesystem results described in Markdown.

**Input / context:** Start from the Task 1 docs and the Task 2 loader. Use the
upstream spec sections `Runner Behavior`, `CLI Execution`, and
`Expected Process Result`. Match the existing integration style in
`test/helpers.ts`, but configure process capture so final newlines are
preserved.

**Steps:**
1. Create `test/ux/cli-ux.test.ts`.
2. Import `loadUxCases` from `test/ux/case-loader.ts`.
3. Add local temp-project helpers that create an isolated directory for each
   case and clean it up after the case finishes.
4. Add a helper that writes setup files from the case `files` map, creating
   parent directories as needed.
5. Add a helper that resolves the case `cwd`, defaults it to the temp project
   root, creates it when needed, and verifies it stays inside the temp project.
6. Add a CLI execution helper that invokes `src/cli/index.ts` through Bun with
   `execa("bun", [cliPath, ...case.command], { cwd, reject: false,
   stripFinalNewline: false })`.
7. Ensure the runner never invokes a shell and never accepts a command string.
8. Create one Vitest `it(...)` per loaded UX case, with a test title containing
   the case ID and heading when available.
9. Assert `exitCode`, `stdout`, and `stderr` exactly.
10. Add exact expected file assertions for `expect.files`.
11. Add substring presence assertions for `expect.fileContains`.
12. Add substring absence assertions for `expect.fileNotContains`.
13. Include the case ID and relevant field or file path in assertion failure
    messages where a custom message is practical.
14. Run the UX test file and adjust only the Markdown case expectations if an
    expectation does not match current documented CLI behavior.

**Files modified:**
- `test/ux/cli-ux.test.ts` (NEW)
- `docs/ux/feature-inventory.md`
- `docs/ux/cases/run.md`
- `docs/ux/cases/generate.md`
- `docs/ux/cases/errors-and-flags.md`

**Verification:**
```bash
bun run test test/ux/cli-ux.test.ts
```

**Acceptance criteria:**
- `test/ux/cli-ux.test.ts` loads all executable UX cases from `docs/ux/cases/`.
- Each UX case runs in an isolated temporary project.
- Setup files are written before the command runs.
- Case `cwd` is created when needed and remains inside the temp project.
- The real `src/cli/index.ts` entry point is exercised.
- The runner does not invoke a shell.
- Final newlines in stdout and stderr are preserved.
- Exit code, stdout, stderr, exact file contents, required substrings, and
  forbidden substrings are asserted.
- `bun run test test/ux/cli-ux.test.ts` exits with code 0.
