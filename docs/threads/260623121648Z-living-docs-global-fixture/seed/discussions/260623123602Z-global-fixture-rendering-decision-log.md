# Decision log — living-docs global-fixture rendering (the seed)

Thread: docs/threads/260623121648Z-living-docs-global-fixture/
Target: the seed (seed/260623121648Z-living-docs-global-fixture-seed.md) — genesis discussion
Subject: designing the fix for the living-docs generator (`packages/cli/scripts/living-docs.ts`) ignoring `global-fixture/`, so the 17 global e2e cases render their global root in `packages/cli/docs/BEHAVIOR.md`.

## P1: Input-section presentation of the two roots

Point: How should the rendered case "Input" section present the two roots (local `fixture/` + global `global-fixture/`) so the global cases become useful again?

What you need to know: Today every case renders one block labelled "Input project" built only from `fixture/` (`renderInputSection`, `living-docs.ts:331-343`), falling back to "_Empty — no `.jastr/` directory present_" when that's empty. `renderFileBlock` and `buildFileTree` are root-agnostic helpers reusable for a second root. 17 of the 20 new cases carry a `global-fixture/`; 12 of those also carry a local `fixture/`, and 5 are global-only (local empty). There is a churn-vs-symmetry trade-off: keeping the local block labelled exactly "Input project" and only adding a global block leaves every existing non-global case byte-identical (purely additive diff), whereas relabeling for symmetry ("Local project" / "Global root") changes every existing case's block (large but low-risk diff in a generated file). The user chose the symmetric presentation. The remaining sub-fork is empty-root handling. Option A — symmetric labels, render a root section only when it has content: local-only cases change by one line ("Input project" → "Local project") with no empty "Global root" noise; a global-only case shows "Local project: _Empty_" followed by a "**Global root** — `$JASTR_HOME/.jastr`" section; both present shows both; both empty shows a single Empty note. Option B — always render BOTH sections for every case with "_Empty_" for whichever is missing (maximally uniform, but ~60 local-only cases gain a "Global root: Empty" block). Option C — restructure only cases that have a `global-fixture/`, leaving non-global cases byte-identical (minimal churn but inconsistent labels across the doc).

Decision: Option A — symmetric "Local project" / "Global root (`$JASTR_HOME/.jastr`)" labels, with each root section rendered only when it has content; an empty local root in a global case shows an explicit "_Empty_" note, and a both-empty case shows a single Empty note.

Rationale: A gives the symmetric two-root presentation the user wanted, surfaces the global root exactly where it is in play, and fixes the misleading "Empty input → magic global output" cases — without spraying a "Global root: Empty" block across the ~60 local-only cases that have nothing to do with the global root (the noise B would add). The only thing given up versus B is an explicit "no global root" line on local-only cases, judged to be noise there rather than useful signal. The churn (relabel "Input project" → "Local project" on existing cases) is accepted because it is a one-time regeneration of a generated file with no content to hand-verify beyond spot-checks.
