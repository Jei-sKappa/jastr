# Named Group Location Realignment Implementation Plan

**Goal:** Move named template groups from the project-root location
`<root>/<group>/templates/<id>/TEMPLATE.md` to live inside `.jastr/` at
`<root>/.jastr/<group>/templates/<id>/TEMPLATE.md`, so every named template
(standalone and grouped) has a single predictable home under `.jastr/`.

**Architecture:** This is a one-line resolution change plus its ripples. The
named-group root is constructed in exactly one place
(`packages/cli/src/templates/template-ref.ts`, function
`loadGroupedNamedTemplate`). We insert a `.jastr` path segment into that join
and prefix the not-found message with `.jastr/`. Everything else — the
reference grammar, the CLI surface, direct `.md` classification, include
containment, project-config keys, and the `.jastrgroup` marker requirement — is
unchanged. The rest of the work is realigning the on-disk test fixtures and the
docs/requirements that asserted the old layout, then regenerating the living
behavior doc.

**Tech Stack:** TypeScript, Bun (runtime + package manager + bundler), Vitest
(unit/integration + a fixture-driven e2e harness), Biome (format/lint), YAML
requirement + e2e-case files, a `tsx` living-docs generator.

## Background you need before starting

### How a named reference resolves today

`jastr run <ref>` (and `generate` / `validate`) calls
`loadTemplateReference` in
`packages/cli/src/templates/template-ref.ts`. For a *named* (non-`.md`) ref it
first finds the **project root** by walking up from the cwd to the nearest
ancestor that contains a `.jastr/` directory (`findProjectRoot` in
`packages/cli/src/fs/project-root.ts`). Then:

- **Standalone** `jastr run <id>` → `loadStandaloneNamedTemplate` →
  `<root>/.jastr/<id>/TEMPLATE.md`. **Inside `.jastr/`. This is unchanged.**
- **Grouped** `jastr run <group>/<id>` → `loadGroupedNamedTemplate` →
  builds `groupRoot = path.join(projectRoot, group)` (the project root, *beside*
  `.jastr/`), requires `<groupRoot>/.jastrgroup`, and loads
  `<groupRoot>/templates/<id>/TEMPLATE.md`. **This is what we move under
  `.jastr/`.**

After this change, the grouped `groupRoot` becomes
`path.join(projectRoot, ".jastr", group)`, matching the standalone layout.

### What does NOT change (do not touch these)

- **Direct `.md` access** (`classifyDirectTemplate`): a direct grouped template
  is still any realpath of the exact shape
  `<group>/templates/<id>/TEMPLATE.md` with a sibling `<group>/.jastrgroup`,
  located anywhere, needing no `.jastr/`. This function derives the group root
  from the supplied path's own ancestry and is location-agnostic.
- **The reference grammar / CLI surface**: `<group>/<id>`, the `#<variant>`
  suffix, one-segment standalone refs, and the invalid-reference rules.
- **Project-config keys**: `inputs.<group>/<id>` and `variants.<group>/<id>` use
  the unchanged ref *string* `<group>/<id>`, so config files keep the same keys.
- **Include containment model**: realpath containment, the three `root`
  selectors (`template`, `group`, `file`), and the group-root boundary. Only the
  *filesystem location* of a named group's root moves under `.jastr/`.
- **The `.jastrgroup` marker**: still an existence-only marker (contents
  ignored), still **required** for every group in both named and direct access.
- **`@jastr/engine`**: must never learn about `.jastr`, groups, project roots,
  markers, or filesystem layout. All of this lives in `@jastr/cli`.

### Why one big implementation commit (read this — it changes the task shape)

`bun run test` is `vitest run`, which runs the **entire** Vitest suite,
**including** the fixture-driven e2e cases in
`packages/cli/test/e2e/e2e.test.ts`. The source change and the fixture moves are
inseparable: change the source and the old-location e2e fixtures fail; move the
fixtures without the source change and they fail against the old resolver. So
**Tasks 1–4 stage one coherent change and commit once, at the end of Task 4.**
Tasks 1–3 say "do not commit yet" and verify with *focused* test runs; Task 4
runs the full gauntlet and makes the single implementation commit. Task 5 is a
separate docs-only commit.

### Spec discrepancy you must know about

The spec's acceptance guidance enumerates **five** named grouped e2e cases to
realign (`grouped-named-run`, `variant-run-grouped`, `validate-grouped`,
`validate-grouped-variant`, `config-grouped-template-key`). That enumeration is
**incomplete**. Three more e2e cases invoke a named grouped ref (`run team/demo`
or `validate team/...`) and therefore break under the resolver change too:

- `grouped-named-missing-marker` — asserts the missing-marker not-found path
  (its expected stderr message changes).
- `include-group-root-escape` — named grouped `run team/demo`, group-root
  include escape.
- `include-file-root-escape` — named grouped `run team/demo`, file-root include
  escape.

This plan realigns **all eight** named grouped cases. The two genuinely
*direct* grouped cases the spec wants untouched —
`direct-grouped-template-include` (`run team/templates/demo/TEMPLATE.md`) and
`include-group-root-standalone-rejected` (`run solo/TEMPLATE.md`) — stay exactly
as they are; leaving them untouched is the proof that direct classification is
unaffected.

### How an e2e case works

Each case lives at `packages/cli/test/e2e/cases/<id>/case.yml`. Its sibling
`fixture/` folder's **contents** are copied to a fresh temp dir that becomes the
**project root** the CLI runs in. So `fixture/.jastr/team/...` lands at
`<root>/.jastr/team/...`. The runner compares exit code, stdout, and stderr
against `case.yml`'s `expect` block. `covers:` lists `<FR-ID>.AC-NNNN` refs that
must point to live acceptance criteria in
`packages/cli/requirements/functional/`. Traceability
(`packages/cli/test/e2e/harness/traceability.ts`) enforces two invariants that
matter here:

1. Every `covers` ref must resolve to an existing, non-removed acceptance
   criterion.
2. Every active, non-removed acceptance criterion must be covered by ≥1 case.

So a new acceptance criterion and a covering case must land together.

### The exact not-found message strings

- Old grouped miss:
  `Template team/demo was not found at team/templates/demo/TEMPLATE.md.`
- New grouped miss (this change):
  `Template team/demo was not found at .jastr/team/templates/demo/TEMPLATE.md.`
- Standalone miss (unchanged):
  `Template demo was not found at .jastr/demo/TEMPLATE.md.`

All CLI failures print `Error: <message>\n` to stderr with exit code 1. No new
`JastrErrorCode` is introduced; the move reuses `template_not_found`.
