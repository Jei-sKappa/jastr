# Executable UX Regression Coverage V1 Spec

## Intended Outcome

This spec adds an executable UX regression layer to Skillrouter. When
implemented, the project will have readable Markdown documentation that lists
important user-facing CLI behaviors and embeds structured examples that run as
Vitest tests.

The practical outcome is a review loop for AI-built changes:

1. The user describes a desired user-facing behavior.
2. The agent captures that behavior as an executable UX case.
3. The case runs the real Skillrouter CLI against a temporary project.
4. The case asserts exit code, stdout, stderr, and generated files.
5. The passing case remains living documentation for future agents and
   reviewers.

The long-term rule is:

> Any new user-facing CLI behavior, flag, error, config rule, or output contract
> must be represented by at least one executable UX case.

## Context

Skillrouter is a private, early-stage CLI for deterministic AI-agent skill
specialization. Its primary user-facing commands are:

```bash
skillrouter run <skill> [input flags...]
skillrouter generate <skill> --out <path> [--force]
```

The existing codebase already has Vitest tests under `test/` for parser,
compiler, condition, include, interpolation, generation, and CLI integration
behavior. Those tests are valuable and should remain. The gap is that they are
mostly implementation-oriented or narrowly integration-oriented. They do not
provide a single readable product contract that the project owner can inspect
to confirm, "This is the UX I meant."

The v1 Skillrouter spec already includes acceptance guidance for command
behavior and output contracts. This spec turns that kind of guidance into
Markdown-backed executable examples so expected UX is both readable and
regression-tested.

## Scope

V1 includes:

- A Markdown feature inventory at `docs/ux/feature-inventory.md`.
- Markdown UX case files under `docs/ux/cases/`.
- Executable `skillrouter-case` fenced blocks embedded in those Markdown files.
- A Vitest-backed case loader and runner under `test/ux/`.
- A small initial suite of high-value UX cases for `run`, `generate`, flags,
  errors, include edge cases, and help/version behavior.
- Exact process assertions for exit code, stdout, and stderr.
- Filesystem assertions for generated or modified files.
- Case validation that fails fast on duplicate case IDs, malformed blocks,
  unsafe paths, or unsupported fields.
- Integration into the normal `bun run test` path.
- Contributor guidance in `AGENTS.md` and a short developer-facing README note
  once the executable UX workflow exists.

V1 does not include:

- Cucumber, Gherkin, or a BDD framework.
- A generated documentation website.
- Browser, UI, or visual regression testing.
- Exhaustive duplication of unit tests for parser/compiler internals.
- Arbitrary shell command execution from Markdown.
- Test-case support for stdin, environment variables, interactive prompts,
  terminal colors, permissions fixtures, symlink fixtures, or platform-specific
  filesystem behavior.
- Automatic generation of UX cases from source code.
- A new Skillrouter CLI command for listing or running UX cases.
- Changes to the product behavior of `skillrouter run` or
  `skillrouter generate` except where tests expose an existing bug.

The executable UX layer is a documentation and regression harness. It is not a
replacement for the existing test suite and not a new product feature exposed to
Skillrouter users.

Add these files:

```text

docs/ux/cases/run.md
docs/ux/cases/generate.md

test/ux/case-loader.ts

```

Update these files:

```text

```

`README.md` should mention the executable UX cases in the development/test
section after the feature exists.

`AGENTS.md` should record the contributor rule that user-facing CLI behavior,
flags, errors, config rules, and output contracts require a corresponding
executable UX case.

No package dependency is required for YAML parsing because the project already
depends on `yaml`. Do not add a BDD framework.

## Feature Inventory

`docs/ux/feature-inventory.md` is the reviewer-facing index of the UX contract.
It should be concise and table-driven. It should not restate every case in full.

It must include:

- The purpose of executable UX coverage.
- The rule for when a UX case is required.
- A feature matrix with stable case IDs.
- A note that normal unit/module tests remain responsible for exhaustive
  implementation-level coverage.
- A note that `bun run test` executes the UX cases.

The feature matrix should use this shape:

```md
| Area | User-facing behavior | Case IDs | Notes |
| --- | --- | --- | --- |
| run | Renders selected instructions from a template | RUN-001 | Covers stdout contract |
| generate | Writes a router skill to explicit `--out` | GENERATE-001 | Covers generated content |
```

Every executable case ID must appear in the inventory. The inventory may include
planned case IDs only if they are clearly marked as planned and not implemented.
The initial implementation should avoid planned-only rows unless they help a
reviewer understand intentional gaps.

## Markdown Case Files

UX cases live in Markdown files under `docs/ux/cases/`.

The runner must discover case files by recursively scanning:

```text
docs/ux/cases/**/*.md
```

Discovery order must be deterministic. Sort discovered file paths
lexicographically, then execute case blocks in file order.

Each case is a fenced code block whose info string is exactly:

```text
skillrouter-case
```

The fenced block body is YAML. The surrounding Markdown is ordinary narrative
documentation for humans. The runner only executes `skillrouter-case` blocks.

Example:

````md
## RUN-001 renders selected TypeScript branch

This case verifies that `run` prints only rendered Markdown to stdout.

```skillrouter-case
id: RUN-001
cwd: nested
command: ["run", "demo", "--language=typescript", "--target-file=src/index.ts"]
files:
  nested/.keep: ""
  .skillrouter/demo/SKILL.template.md: |
    ---
    name: demo
    description: Demo skill
    inputs:
      language:
        type: enum
        values: [typescript, python]
        required: true
      target-file:
        type: string
        required: false
    ---
    # Demo

    :::if{condition="${language} == 'typescript'"}
    Analyze {{target-file}} as TypeScript.
    :::
expect:
  exitCode: 0
  stdout: |
    # Demo

    Analyze src/index.ts as TypeScript.
  stderr: ""
```
````

The nearest preceding Markdown heading may be used in the Vitest test title, but
the executable identity is the YAML `id`.

## Case Schema

Each `skillrouter-case` YAML block must parse to an object with this schema:

```ts
type SkillrouterCase = {
  id: string;
  cwd?: string;
  command: string[];
  files?: Record<string, string>;
  expect: {
    exitCode: number;
    stdout: string;
    stderr: string;
    files?: Record<string, string>;
    fileContains?: Record<string, string[]>;
    fileNotContains?: Record<string, string[]>;
  };
};
```

Required fields:

- `id`
- `command`
- `expect`
- `expect.exitCode`
- `expect.stdout`
- `expect.stderr`

Optional fields:

- `cwd`
- `files`
- `expect.files`
- `expect.fileContains`
- `expect.fileNotContains`

Unknown top-level fields are errors. Unknown fields under `expect` are errors.
This keeps case files predictable and prevents documentation from implying
unsupported behavior.

### IDs

Case IDs must be stable, unique, and reviewer-friendly.

The required grammar is:

```text
^[A-Z][A-Z0-9-]*-[0-9]{3}$
```

Valid examples:

```text
RUN-001
FLAGS-002
GENERATE-001
EDGE-001
CLI-001
```

Duplicate IDs across all discovered case files are errors.

Do not renumber existing IDs just to reorder documentation. If a case is
replaced, either update the existing case in place before it becomes part of a
released baseline, or add a new case ID when preserving review history matters.

### Commands

`command` is the argument vector passed after the Skillrouter executable. It is
not a shell command.

Example:

```yaml
command: ["run", "demo", "--language=typescript"]
```

The runner must not invoke a shell and must not parse a command string. This is
a safety constraint and a portability constraint.

Every command element must be a string. Empty command arrays are invalid.

### Current Working Directory

`cwd` is optional. When omitted, it defaults to `"."`, meaning the temporary
project root created for that case.

When present, `cwd` is a relative path inside the temporary project root.

The runner must create the `cwd` directory if it does not already exist after
writing setup files. This lets cases describe nested invocation behavior without
adding placeholder files unless the placeholder is useful documentation.

`cwd` must not be absolute and must not escape the temporary project root.

### Setup Files

`files` is an optional map from project-relative path to UTF-8 text contents.

Setup file paths:

- Must be relative paths.
- Must not be empty.
- Must not contain `..` path segments.
- Must not resolve outside the temporary project root.
- Must use forward slashes in docs, even on platforms where Node resolves them
  to platform separators.

The runner writes setup files before running the command. Missing parent
directories are created automatically.

The runner should not pre-create `.skillrouter/` for every case. Cases should
declare the files and directories needed to express the behavior. If a case is
testing missing project root behavior, it should omit `.skillrouter/`.

### Expected Process Result

`expect.exitCode` is compared exactly against the process exit code.

`expect.stdout` and `expect.stderr` are compared exactly against the captured
process streams.

The UX runner must capture stdout and stderr without stripping final newlines.
Use an execution helper configured with final-newline preservation rather than
the existing integration helper if the existing helper strips final newlines.

This is deliberate: rendered Markdown and CLI error formatting are part of the
UX contract. If the process writes a trailing newline, the YAML expectation must
include it. If it does not, the expectation must omit it.

For readability, cases may use YAML block scalars for multiline stdout/stderr.
Reviewers should be able to see blank lines and trailing-newline intent in the
case.

### Expected Files

`expect.files` is an optional map from project-relative path to exact UTF-8
contents expected after the command exits.

`expect.fileContains` is an optional map from project-relative path to a list of
substrings that must be present after the command exits.

`expect.fileNotContains` is an optional map from project-relative path to a list
of substrings that must not be present after the command exits.

Expected file paths follow the same safety rules as setup file paths.

Use `expect.files` when the full output is stable and relevant. Use
`fileContains` and `fileNotContains` when only a few generated-file guarantees
matter and full-file comparison would make the case noisy.

If an expected file path does not exist after the command, the case fails with a
message that includes the case ID and file path.

## Runner Behavior

`test/ux/case-loader.ts` should expose small testable functions rather than
embedding all behavior inside a Vitest callback.

Recommended functions:

```ts
export type SkillrouterCase = { ... };

export async function loadUxCases(root: string): Promise<LoadedUxCase[]>;
export function parseSkillrouterCaseBlock(
  source: string,
  filePath: string,
): ParsedCaseBlock[];
export function validateSkillrouterCase(
  value: unknown,
  source: { filePath: string; idHint?: string },
): SkillrouterCase;
```

`LoadedUxCase` should include:

- the parsed case
- the source Markdown file path
- the nearest preceding heading, when available
- enough source-location context for useful failure messages

The loader must:

- Find all `skillrouter-case` fenced blocks.
- Parse YAML with the existing `yaml` package.
- Validate required fields and field types.
- Reject unknown fields.
- Reject duplicate IDs.
- Reject unsafe paths.
- Preserve Markdown narrative outside case blocks.

`test/ux/cli-ux.test.ts` should:

1. Load all UX cases.
2. Create one Vitest `it(...)` per case.
3. Create an isolated temporary project for each case.
4. Write declared setup files.
5. Create the case `cwd` if needed.
6. Run the real CLI entry point with the declared command argv.
7. Compare exit code, stdout, stderr, and expected file assertions.
8. Clean up the temporary project.

Test titles should include the case ID. Prefer:

```text
RUN-001 renders selected TypeScript branch
```

where the suffix comes from the nearest Markdown heading with the case ID
removed if practical. If heading extraction is awkward, using only the ID plus
source path is acceptable for v1.

Failure messages should include the case ID and the relevant field or path.

## CLI Execution

The UX runner must execute the same TypeScript CLI entry point used by the
current integration tests:

```text
src/cli/index.ts
```

The runner may invoke it through Bun in tests, matching the existing project
test style:

```ts
execa("bun", [cliPath, ...command], {
  cwd,
  reject: false,
  stripFinalNewline: false,
});
```

This test harness use of Bun is acceptable because the project already uses Bun
for development and existing CLI integration tests. Runtime source code should
continue avoiding Bun-specific APIs unless separately decided.

The runner must not execute commands through `sh`, `zsh`, `bash`, or any other
shell.

## Initial UX Cases

The initial implementation must add at least these executable cases.

### RUN-001

`run` from a nested cwd discovers the nearest `.skillrouter`, renders the
selected branch, strips root frontmatter, strips directive syntax, and prints
Markdown only to stdout.

Required assertions:

- Exit code is `0`.
- Stdout is the rendered Markdown.
- Stderr is empty.
- Output does not include root frontmatter.
- Output does not include directive syntax.

### RUN-002

`include` evaluates directives and interpolation inside included files, while
`include-raw` inserts literal content without evaluating Skillrouter syntax.

Required assertions:

- Selected `include` content appears with interpolation applied.
- `include-raw` content appears literally.
- Raw included `{{input}}` syntax remains raw.
- Raw included `::include{...}` syntax remains raw.

### RUN-003

A missing optional input is falsey in conditions, but direct interpolation of
that missing optional input fails.

This may be represented as two separate executable cases if that keeps the docs
clearer:

- `RUN-003`: missing optional input selects the falsey branch.
- `RUN-004`: missing optional input interpolation exits non-zero with empty
  stdout and an `Error: ...` stderr message.

If split, both IDs must appear in the feature inventory.

### FLAGS-001

Accepted flag forms for enum, string, and boolean inputs:

- `--language=typescript`
- `--target-file=src/index.ts`
- `--dry-run`
- `--dry-run=true`
- `--dry-run=false`

The case may use multiple commands only by defining multiple case blocks. V1
case blocks run one command each.

### FLAGS-002

Rejected flag forms and values:

- unknown input flag
- duplicate flag
- `--no-dry-run`
- bare string flag
- bare enum flag
- invalid enum value
- empty string or enum value

These should be split into multiple case blocks if a single case would obscure
which behavior failed. It is acceptable for the initial implementation to cover
the highest-value rejected forms first, provided the inventory marks remaining
forms as planned or leaves them to existing unit tests.

### GENERATE-001

`generate` writes a router skill.

Required assertions:

- Exit code is `0`.
- Stdout includes the generated path and source template path in the documented
  success format.
- The generated file contains `skillrouter run demo $ARGUMENTS`.
- The generated file contains the non-zero failure instruction.
- Valid extra frontmatter fields are preserved.
- Skillrouter-owned fields such as `inputs` are omitted.

Use `fileContains` and `fileNotContains` unless exact full-file comparison is
clearer.

### GENERATE-002

`generate` enforces output path behavior.

Required assertions across one or more cases:

- Missing `--out` exits non-zero.
- Relative `--out` resolves from the invoking cwd.
- Missing parent directories are created.
- Existing output is refused without `--force`.
- Existing output is overwritten with `--force`.

### EDGE-001

Missing `.skillrouter` root fails.

Required assertions:

- Exit code is non-zero.
- Stdout is empty.
- Stderr is a single `Error: ...` diagnostic emitted by Skillrouter.

### EDGE-002

Include paths reject unsafe or explicitly disallowed forms.

Required assertions across one or more cases:

- Absolute include path is rejected.
- `~` include path is rejected.
- Include path escaping the project root is rejected.
- `.env` include path is rejected.
- `.env.*` include path is rejected.

### CLI-001

Help and version commands are successful UX paths.

Required assertions across one or more cases:

- `--help` exits `0` and lists canonical commands.
- `help` exits `0` and lists commands.
- `help run` exits `0` and describes `skillrouter run`.
- `--version` exits `0` and matches source/test version behavior.

Version output may assert the exact current source behavior if stable in tests,
or a strict pattern if the value depends on build-time injection. The case
should document which behavior it protects.

The new UX suite should coexist with:

- `test/frontmatter-schema.test.ts`
- `test/flags.test.ts`
- `test/interpolation.test.ts`
- `test/errors.test.ts`
- `test/generate.test.ts`
- `test/scaffold.test.ts`
- `test/project-skill.test.ts`
- `test/cli-args.test.ts`
- `test/conditions.test.ts`
- `test/render.test.ts`
- `test/cli-integration.test.ts`
- `test/includes.test.ts`
- `test/directives.test.ts`

The existing tests remain the right place for exhaustive low-level permutations.
The UX docs should cover representative behavior that matters to users and
reviewers.

If an existing CLI integration case becomes better expressed as a UX case, it
may be mirrored first and moved later. Avoid large test-file churn in the first
implementation unless duplication becomes confusing.

## Constraints

Use Vitest for the executable UX test runner. Do not introduce Cucumber,
Jest, Playwright, or another test framework for this v1 layer.

Use the existing `yaml` package for case block parsing.

Use Node standard library APIs and existing project test patterns for temporary
directories, file writes, and cleanup.

Keep source code portable. The test harness may invoke Bun because existing CLI
integration tests do, but production source code should not gain Bun-specific
runtime APIs for this feature.

Keep the case format intentionally small. Do not add a general fixture language,
variables, templating, command interpolation, setup hooks, teardown hooks,
environment configuration, stdin, or multi-command scripts in v1.

The Markdown docs are trusted repository content, but the runner must still
avoid shell execution and reject unsafe file paths. A malformed docs case should
fail the test suite, not write outside its temporary project.

Use ASCII in new files unless existing content requires otherwise.

Do not rely on automatic snapshot update workflows for UX expectations.
Expected stdout/stderr and important generated-file content should be explicit
and reviewable in Markdown.

Biome, TypeScript, and Vitest must remain clean:

```bash
bun run check
bun run typecheck
bun run test
```

## Explicit Trade-Offs

Inline Markdown case blocks are the v1 format. Separate `.case.yml` files are
not used in v1 because the primary goal is reviewable living documentation, and
inline blocks keep prose and executable examples together.

The UX runner uses structured YAML blocks, not Gherkin. Skillrouter's current
UX contract is deterministic CLI I/O, so a direct schema is simpler and clearer
than natural-language step definitions.

`bun run test` must execute the UX cases. A separate focused script such as
`bun run test:ux` may be added later or as a convenience, but it is not required
for v1 and must not make UX regression optional.

Expected stdout and stderr are exact. This is stricter than broad contains-style
CLI checks because Skillrouter's rendered Markdown and single-line error UX are
part of its public contract.

Generated files may use exact file assertions or contains/not-contains
assertions. Exact assertions are preferred when the whole file is stable and
important; contains assertions are acceptable when they make the UX case easier
to review.

The first implementation may duplicate a few existing CLI integration checks in
the docs-backed suite. This is acceptable because the new suite has a different
purpose: making UX expectations readable to the project owner and future agents.

## Unresolved Questions

These questions do not block v1:

- Whether existing `test/cli-integration.test.ts` cases should eventually be
  moved fully into docs-backed UX cases or kept as a smaller command-plumbing
  smoke suite.
- Whether a future docs website should be generated from `docs/ux/` once the
  Markdown stabilizes.
- Whether later UX cases need richer fixtures for environment variables, stdin,
  filesystem permissions, symlinks, or absolute temporary paths.
- Whether case IDs should eventually be grouped by command-specific files only,
  or whether cross-cutting files such as `errors-and-flags.md` remain clearer.
- Whether future specs should require a dedicated `bun run test:ux` script in
  addition to `bun run test`.

## Acceptance Guidance

An implementation satisfies this spec when a reviewer can verify all of the
following.

Documentation artifacts exist:

- `docs/ux/feature-inventory.md`
- `docs/ux/cases/run.md`
- `docs/ux/cases/generate.md`
- `docs/ux/cases/errors-and-flags.md`

Test harness artifacts exist:

- `test/ux/case-loader.ts`
- `test/ux/cli-ux.test.ts`

The feature inventory:

- Explains the purpose of executable UX coverage.
- States that new user-facing CLI behavior requires a UX case.
- Lists every implemented UX case ID.
- Maps case IDs to feature areas and user-facing behaviors.
- Makes clear that unit/module tests still cover lower-level permutations.

The case loader:

- Discovers Markdown files under `docs/ux/cases/**/*.md` deterministically.
- Extracts only `skillrouter-case` fenced blocks.
- Parses block bodies as YAML.
- Validates required fields and field types.
- Rejects unknown fields.
- Rejects duplicate IDs.
- Rejects unsafe setup, cwd, and expected file paths.
- Provides failure messages that include case ID and source file context.

The case runner:

- Creates an isolated temporary project for each case.
- Writes declared setup files.
- Creates the declared cwd when needed.
- Runs `src/cli/index.ts` through the real CLI path.
- Does not execute through a shell.
- Preserves final newlines in stdout and stderr capture.
- Compares exit code, stdout, and stderr exactly.
- Supports exact expected file content.
- Supports expected file contains and not-contains assertions.
- Cleans up temporary projects after each case.

The initial executable case suite covers at least:

- `RUN-001`
- `RUN-002`
- `RUN-003` or equivalent split cases
- `FLAGS-001`
- `FLAGS-002` or a documented initial subset
- `GENERATE-001`
- `GENERATE-002`
- `EDGE-001`
- `EDGE-002`
- `CLI-001`

The existing Vitest tests still run and are not replaced wholesale.

`README.md` mentions the executable UX docs in the development workflow.

`AGENTS.md` records the rule that user-facing CLI behavior, flags, errors,
config rules, and output contracts require an executable UX case.

The full project verification passes:

```bash
bun run check
bun run typecheck
bun run test
```

Manual review of one positive case and one negative case should be enough for a
reviewer to understand the case format without reading the runner
implementation.
