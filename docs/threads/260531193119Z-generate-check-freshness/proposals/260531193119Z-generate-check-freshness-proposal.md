# Proposal: `skillrouter generate --check` freshness mode

## Intent

Add a `--check` flag to `generate` that re-builds the router-skill content in
memory and byte-compares it against the file already on disk at `--out`,
exiting non-zero on mismatch and writing nothing. It answers: *"is the committed
router skill at this path still up to date with its template?"* â€” a drift gate
for CI and pre-commit hooks.

## Context

`generate` is skillrouter's only commit-the-output path. The router skill it
writes (e.g. into `.claude/skills/<skill>/SKILL.md`) is meant to be committed to
a repo. Its content is derived from the template's frontmatter: `name`,
`description`, and passthrough kebab-case fields all flow through
`buildRouterSkillContent` in `src/generate/router-skill.ts`.

When that template frontmatter later changes â€” a reworded `description`, a new
passthrough field, a renamed skill â€” the committed router file goes **stale
silently**. Nothing detects it. The next person to read the committed skill sees
outdated metadata, and the only way to notice today is to regenerate and eyeball
the diff. Hand-editing the committed file (not realizing it is generated) has
the same failure mode in reverse.

This is a known, solved pattern *inside this very project*: `bun run docs:living
--check` re-renders `docs/BEHAVIOR.md` in memory and exits 1 if it differs from
the committed file, gating freshness in CI. The vendored gstack reference does
the identical thing for its generated `SKILL.md` files via `--dry-run`. The
proposal is simply to extend that established, trusted pattern to `generate`,
the one place skillrouter emits a committed artifact and currently has no
freshness guard.

## Rough shape

- Add `--check` as a Commander option on the `generate` command
  (`src/cli/commands/generate.ts`), declared alongside `--out` / `--force`.
- In `executeGenerate` (`src/cli/commands.ts`): run the same validation and
  `buildRouterSkillContent` as a normal generate, then branch:
  - default â†’ write to `--out` (current behavior).
  - `--check` â†’ read the existing file at the resolved `--out` path and compare
    byte-for-byte. Equal â†’ success, write nothing. Different (or file missing)
    â†’ `Error: <message>` to stderr, exit 1, write nothing.
- `--check` requires `--out` (it is a comparison against a specific path) and is
  mutually exclusive with `--force` (force is about overwriting; check never
  writes).
- Determinism is already in place: `buildRouterSkillContent` builds frontmatter
  by iterating `Object.entries` over `YAML.parse` output (insertion order
  preserved) and serializes via `YAML.stringify`, so the built bytes are stable
  for a given template across machines. `--check` will not flap.

## Relationship to a standalone `validate` command

Complementary, not redundant â€” the distinction is worth pinning so a reader does
not collapse them:

- `validate` guards the **input**: is the template authorable at all? No `--out`,
  reads nothing back, writes nothing, destination-independent.
- `generate --check` guards the **output**: does the committed router file at a
  specific path still match what the template produces now? Path-specific,
  compares against an existing artifact.

Because `generate` already validates statically before writing, `generate
--check` effectively *contains* the validation `validate` would run, then adds a
comparison against `--out`. The failures they catch differ: a valid template
with a stale committed copy passes `validate` and fails `generate --check`. Of
the two, `--check` is the higher-value capability because it closes a *silent*
drift (nobody notices a stale committed file), whereas a malformed template
fails loudly the next time anyone runs it.

## Open questions

- **Behavior when the target file is absent.** Treat "no file at `--out`" as a
  check failure (exit 1, "would create"), or as a distinct exit code so CI can
  tell "stale" apart from "never generated"? Leaning toward failure with a
  clear message, but flagging it.
- **Output verbosity.** On mismatch, emit just `Error: <skill> router at <path>
  is stale; regenerate with skillrouter generate ...`, or also surface a diff?
  The v1 output contract favors a single concise `Error:` line and defers
  verbose/diff output â€” likely keep it terse, with a diff left to a future
  diagnostic mode.
- **Provenance header coupling.** A sibling idea is stamping the generated router
  with a `<!-- generated by skillrouter; edit the template, not this file -->`
  comment so hand-edits are discouraged and `--check` failures read clearly. But
  the v1 spec pins the *exact* router-skill bytes (asserted in e2e). Adding a
  header is a deliberate contract change that must land in the spec, and it
  changes what `--check` compares against. Raising it here so it is not assumed
  settled; it can ship independently of `--check` or alongside it.
- **Multiple destinations.** skillrouter is agent-agnostic: one skill may be
  generated to several `--out` paths. `--check` is inherently per-path, so a CI
  sweep would invoke it once per committed destination. Whether to add a
  higher-level "check all known outputs" affordance (and where the list of
  destinations would even live) is unresolved and out of scope for a first cut.
- **Spec scope.** Adding `--check` extends the `generate` contract in the v1
  spec; it should land as a spec delta, not a silent addition. In-bounds to
  decide now given the project is private and pre-release.
