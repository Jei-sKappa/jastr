# `jastr validate <template-ref>` Command v1 Spec

## Intended Outcome

Jastr gains a third top-level command, `jastr validate <template-ref>`, that runs
the existing static-validation pipeline against a template and reports pass/fail
without writing any file, requiring an output destination, or taking invocation
inputs. It answers one question for a template author: *"is this template
well-formed enough to be used at all?"*

On success it prints a single confirmation line to stdout and exits `0`. On any
defect it prints the existing `Error: <message>` to stderr and exits `1`, reusing
each engine error's specific code so diagnostics match what `run`/`generate`
already emit for the same defect.

## Context

Jastr already owns the entire static-validation pipeline â€” frontmatter parsing,
schema validation, input-name grammar, directive scanning, condition parsing,
condition/interpolation reference checks, include-path resolution, missing
include detection, and include-cycle detection. But that logic is only reachable
as a side effect of two operations:

- `run` validates, then renders to stdout â€” so it needs invocation inputs and
  produces a full render.
- `generate agent-skill` validates, then writes a wrapper â€” so it needs an
  `--out` path and produces a committed artifact.

There is no way to ask "is this template valid?" on its own. An author iterating
on a template must either invoke `run` with a plausible set of flags (mixing
authoring errors with input errors) or `generate` to a throwaway path. Both are
indirect. The validation capability exists; only the front door is missing.

The upstream input is the proposal at

That proposal predates the package split and the rename to Jastr (it speaks of
"skillrouter", `<skill>` slugs, `SKILL.template.md`, and a monolithic
`validateSkillTemplate`); this spec re-maps its intent onto the current v2
surface â€” a `<template-ref>` accepting every form `run`/`generate` accept,
`@jastr/engine`'s decomposed validation API (`parseTemplateSource`,
`validateTemplateSchema`, `renderTemplateSource`), and the CLI's existing
project-discovery, variant-resolution, and agent-skill target-validation code.
The proposal's open questions are resolved here (see the inline decisions below);
none block emission.

This is the **input gate**, complementary to the **output gate** shipped in the
generate freshness-check thread at
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260531193119Z-generate-check-freshness/specs/260620090906Z-v1-spec.md`.
`validate` asks "is the template authorable?"; `generate --check` asks "does the
committed artifact still match what the template produces?" A template can pass
`validate` and fail `generate --check` (valid source, stale committed copy); a
template that fails `validate` fails everything downstream. Choosing `--check`
for the output gate deliberately reserved the bare word `check`, so the input
gate takes the unambiguous name `validate`.

## Scope And Non-Scope

In scope:

- A new top-level command `jastr validate <template-ref>` that runs static
  validation and reports pass/fail, writing nothing.
- `<template-ref>` accepts every form `run`/`generate` accept: bare named,
  grouped named, named variant (`<ref>#<variant-id>`), and direct `.md` path.
- The "everything statically checkable" check scope (decision (b), ratified with
  the user): the run-side static pipeline always, plus the agent-skill target
  metadata shape **when the resolved ref declares it**, plus â€” for a variant ref
  â€” the selected variant config, locked inputs against the schema, and the merged
  agent-skill frontmatter.
- A single concise stdout success line, exit 0; uniform `Error:` + exit 1 on
  failure reusing existing engine error codes.
- CLI argv validation for the new command, Commander wiring, the updated
  command-shape message, functional requirements in a new `13-validate.yml`, a
  help update in `07-help.yml`, e2e cases, README, `AGENTS.md`, and `BEHAVIOR.md`.

Out of scope:

- `--watch` / re-validate-on-save. The path-free, output-free shape of `validate`
  is what would later make a watch loop clean, but no watch mode ships here.
- A multi-template / all-templates sweep (`validate` across every template under
  `.jastr/` for a CI pass). `validate` takes exactly one ref in this version.
- Any input flags or `--out`. `validate` evaluates no invocation inputs and
  writes nothing.
- A "run-only" narrowing flag. Decision (ratified): the check scope is fixed at
  (b) with no flag (YAGNI); there is no mode that skips the agent-skill target
  shape check when a target is declared.
- Requiring agent-skill metadata. `validate` never *requires* `targets.agent-skill`,
  even for a variant ref â€” it only validates such metadata when declared. A
  template (or variant) with no agent-skill target is still valid because it
  remains runnable via `jastr run`. This keeps `validate` an input gate, not a
  generate-readiness gate, and means a variant ref can pass `validate` yet fail
  `generate agent-skill` for lack of a target â€” the intended asymmetry.
- New error codes. `validate` surfaces only existing `JastrErrorCode` values.

## Expected Behavior

### Command Surface

```text
jastr validate <template-ref>
```

`validate` takes exactly one positional `<template-ref>` and no options. It must
be added to the CLI argv command allow-list (which today admits only `run` and
`generate`) and wired as a Commander command. A missing ref, an unexpected extra
positional, or any flag is an `invalid_command` failure (exit 1):

- Missing ref â†’ `Missing template reference for validate.`
- Any option â†’ `Unknown validate option <option>.`
- Extra positional â†’ `Invalid validate argument <argument>.`

These mirror the existing `run`/`generate` argv-shape messages and are exact
contracts. `--help`/`-h` after `validate` remains the informational help path
(exit 0).

### Validation Pipeline

`validate` reuses the CLI's existing template loading and the engine's validation
API; it introduces no new compiler logic, so `run`, `generate`, and `validate`
cannot drift. For the resolved ref it performs, in order:

1. Resolve the project root and load the template exactly as `run`/`generate`
   do (`loadTemplateReference`). Direct `.md` refs do not read `.jastr/config.yml`,
   consistent with every other command.
2. `parseTemplateSource` then `validateTemplateSchema` over the frontmatter.
3. For a variant ref, resolve and validate the selected variant config with the
   existing selected-only strict checks (per
   `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/specs/260614165823Z-v2-spec.md`):
   `variant_not_found` when absent, `invalid_config` for malformed selected
   shape or unknown selected fields.
4. A static render (`renderTemplateSource`) using sampled synthetic inputs for
   declared inputs, and â€” for a variant ref â€” the real selected `locked-inputs`
   values for locked keys (the same `sampleInputsForStaticRender` strategy
   `generate` uses). This exercises directive parsing, condition parsing,
   condition/interpolation reference checks, include resolution and containment,
   missing-include detection, include-cycle detection, and validates the
   effective inputs (including defaults and locked values) through the engine.
   It does **not** assert which branch a real invocation would take â€” there are
   no invocation inputs â€” only that the body is statically well-formed and
   renderable.
5. When the resolved ref declares agent-skill target metadata â€” the template's
   `targets.agent-skill`, overlaid by any selected variant
   `agent-skill.frontmatter` for a variant ref â€” validate that effective metadata
   with the existing agent-skill target/frontmatter rules. When no agent-skill
   metadata is declared anywhere for the ref, this step is skipped and is not an
   error.

Any defect raised at any step aborts with that defect's existing message and
code; `validate` never remaps them.

### Success Output

On a clean pass, `validate` writes a single line to stdout and exits `0`, writing
nothing to disk:

```text
Template <template-ref> is valid.
```

`<template-ref>` is echoed exactly as provided on the command line, including a
`#<variant-id>` suffix or a `.md` path. This string is an exact contract for
tests and docs.

### Error Output And Codes

All failures print `Error: <message>` to stderr and exit `1`, per the uniform
error contract. `validate` introduces no new error codes; it reuses the existing
ones the same defect already produces under `run`/`generate`, including but not
limited to: `invalid_template_reference`, `missing_project_root`,
`template_not_found`, `invalid_frontmatter`, `malformed_schema`,
`invalid_directive`, `invalid_input_name`, `condition_parse_error`,
`undeclared_condition_input`, `invalid_interpolation`,
`absent_optional_interpolation`, `include_not_found`, `include_read_error`,
`include_outside_root`, `invalid_include_root`, `include_group_missing`,
`include_cycle`, `invalid_target_metadata`, `missing_target_metadata`,
`variant_not_found`, `invalid_config`, and `invalid_command` for argv-shape
errors.

## Constraints And Active Contract Updates

- **Pure reuse; package boundaries unchanged.** `validate` is entirely a
  `@jastr/cli` command â€” argv parsing, project discovery, variant resolution,
  agent-skill target validation, output formatting, exit code. It composes the
  existing engine API (`parseTemplateSource`, `validateTemplateSchema`,
  `renderTemplateSource`) and existing CLI helpers (`loadTemplateReference`,
  `loadProjectConfigVariant`, `createFileIncludeResolver`,
  `sampleInputsForStaticRender`, `validateAgentSkillTarget` /
  `validateAgentSkillFrontmatter`). `@jastr/engine` gains nothing â€” no new
  exports and, notably, no new error codes. No filesystem, Commander, argv, or
  Bun runtime dependency is added to `@jastr/engine`.
- **No new compiler logic.** `validate` must not re-implement any validation step;
  it calls the shared path so `run`, `generate`, and `validate` report the same
  defect identically. The natural seam is a small shared static-validation helper
  used by both `executeGenerate` and a new `executeValidate`, but whether to
  extract one or have `executeValidate` compose the engine calls directly is at
  the implementer's discretion provided behavior, messages, and codes match.
- **Uniform error UX preserved** (per
  `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260605091319Z-core-cli-package-split/specs/260606182808Z-v2-spec.md`):
  every failure is `Error: <message>` + exit 1; a clean validation exits 0 with
  the success line on stdout. No validate-specific exit-code vocabulary.
- **`expectedCommandShape` message contract update.** The shared command-shape
  string in `packages/cli/src/args.ts` must extend to name `validate`, e.g.:
  `Expected command shape: jastr run <template-ref> [input flags...], jastr generate agent-skill <template-ref> --out <path> [--check] [--force], or jastr validate <template-ref>.`
  This is an exact-message contract; existing e2e cases that assert command-shape
  errors must be updated to match.
- **Implementation surfaces that change together:** the argv command allow-list
  and a `validate` arg check in `packages/cli/src/args.ts`; the updated
  `expectedCommandShape`; a `makeValidateCommand` wired into
  `packages/cli/src/program.ts`; and a new `executeValidate` in
  `packages/cli/src/commands.ts`. Internal shape is at the implementer's
  discretion provided the public contract above holds.
- **Requirement ownership.** Add a new area file
  `packages/cli/requirements/functional/13-validate.yml` with a `VALIDATE-FR-<NNNN>`
  prefix (the `VAR` prefix is already taken by variants), and update
  `07-help.yml` so help lists `validate`. New e2e cases under
  `packages/cli/test/e2e/cases/` cover each public behavior with `covers:` refs
  (suggested slugs, not pinned: `validate-valid-template`,
  `validate-no-agent-skill-target`, `validate-invalid-schema`,
  `validate-missing-include`, `validate-include-cycle`,
  `validate-bad-agent-skill-target`, `validate-variant`,
  `validate-missing-variant`, `validate-direct-md`, `validate-missing-ref`).
- **Docs to keep in lockstep.** Update `README.md` and the `AGENTS.md` canonical
  commands list to include `jastr validate <template-ref>`, and regenerate
  `packages/cli/docs/BEHAVIOR.md` so `bun run docs:cli:living --check` passes.

## Unresolved Questions

None block emission. Flagged for later:

- **`--watch`.** A re-validate-on-save loop is the obvious next step and is what
  the path-free shape enables; deferred to its own thread.
- **Multi-template sweep.** A `validate`-all affordance for CI (and where the set
  of templates to sweep would be enumerated) is left open; for now CI invokes
  `validate` once per ref.
- **Config `inputs.<ref>` validation for bare refs.** `validate` deliberately does
  not pre-validate the overridable `inputs.<template-ref>` config block for a
  bare named ref, matching how `run` declines to pre-validate config inputs
  (they may be intentionally overridden). Whether a separate config-lint should
  check that block is a different feature, out of scope here.

## Acceptance Guidance

The implementation is correct when a reviewer can verify all of the following:

- `jastr validate <ref>` on a well-formed template exits `0`, prints
  `Template <ref> is valid.` to stdout, and writes nothing.
- A template that is well-formed for `run` but declares **no** `targets.agent-skill`
  passes `validate` (absent target metadata is not an error).
- A template with a malformed `targets.agent-skill` (bad `name`, over-long
  `description`, non-kebab passthrough field, reserved field) fails with the same
  `invalid_target_metadata`/`missing_target_metadata` message and code `generate`
  produces for that defect.
- Schema, directive, condition, interpolation, and include defects â€” including a
  missing include and an include cycle â€” fail with the exact existing message and
  code, identical to what `run`/`generate` emit for the same template.
- `jastr validate <ref>#<variant-id>` validates the selected variant config
  (failing with `variant_not_found` when absent and `invalid_config` for
  malformed/unknown selected shape), validates locked inputs against the base
  schema through the engine, and validates the merged agent-skill frontmatter
  when any agent-skill metadata is present; a variant ref with no agent-skill
  metadata anywhere still passes.
- Direct `.md` refs validate against the file without reading `.jastr/config.yml`;
  grouped refs (`group/template`) and grouped variant refs (`group/template#variant`)
  resolve and validate correctly.
- `validate` with no ref, with an unexpected extra positional, or with any flag
  fails with `invalid_command` and the exact argv-shape messages above; `validate
  --help` shows help and exits 0.
- No new error codes were added and `@jastr/engine` gained no new exports,
  filesystem, Commander, or Bun dependency.
- `expectedCommandShape` names `validate`, and existing command-shape e2e cases
  are updated to the new string.
- Functional requirements exist in `13-validate.yml` with e2e cases covering each
  behavior via `covers:` refs, `07-help.yml` lists `validate`, and `README.md`,
  `AGENTS.md`, and `packages/cli/docs/BEHAVIOR.md` describe the same contract.
