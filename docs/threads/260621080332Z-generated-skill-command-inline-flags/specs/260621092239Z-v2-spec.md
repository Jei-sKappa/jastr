# Spec: Generated Agent Skill — inline input flags in the command (v2)

## Status

Active v2 contract for how the generated Agent Skill wrapper renders a
template's inputs **inside the `jastr run …` command** rather than leaving the
command bare. This spec supersedes, by prose, specific rules of the show-inputs
v2 spec
(`docs/threads/260531211938Z-generated-router-skill-show-inputs/specs/260620203922Z-v2-spec.md`)
and formally closes that thread's deferred P4 (the placeholder skeleton); the
prior spec file is not edited (see "Supersession").

Changed from v1 (`260621091809Z-v1-spec.md`, locked): **presentation only, no
contract change.** v1 rendered the two single-input pinned sentences in the
"Per-input descriptor formatter" section using blockquote code-spans and a
literal `&nbsp;`. Because this spec's contract is `--check` byte-equality, a
non-breaking space (U+00A0) shown where the generator emits a regular space
(U+0020) misrepresented a byte. v2 presents those pinned sentences in fenced
`text` blocks (the same style as the body-shape and worked-example blocks), where
backticks are literal and spaces are real. No pinned string, shape rule, or
decision changed.

Every settled trade-off in this spec is cited inline back to its source decision
in
`docs/threads/260621080332Z-generated-skill-command-inline-flags/discussions/260621080332Z-command-inline-flags-design-decision-log.md`
by its heading id. A citation like `(per <log-path> P3)` points at that log's
`## P3:` record. Where a rule originates in the earlier show-inputs thread, the
citation names that thread's log
(`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`)
explicitly.

## Intended outcome

When `jastr generate agent-skill <template-ref> --out <path>` is run against a
template (or variant) that declares **required** inputs, the generated wrapper's
command block carries those flags as `--flag=<value>` placeholders the agent
fills in — e.g. `jastr run demo --language=<value>` — instead of the bare
`jastr run demo` it emits today. A wrapper for a template with exactly **one**
input drops the `## Inputs` section entirely and folds that single input into a
tailored instruction sentence, so the most common simple skill reads as a
purpose-built wrapper around its one parameter rather than a generic list of
one.

The result: a generated wrapper no longer looks incomplete when a required input
is missing from the command, and an agent reading it has both the inputs view
and a command shaped like the invocation it must produce.

## Context

This work was triggered by a backlog observation: the show-inputs feature (just
landed on branch `feat/generated-router-skill-show-inputs`) renders a `## Inputs`
section and a construct-flags instruction, but keeps the command line a **bare**
`jastr run <template-ref>` in all cases. For a template whose sole input is a
required one, `jastr run demo` with no flag reads as odd/incomplete and weakly
invites the agent to run it verbatim and trip `run`'s validation.

The show-inputs thread considered putting flags into the command — a "placeholder
usage skeleton" — and **deliberately deferred it** under YAGNI
(`docs/threads/260531211938Z-generated-router-skill-show-inputs/discussions/260620172031Z-show-inputs-spec-design-decision-log.md`
P4), with an explicit "revisit only on evidence agents make flag mistakes"
clause and a note that the upgrade path is a pure generator change with no
schema or contract impact. This thread revisits that deferral on aesthetic/trust
grounds and adopts the upgrade (per
`docs/threads/260621080332Z-generated-skill-command-inline-flags/discussions/260621080332Z-command-inline-flags-design-decision-log.md`
P1).

The change is implemented entirely in `@jastr/cli`'s generator. The engine is
untouched: `description`, `required`, `default`, `type`, and the unlocked-input
list all already exist (the show-inputs thread added `description`), and no new
`JastrErrorCode` is introduced (per
`docs/threads/260621080332Z-generated-skill-command-inline-flags/discussions/260621080332Z-command-inline-flags-design-decision-log.md`
P7).

## Scope

In scope:

- Inlining **required** rendered inputs into the generated command as
  `--<name>=<value>`, in template declaration order (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P1, P2, P3).
- A dedicated **single-input** rendering path (no `## Inputs` section; a tailored
  instruction sentence) for any wrapper with exactly one rendered input, with
  separate sentences for the required and optional cases (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P2 choice C, P5).
- Selecting the body shape by **rendered input count** — all declared inputs for
  a non-variant ref, the unlocked inputs for a variant ref (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P6).
- A shared per-input descriptor formatter feeding both the multi-input bullet and
  the single-input sentence (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P5).

Out of scope (explicitly not built here):

- Inlining **optional** inputs into the command. Optionals never appear in the
  command block; they live in the `## Inputs` bullets (2+ case) or the
  single-input optional prose. A lone optional input keeps a bare command (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P2 choice C, P4).
- **Bracket** usage notation (`[--flag=<value>]`) anywhere in the output. It is
  not introduced for the lone-optional case or any other (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P4).
- Splitting `## Inputs` into "Required"/"Optional" subsections. The section stays
  one flat list (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P2).
- **Type-aware** command placeholders (`<true|false>`, `<dev|prod>`). The
  placeholder is the uniform literal `<value>` for every type (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P3).
- **Rewording** the multi-input construct sentence. It is kept byte-for-byte as
  the show-inputs spec pinned it (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P5).
- Any `@jastr/engine`/schema change, any new `JastrErrorCode`, and any frontmatter
  change (inputs never leak into frontmatter) (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P7, P1).

## The four body shapes

The generator emits exactly one of four body shapes. The frontmatter block
(`---` … `---`, built unchanged from `target.name`, `target.description`, and
passed-through fields via `buildAgentSkillContent`,
`packages/cli/src/targets/agent-skill.ts:187-192`) and the trailing failure line
`If the command exits non-zero, report the exact error output to the user and
stop.` are **identical in all four shapes**. `$ARGUMENTS` appears in none of them
(already removed by the show-inputs thread; this invariant is preserved).

`{ref}` is the requested ref as supplied to `generate`
(`template.requestedTemplateRef`, `packages/cli/src/commands.ts:160`), so variant
refs render as `jastr run <ref>#<variant-id>`. `{command}` is defined under
"Command construction" below. `{frontmatterSource}` is the existing
`YAML.stringify(frontmatter).trimEnd()` output. (Each body template below uses a
nested ```` ```bash ```` fence; read these spec blocks as source — the nested
fence is the established house style in the sibling show-inputs spec.)

**Shape A — no rendered inputs.** Used when the rendered input list is empty (a
template with no inputs, or a variant that locks all declared inputs). This is
**byte-identical to today's Shape 1** (`agent-skill.ts:195-208`):

```text
---
{frontmatterSource}
---

Run this command and follow its output exactly:

```bash
{command}
```

If the command exits non-zero, report the exact error output to the user and stop.
```

**Shape B — single required input.** Used when exactly one input is rendered and
it is `required: true`:

```text
---
{frontmatterSource}
---

{singleton-required-sentence}

```bash
{command}
```

If the command exits non-zero, report the exact error output to the user and stop.
```

**Shape C — single optional input.** Used when exactly one input is rendered and
it is `required: false`. The command stays bare because optionals are never
inlined (per `…/260621080332Z-command-inline-flags-design-decision-log.md` P4):

```text
---
{frontmatterSource}
---

{singleton-optional-sentence}

```bash
{command}
```

If the command exits non-zero, report the exact error output to the user and stop.
```

**Shape D — two or more rendered inputs.** Used when ≥2 inputs are rendered.
Structurally identical to today's Shape 2 (`agent-skill.ts:214-229`), with the
only change being that `{command}` now carries the inlined required flags:

```text
---
{frontmatterSource}
---

## Inputs

{bullets}

Map the user's request to the inputs above and append them as `--flag=value` arguments, including every required input. Then run this command and follow its output exactly:

```bash
{command}
```

If the command exits non-zero, report the exact error output to the user and stop.
```

All four bodies end with a single trailing newline, as the current generator
does.

## Shape selection rule

The "rendered inputs" are exactly the list the existing call site already
computes and passes to `buildAgentSkillContent`:
`listUnlockedTemplateInputs(schema, selectedVariant?.lockedInputs ?? {})`
(`packages/cli/src/commands.ts:162-165`, `packages/cli/src/variants.ts:61-68`).
For a non-variant ref the locked map is empty, so this returns **all** declared
inputs in declaration order; for a variant ref it returns the **unlocked** inputs
in declaration order. The shape is chosen from this list (per
`…/260621080332Z-command-inline-flags-design-decision-log.md` P6):

- `length === 0` → **Shape A**.
- `length === 1` and the single input has `required: true` → **Shape B**.
- `length === 1` and the single input has `required: false` → **Shape C**.
- `length >= 2` → **Shape D**.

A variant's unlocked inputs keep their own `required`/`optional` status from the
schema (a variant locks inputs by supplying values; it does not change the
required-ness of the inputs it leaves open). So a variant with one unlocked
required input → Shape B; one unlocked optional input → Shape C; all locked →
Shape A.

## Command construction

`{command}` is built uniformly for every shape. It starts as:

```text
jastr run {ref}
```

followed by, for **each rendered input whose `required` is `true`, in
declaration order**, the suffix (one leading space, the backtickless flag,
`=<value>` with the literal token `<value>`):

```text
 --{name}=<value>
```

Consequences, all of which fall out of this one rule (per
`…/260621080332Z-command-inline-flags-design-decision-log.md` P2, P3, P4):

- Shape A and Shape C never have a required rendered input, so `{command}` is the
  bare `jastr run {ref}`.
- Shape B always has exactly one required rendered input, so `{command}` is
  `jastr run {ref} --{name}=<value>`.
- Shape D inlines its required inputs only; its optional inputs are **not**
  appended. A Shape D wrapper whose rendered inputs are **all optional** yields a
  bare `jastr run {ref}` — byte-identical to today's Shape 2 command.

The placeholder is the uniform literal `<value>` for every input type, including
`boolean` (rendered `--<name>=<value>`, never the bare `--<name>` form), which is
why a boolean's true/false choice stays expressible (per
`…/260621080332Z-command-inline-flags-design-decision-log.md` P3). With
placeholders the command block is a fill-in template for the agent, not a
copy-paste-runnable line; this is intended.

## Per-input descriptor formatter

A single descriptor routine computes three segments from a
`TemplateInputDefinition`; both the Shape D bullet and the single-input sentence
consume it (per `…/260621080332Z-command-inline-flags-design-decision-log.md`
P5). The bullet additionally includes the required/optional token; the sentence
omits it because the sentence's own wording carries that. This mirrors the
existing `renderInputBullet` (`agent-skill.ts:232-248`), which stays the source
of truth for the bullet:

- `{type-token}`:
  - `string` → `string`
  - `boolean` → `boolean`
  - `enum` → `enum: ` followed by the declared values joined with `|` (pipe), in
    declared order, e.g. `enum: dev|prod`.
- `{default-seg}`: empty unless the input is optional **and** declares a
  `default`, in which case `, default: {default-value}`, where `{default-value}`
  is `String(definition.default)` (boolean → `true`/`false`; string/enum → the
  literal default string). Required inputs never carry a default
  (`packages/engine/src/schema.ts`), so this segment never co-occurs with the
  required wording.
- `{desc-seg}`: empty unless the input declares a `description`, in which case
  ` — {description}` (one space, an em dash U+2014, one space, then the verbatim
  description, echoed with no escaping).

**Shape D bullet** (unchanged from `renderInputBullet`). Each bullet, joined by
single newlines in declaration order, where `{req-token}` is `required` or
`optional`:

```text
- `--{name}` ({type-token}, {req-token}{default-seg}){desc-seg}
```

**Single-input sentences.** These are pinned **verbatim** for `--check`; the
backticks shown are literal characters in the emitted Markdown, and every space
is a regular space U+0020 (per
`…/260621080332Z-command-inline-flags-design-decision-log.md` P5).

Shape B — `{singleton-required-sentence}`:

```text
This skill takes one input, `--{name}` ({type-token}){desc-seg}. Fill in `--{name}=<value>` from the user's request. Then run this command and follow its output exactly:
```

Shape C — `{singleton-optional-sentence}`:

```text
This skill takes one optional input, `--{name}` ({type-token}{default-seg}){desc-seg}. Add `--{name}=<value>` if the user's request calls for it; otherwise leave it out. Then run this command and follow its output exactly:
```

Notes on the single-input sentences:

- The flag name is backticked (`` `--{name}` ``); the parenthetical
  `({type-token}…)` is plain text; `{desc-seg}` (when present) is appended
  **after** the closing parenthesis, exactly as in the bullet.
- There is **no** `required`/`optional` token inside the parenthesis — the phrase
  "one input" vs. "one optional input" carries that distinction.
- Shape B's parenthesis is just `({type-token})` because a required input never
  carries a default, so `{default-seg}` is always empty there.
- For an optional input **without** a default, `{default-seg}` is empty and
  "otherwise leave it out" still reads correctly (omitting the flag yields either
  the author default or absence).

**Multi-input construct sentence (Shape D).** Kept **byte-for-byte** as the
show-inputs spec pinned it (per
`…/260621080332Z-command-inline-flags-design-decision-log.md` P5):

```text
Map the user's request to the inputs above and append them as `--flag=value` arguments, including every required input. Then run this command and follow its output exactly:
```

## Worked examples (byte-exact bodies, frontmatter elided)

**Shape B — enum, required, with description** (`demo`, input `language`,
`enum: [typescript, python]`, description `target language`):

```text
This skill takes one input, `--language` (enum: typescript|python) — target language. Fill in `--language=<value>` from the user's request. Then run this command and follow its output exactly:

```bash
jastr run demo --language=<value>
```

If the command exits non-zero, report the exact error output to the user and stop.
```

**Shape B — string, required, no description** (`demo`, input `tag`):

```text
This skill takes one input, `--tag` (string). Fill in `--tag=<value>` from the user's request. Then run this command and follow its output exactly:

```bash
jastr run demo --tag=<value>
```
```

**Shape C — enum, optional, with default + description** (`demo`, input
`language`, default `typescript`, description `target language`):

```text
This skill takes one optional input, `--language` (enum: typescript|python, default: typescript) — target language. Add `--language=<value>` if the user's request calls for it; otherwise leave it out. Then run this command and follow its output exactly:

```bash
jastr run demo
```
```

**Shape C — boolean, optional, no default, no description** (`demo`, input
`verbose`):

```text
This skill takes one optional input, `--verbose` (boolean). Add `--verbose=<value>` if the user's request calls for it; otherwise leave it out. Then run this command and follow its output exactly:

```bash
jastr run demo
```
```

**Shape D — mixed** (`deploy`; `env` enum `[dev, prod]` required, description
`target environment`; `region` string optional default `us-east-1`, description
`deployment region`; `dry-run` boolean optional, description `preview without
applying`). Only `env` is required, so only it is inlined:

```text
## Inputs

- `--env` (enum: dev|prod, required) — target environment
- `--region` (string, optional, default: us-east-1) — deployment region
- `--dry-run` (boolean, optional) — preview without applying

Map the user's request to the inputs above and append them as `--flag=value` arguments, including every required input. Then run this command and follow its output exactly:

```bash
jastr run deploy --env=<value>
```
```

**Shape D — all required** (`x`; `tag` string required, `env` enum `[dev, prod]`
required). Both inlined in declaration order:

```text
jastr run x --tag=<value> --env=<value>
```

**Shape D — all optional** (`x`; `region` string optional default `us-east-1`;
`verbose` boolean optional default `false`). No required input → bare command,
byte-identical to today's Shape 2:

```text
jastr run x
```

**Shape B — variant with one unlocked required input** (`deploy#prod` where the
`prod` variant locks every input except a required string `tag`):

```text
This skill takes one input, `--tag` (string). Fill in `--tag=<value>` from the user's request. Then run this command and follow its output exactly:

```bash
jastr run deploy#prod --tag=<value>
```
```

## Signature and call-site changes

- `buildAgentSkillContent` (`packages/cli/src/targets/agent-skill.ts:182`) keeps
  its current option shape (`templateRef`, `target`, `inputs:
  ReadonlyArray<{ name; definition }>`). The `inputs` array it already receives
  **is** the rendered list, so shape selection, the required/optional partition,
  and command construction all happen **inside** this function. No new parameter
  is required.
- `executeGenerate` (`packages/cli/src/commands.ts:159-166`) already passes
  `listUnlockedTemplateInputs(schema, selectedVariant?.lockedInputs ?? {})`,
  which is the correct rendered list for both non-variant and variant refs (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P6). **No
  call-site change is anticipated** — the junior dev should confirm this rather
  than refactor the call site.
- `renderInputBullet` (`agent-skill.ts:232-248`) stays for Shape D. Factor its
  `{type-token}`/`{default-seg}`/`{desc-seg}` computation into the shared
  descriptor routine so the single-input sentence reuses the identical segments
  (DRY); the bullet then composes those segments plus the `{req-token}`.
- `checkAgentSkillOutput` / `writeAgentSkill` / `assertAgentSkillOutputAvailable`
  (`agent-skill.ts:254-313`) are unchanged; they operate on the produced
  `content` string.

## Constraints

- **Determinism / byte-stability.** Generated output must be a pure function of
  template + selected variant. The command (`jastr run <ref>` + required flags in
  declaration order), the two single-input sentences, the multi-input sentence,
  and the `|` / `, ` / em-dash separators are all fixed, so `--check`'s exact-byte
  comparison (`checkAgentSkillOutput`, `agent-skill.ts:304`, compares raw
  `Buffer`s with no normalization) never flaps (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P5).
- `@jastr/engine` must stay free of generator/FS/CLI concerns; all body rendering
  remains in `@jastr/cli`. This change touches no engine source.
- **No new `JastrErrorCode`.** The generate/check paths reuse
  `output_stale` / `output_missing` / `output_exists` (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P1, P7).
- Frontmatter pass-through is unchanged; `inputs` never leaks into frontmatter
  (`RESERVED_FRONTMATTER_FIELDS`, `agent-skill.ts:10`). The inputs view is
  body-only.
- Node-compatible source only (no Bun-specific runtime APIs) in both packages.

## Expected behavior (observable)

1. A no-input template generates Shape A, **byte-identical to before this
   change**.
2. A template with one required input generates Shape B: no `## Inputs` section,
   the tailored required sentence, and a command carrying `--<name>=<value>`.
3. A template with one optional input generates Shape C: no `## Inputs` section,
   the tailored optional sentence, and a **bare** command.
4. A template with ≥2 inputs generates Shape D: a `## Inputs` bullet per rendered
   input in declaration order, the verbatim multi-input sentence, and a command
   that inlines the **required** inputs (in declaration order) and omits the
   optionals.
5. A ≥2-input template whose inputs are **all optional** generates Shape D with a
   bare command — **byte-identical to today's Shape 2**.
6. For a variant ref, the shape is chosen by the **unlocked** input count: one
   unlocked → Shape B/C; ≥2 unlocked → Shape D; all locked → Shape A. Locked
   inputs appear nowhere (per the show-inputs locked-inputs-invisible rule,
   `…/260620172031Z-show-inputs-spec-design-decision-log.md` P5).
7. A boolean input is inlined as `--<name>=<value>`, never the bare `--<name>`.
8. No generated wrapper contains the token `$ARGUMENTS` (preserved invariant).
9. `generate agent-skill --check` byte-compares the committed file against the new
   body for its shape: exit 0 with `agent-skill at <out> is up to date.` on an
   exact match, `output_stale` when bytes differ, `output_missing` when absent —
   mechanism unchanged (`agent-skill.ts:280-313`).

## Supersession

This spec supersedes, by prose, only the following rules of the show-inputs v2
spec
(`docs/threads/260531211938Z-generated-router-skill-show-inputs/specs/260620203922Z-v2-spec.md`);
that file is not edited:

- Its "Two body shapes" model (Shape 1 / Shape 2) and the rule that "the command

  **four** shapes, and the command inlines required inputs as `--<name>=<value>`.

presentation only, as noted in "Status"; the contract is identical.

## Unresolved questions (non-blocking)

- The command placeholder is the uniform literal `<value>`. A `string` input
  whose placeholder the agent fails to substitute would be accepted silently as
  the literal `<value>` — an inherent risk of inlining strings that no
  placeholder choice or engine check can catch (per
  `…/260621080332Z-command-inline-flags-design-decision-log.md` P1, P3). It is
  accepted, not mitigated; revisit only on evidence agents leave string
  placeholders unsubstituted in practice. No trigger metric is defined.
- A template with many required inputs produces a long command line. This is
  judged the author's call and is not capped or wrapped here.
