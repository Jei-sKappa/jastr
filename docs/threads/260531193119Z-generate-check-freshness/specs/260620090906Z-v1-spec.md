# Generate Agent-Skill Freshness Check (`--check`) v1 Spec

## Intended Outcome

`jastr generate agent-skill` gains a `--check` mode that re-builds the agent-skill
wrapper in memory and byte-compares it against the file already committed at
`--out`, without writing anything. It exits `0` when the committed file is
identical to what the template would produce now, and exits `1` with a stable
`Error:` line when the committed file is stale (differs) or missing.

The capability answers one question for CI and pre-commit hooks: *"is the
committed agent-skill at this path still up to date with its template?"* It
closes a currently silent drift â€” when a template's frontmatter or body changes,
the committed wrapper goes stale and nothing detects it until someone
regenerates and eyeballs the diff. After this spec, a single command gates that
freshness deterministically.

## Context

`generate agent-skill` is Jastr's only command that commits an artifact to the
repository. The wrapper it writes (e.g. into `.claude/skills/<skill>/SKILL.md`)
is derived entirely from the template: `targets.agent-skill.frontmatter`, the
template ref, whether the template still declares unlocked inputs, and â€” for
variant refs â€” the selected variant's frontmatter overlay and locked inputs.
When any of those inputs change, the committed wrapper drifts silently. Hand
editing the committed wrapper (not realizing it is generated) is the same failure
in reverse.

This is a known, solved pattern inside this very repository:
`bun run docs:cli:living --check` re-renders `packages/cli/docs/BEHAVIOR.md` in
memory and exits `1` if it differs from the committed file. This spec extends the
same trusted "rebuild-in-memory, compare, exit non-zero on mismatch, write
nothing" shape to the one place Jastr emits a committed artifact and currently
has no freshness guard.

## Scope And Non-Scope

In scope:

- A `--check` boolean option on `jastr generate agent-skill`.
- Building the agent-skill content exactly as a normal generate would (full
  template validation, static render validation, target/variant frontmatter
  validation) and then comparing it byte-for-byte against the file at `--out`
  instead of writing.
- Three terminal outcomes â€” match (exit 0), stale (exit 1), missing (exit 1) â€”
  with stable messages and error codes.
- Flag-combination rules: `--check` requires `--out` (inherited) and is
  rejected when combined with `--force`.
- Variant refs: `jastr generate agent-skill <template-ref>#<variant-id> --check`
  compares against the variant-specific generated content.
- CLI argv pre-validation, Commander option wiring, the updated command-shape
  message, new error codes, functional requirements in `06-generate.yml`, e2e
  cases, README, `AGENTS.md`, and `BEHAVIOR.md` updates needed to make the
  feature discoverable and traceable.

Out of scope:

- A standalone `validate` command. `--check` guards the **output** (does the
  committed artifact still match?); `validate` would guard the **input** (is the
  template authorable?). They are complementary and tracked in a separate thread.
  Naming note: choosing `--check` for the output gate deliberately reserves the
  bare word `check` here so a future input gate can take the name `validate`
  without collision.
- Diff output. On mismatch the command emits a single concise `Error:` line, not
  a rendered diff. A verbose/diagnostic diff mode may be considered later.
- A provenance header (e.g. `<!-- generated by jastr; edit the template -->`) in
  the wrapper. Adding one is a separable change to the wrapper byte contract; it
  is not introduced here, and `--check` compares the bytes exactly as
  `buildAgentSkillContent` produces them today.
- A "check all committed outputs" sweep. `--check` is inherently per-`--out`; a
  CI sweep invokes it once per committed destination. A higher-level multi-output
  affordance (and where the destination list would live) is unresolved and out
  of scope.
- Distinct exit codes for stale versus missing. Per the uniform error contract,
  every failure exits `1`; the two cases are distinguished by message and error
  code, not exit code.

## Expected Behavior

### Command Surface

```text
jastr generate agent-skill <template-ref> --out <path> [--check] [--force]
```

`--check` is a boolean flag. It must be accepted by the generate argv
pre-validator and declared as a Commander option; the pre-validator currently
rejects unknown generate options, so without whitelisting it `--check` would fail
with `Unknown generate option --check.` `<template-ref>` accepts the same forms
as a normal generate: bare named, grouped named, named variant
(`<ref>#<variant-id>`), or a direct `.md` path.

### Build-Then-Compare, Not Build-Then-Write

In `--check` mode the command performs every step a normal
`generate agent-skill` performs up to and including constructing the wrapper
content with `buildAgentSkillContent` â€” template loading, `parseTemplateSource`,
`validateTemplateSchema`, variant resolution and selected-config validation,
target/merged-frontmatter validation, and the static render validation
(`renderTemplateSource` with sampled inputs, using real locked-input values for
variant refs). It then diverges at the final step:

- normal mode â†’ write the content to `--out`.
- `--check` mode â†’ read the existing file at `--out` and compare, writing
  nothing.

A consequence of building first: if the template itself is now invalid â€” bad
schema, render error, missing or invalid `targets.agent-skill` metadata,
missing/malformed selected variant â€” `--check` fails with that existing
validation error and error code, **not** a stale/missing error. `--check` only
reports stale or missing when the template is valid and the wrapper would build
successfully but the committed file does not match. This is the sense in which
`--check` contains the static validation a future `validate` command would run.

### Comparison Semantics

The comparison is exact byte-for-byte equality between the UTF-8 bytes of the
freshly built content and the UTF-8 bytes of the file at `--out`, including the
trailing newline `buildAgentSkillContent` emits. There is no end-of-line, BOM,
or whitespace normalization: a committed file that differs only in line endings
or a stray trailing newline is correctly reported as stale. Exact comparison is
sound because generation is deterministic â€” `buildAgentSkillContent` serializes
frontmatter with `YAML.stringify` over insertion-ordered keys and a fixed body
template, so the built bytes are stable for a given template/ref across machines
and `--check` will not flap.

### Outcomes

The output path is resolved exactly as a normal generate resolves `--out`
(absolute as-is, otherwise relative to the working directory).

- **Match** â€” file exists and bytes are equal. Exit `0`. Write a single concise
  confirmation line to stdout (success message below). Write nothing to disk.
- **Stale** â€” file exists and bytes differ. Exit `1`, `Error:` to stderr, code
  `output_stale`. Write nothing.
- **Missing** â€” no file exists at the resolved path. Exit `1`, `Error:` to
  stderr, code `output_missing`. Write nothing. (Decision: a missing target is a
  check failure rather than a separate success/exit code, per the proposal's
  leaning and the uniform error contract.)

### Flag Validation

- `--check` requires `--out`. `--out` is already mandatory for
  `generate agent-skill`, so `--check` without `--out` produces the existing
  `missing_output_path` error `Missing required --out <path>.` unchanged â€” no new
  behavior.
- `--check` is mutually exclusive with `--force`. Combining them is a hard error
  (decision, ratified with the user): `--force` governs overwriting and `--check`
  never writes, so the combination is incoherent rather than ignorable. The
  conflict exits `1` with code `invalid_command`, consistent with the other
  generate flag-shape diagnostics that already live in the argv pre-validator.
- In `--check` mode the normal "output already exists" guard does **not** apply.
  A pre-existing file at `--out` is the expected input to a check, not the
  `output_exists` error case.

### Variant Refs

`jastr generate agent-skill <template-ref>#<variant-id> --check` resolves and
validates the selected variant exactly as variant generation does (per
`/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260614103530Z-generated-skill-variants/specs/260614165823Z-v2-spec.md`),
then compares against the variant-specific built content â€” merged
variant/base agent-skill frontmatter and the variant-aware wrapper command
(`jastr run <template-ref>#<variant-id>` with or without `$ARGUMENTS` depending
on whether any base input remains unlocked). No comparison logic is variant
specific; `--check` simply compares whatever a variant generate would have
written.

### Success And Error Output

The `--out` value is rendered in every message exactly as provided on the command
line (matching the existing `output_exists` message, which echoes `--out`
verbatim). `<template-ref>` is rendered as provided.

- Success (match), stdout, exit 0:

  ```text
  agent-skill at <out> is up to date.
  ```

- Stale, stderr `Error: ` prefix, exit 1:

  ```text
  Generated agent-skill at <out> is stale; regenerate it with jastr generate agent-skill <template-ref> --out <out> --force.
  ```

- Missing, stderr `Error: ` prefix, exit 1:

  ```text
  No agent-skill found at <out> to check; generate it with jastr generate agent-skill <template-ref> --out <out>.
  ```

- `--check` with `--force`, stderr `Error: ` prefix, exit 1:

  ```text
  --check cannot be combined with --force.
  ```

These four strings are exact contracts for tests and docs. The regenerate hints
deliberately reproduce the user's `<template-ref>` and `--out` value so a CI
failure tells the operator the exact command to run; the stale hint includes
`--force` because the target already exists.

### Error Code Mapping

- Stale committed artifact â†’ new code `output_stale`.
- Absent committed artifact â†’ new code `output_missing`.
- `--check` combined with `--force` â†’ existing `invalid_command`.
- Missing `--out` â†’ existing `missing_output_path` (unchanged).
- Any template/schema/render/target/variant defect surfaces with its existing
  code, unchanged â€” `--check` does not remap validation failures.

## Constraints And Active Contract Updates

- **Package boundaries unchanged.** `--check` is entirely a `@jastr/cli`
  concern: argv parsing, the read-and-compare step, output formatting, and exit
  codes. `@jastr/engine` gains nothing except the two new code literals
  `output_stale` and `output_missing` in the `JastrErrorCode` union â€” consistent
  with the union already owning CLI-only codes such as `output_exists`,
  `missing_output_path`, `locked_input_flag`, and `variant_not_found`. No engine
  logic, no filesystem access, no Commander, no Bun runtime APIs in either
  package's source.
- **Uniform error UX preserved** (per
  `/Users/jacopo/Developer/projects/personal/tools/jastr/docs/threads/260605091319Z-core-cli-package-split/specs/260606182808Z-v2-spec.md`):
  every failure prints `Error: <message>` to stderr and exits `1`; a successful
  check exits `0`. No new exit-code vocabulary is introduced.
- **Determinism is a load-bearing precondition.** The exact-byte comparison is
  only valid because `buildAgentSkillContent` is deterministic. Any future change
  that makes generation non-deterministic (e.g. timestamped output) must
  re-spec `--check` first.
- **`expectedCommandShape` message contract update.** The shared command-shape
  string in `packages/cli/src/args.ts` must change from
  `Expected command shape: jastr run <template-ref> [input flags...] or jastr generate agent-skill <template-ref> --out <path> [--force].`
  to
  `Expected command shape: jastr run <template-ref> [input flags...] or jastr generate agent-skill <template-ref> --out <path> [--check] [--force].`
  This is an exact-message contract; e2e cases that assert command-shape errors
  must be updated to match.
- **Implementation surfaces that must change together:** the generate argv
  pre-validator (whitelist `--check`, reject `--check` + `--force`), the
  Commander generate command (`.option("--check", ...)`), and `executeGenerate`
  (the build-then-compare branch and the skipped output-exists guard). Internal
  shape â€” helper names, where the conflict is detected, whether a compare helper
  is extracted â€” is at the implementer's discretion provided the public behavior,
  messages, and codes above hold.
- **Requirement ownership.** Add new `GEN-FR-<NNNN>` acceptance criteria to
  `packages/cli/requirements/functional/06-generate.yml`; do not create a new
  area file. New e2e cases under `packages/cli/test/e2e/cases/` cover each public
  outcome with `covers:` refs (suggested slugs, not pinned:
  `generate-check-up-to-date`, `generate-check-stale`, `generate-check-missing`,
  `generate-check-force-conflict`, `generate-check-variant`).
- **Docs to keep in lockstep.** Update `README.md` and the `AGENTS.md` canonical
  generate command line to include `[--check]`, and regenerate
  `packages/cli/docs/BEHAVIOR.md` so `bun run docs:cli:living --check` passes.

## Unresolved Questions

None block emission. Flagged for later:

- **Non-`ENOENT` read failures.** A `--out` path that exists but cannot be read
  as a regular file (e.g. it is a directory, or permission-denied) is not the
  same as "missing." This spec treats absent (`ENOENT`) as `output_missing` and
  match/stale for a readable file; other read errors are expected to surface as a
  generic exit-1 failure from the IO layer. Whether to give such cases a dedicated
  message/code is deferred.
- **Multiple committed destinations.** A "check every known output" sweep and the
  question of where a destination list would live (project config?) is left open;
  for now CI invokes `--check` once per destination.
- **Provenance header.** Stamping a "generated by jastr" comment into the wrapper
  would make hand-edits self-documenting and `--check` failures read more
  clearly, but it changes the wrapper byte contract and is intentionally not in
  this version.
- **Diff output.** A future diagnostic mode could print what differs; this
  version stays terse.

## Acceptance Guidance

The implementation is correct when a reviewer can verify all of the following:

- `jastr generate agent-skill <ref> --out <path> --check` against a committed
  file identical to freshly built content exits `0`, prints
  `agent-skill at <path> is up to date.` to stdout, and writes nothing.
- The same command against a committed file whose bytes differ (including a
  difference only in trailing newline or line endings) exits `1`, prints the
  exact `output_stale` message to stderr, and writes nothing.
- The same command when no file exists at `--out` exits `1`, prints the exact
  `output_missing` message to stderr, and writes nothing.
- `--check --force` together exits `1` with `--check cannot be combined with
  --force.` and code `invalid_command`; `--check` without `--out` exits `1` with
  the unchanged `Missing required --out <path>.`.
- `--check` against a template that is invalid (bad schema, render error,
  missing/invalid `targets.agent-skill`, missing/malformed selected variant)
  fails with that template's existing error message and code â€” not a
  stale/missing error.
- `jastr generate agent-skill <ref>#<variant-id> --out <path> --check` compares
  against variant-specific content (merged frontmatter and the variant-aware
  wrapper command), exiting `0` only when the committed file matches the variant
  output.
- Comparison is exact bytes with no normalization, and repeated runs on an
  up-to-date file are stable (no flapping).
- The argv pre-validator accepts `--check`, and `expectedCommandShape` reads the
  updated string including `[--check]`.
- `output_stale` and `output_missing` exist in the `JastrErrorCode` union and no
  engine logic, filesystem, or Commander dependency was added to `@jastr/engine`.
- Functional requirements are added to `06-generate.yml` with e2e cases covering
  each outcome via `covers:` refs, and `README.md`, `AGENTS.md`, and
  `packages/cli/docs/BEHAVIOR.md` describe the same contract.
