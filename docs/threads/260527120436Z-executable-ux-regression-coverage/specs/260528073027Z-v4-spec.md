# Functional Requirements Registry With AC-Covered E2E Cases V4 Spec

## Intended Outcome

This spec supersedes
`docs/threads/260527120436Z-executable-ux-regression-coverage/specs/260527192816Z-v3-spec.md`.

When implemented, Skillrouter will have:

- a stable functional requirements registry that names the product promises;
- requirement-local acceptance criteria with stable IDs;
- executable e2e cases that cover acceptance criteria through the real CLI;
- traceability validation that fails when active acceptance criteria are
  uncovered or when cases point at invalid lifecycle states.

The intended workflow is agent-friendly:

- "Add a functional requirement for X."
- "Remove requirement `GEN-FR-0003`."
- "Show uncovered acceptance criteria."
- "Implement until the e2e case covering `INPUT-FR-0003.AC-0001` passes."

The safety claim is precise: green means every active executable acceptance
criterion is covered by at least one e2e case and every e2e case passes. It does
not claim that broad natural-language prose is universally true beyond the
finite acceptance criteria recorded in the registry.

## Context

This thread started with executable UX regression coverage for Skillrouter, a
private early-stage CLI for deterministic AI-agent skill specialization.

Earlier approaches were useful but had boundary problems:

- V1 proposed executable Markdown case blocks. This proved the regression idea
  but was hard to read as final-user documentation.
- V2 proposed a VitePress documentation site backed by executable examples. This
  made the docs readable, but coupled executable cases to site rendering.
- V3 removed the docs-site layer and introduced functional requirements plus e2e
  cases, but left several important conventions ambiguous.

The clarification discussion at
`docs/threads/260527120436Z-executable-ux-regression-coverage/discussions/2026-05-27-v3-spec-clarification-design-discussion.md`
resolved those ambiguities. V4 incorporates those decisions:

- functional requirement IDs are area-first durable IDs such as
  `RUN-FR-0001`;
- e2e case IDs are lowercase descriptive slugs, not embedded requirement IDs;
- traceability is at the acceptance-criterion level, not the requirement level;
- case manifests use `covers`, not `requirementIds`;
- the version case uses a `{{version}}` harness placeholder;
- `bun run test` includes e2e, while `bun run test:e2e` is a focused subset;
- validator rejection behavior is unit-tested, without negative fixture trees;
- `AGENTS.md` and `README.md` receive surgical updates to remove stale docs-site
  claims.

## Scope

V4 includes:

- Removing the current VitePress documentation-site implementation and its
  docs-example coupling.
- Introducing `requirements/functional-requirements.yml` as the canonical
  functional requirements registry.
- Defining stable requirement IDs using `<AREA>-FR-<NNNN>`.
- Defining first-class acceptance criteria under each requirement, using
  `AC-NNNN` IDs scoped to that requirement.
- Migrating current executable docs examples into e2e cases under
  `test/e2e/cases/`.
- Removing site-specific fields from case manifests, including `render`,
  `hidden`, and any docs-reference field.
- Replacing case `requirementIds` with `covers`, where every entry is a
  composite acceptance-criterion reference such as
  `FLAGS-FR-0001.AC-0002`.
- Adding a Vitest e2e harness that validates schemas, validates traceability,
  and runs every case through the real Skillrouter CLI.
- Adding `bun run test:e2e` as a focused command while keeping e2e included in
  the default `bun run test`.
- Extending expected-output placeholders with `{{version}}`.
- Updating `AGENTS.md` and `README.md` to describe the requirement/e2e model and
  remove docs-site commands.

V4 does not include:

- Rebuilding, preserving, or replacing the VitePress documentation site.
- Generating final-user documentation from requirements or e2e cases.
- Rendering e2e cases into Markdown pages.
- Site-reference validation such as "every case must be referenced by docs".
- BDD/Gherkin/Cucumber.
- Browser automation or visual regression testing.
- A database, web UI, external requirements-management service, or hosted
  requirements portal.
- A new Skillrouter product command.
- A split Vitest project configuration that excludes e2e from the default
  `bun run test`.
- General regex or substring matchers for expected stdout/stderr.
- Refactoring core Skillrouter runtime behavior unless the migration exposes a
  real mismatch with the current v1 CLI contract.

## Architecture

The architecture has three layers:

```text
requirements/functional-requirements.yml
  Stable product contract.
  Contains FR records and acceptance-criterion records.

test/e2e/cases/<case-id>/case.yml
  Executable CLI behavior case.
  Covers one or more acceptance criteria.

test/e2e/*.ts
  Requirement loader, case loader, validator, runner, and tests.
```

The dependency direction is one-way:

```text
cases cover acceptance criteria
test harness reads requirements and cases
requirements do not reference test files
cases do not reference docs or site rendering
```

Functional requirements and e2e cases are separate concepts. A functional
requirement is the stable product promise and lifecycle record. Acceptance
criteria are the finite executable certification units. E2e cases are evidence
that acceptance criteria hold through the real CLI.

## Functional Requirements Registry

Create `requirements/functional-requirements.yml` as the canonical registry.
The file must be plain YAML with a top-level sequence of requirement records.

Each requirement record must use this shape:

```yaml
- id: RUN-FR-0001
  title: Run renders a skill template
  status: active
  description: >
    The CLI renders `.skillrouter/<skill>/SKILL.template.md` when the user runs
    `skillrouter run <skill>` from inside a project.
  acceptance:
    - id: AC-0001
      statement: Exits with code 0 when the skill exists and inputs are valid.
    - id: AC-0002
      statement: Prints rendered Markdown to stdout.
    - id: AC-0003
      statement: Prints nothing to stderr.
```

Required requirement fields:

- `id`: stable functional requirement ID.
- `title`: short human-readable requirement name.
- `status`: one of `active`, `deferred`, or `removed`.
- `description`: product-level behavior statement.
- `acceptance`: non-empty list of acceptance-criterion records.

Optional requirement fields:

- `notes`: additional product context.
- `replacedBy`: requirement ID that supersedes a `removed` requirement.
- `removedReason`: explanation for a `removed` requirement.
- `coverage`: for `deferred` requirements, a short reason executable coverage
  is deferred.

Required acceptance-criterion fields:

- `id`: acceptance criterion ID scoped to the parent requirement.
- `statement`: observable, finite acceptance statement.

Optional acceptance-criterion fields:

- `status`: only `removed` is supported in v4; absent means active.
- `removedReason`: required when `status: removed`.

Acceptance criteria are the coverage unit. For an `active` requirement, every
non-removed acceptance criterion must be covered by at least one e2e case.
Removed acceptance criteria are retained only for history and must not be
referenced by e2e cases.

## Requirement And Acceptance ID Rules

Functional requirement IDs must use:

```text
<AREA>-FR-<NNNN>
```

The validator must use this regex for functional requirement IDs:

```text
^[A-Z]+-FR-\d{4}$
```

Examples:

```text
RUN-FR-0001
INPUT-FR-0002
INCLUDE-FR-0001
GEN-FR-0003
```

Initial areas should match Skillrouter's v1 surface:

- `RUN`
- `GEN`
- `HELP`
- `VERSION`
- `FLAGS`
- `INPUT`
- `COND`
- `INCLUDE`

There is no `ERR` area. Error behaviors belong to the feature area they
constrain. For example:

- missing required input belongs under `INPUT`;
- invalid enum value belongs under `INPUT`;
- unsafe include rejection belongs under `INCLUDE`;
- generate overwrite refusal belongs under `GEN`.

Meaning lives in `title`, `description`, and `acceptance`, not in descriptor
segments inside the ID. IDs must never be renumbered or reused. If a
requirement is removed, keep the record and set `status: removed` with
`removedReason`.

Acceptance criterion IDs must use:

```text
AC-<NNNN>
```

The validator must use this regex for acceptance criterion IDs:

```text
^AC-\d{4}$
```

Acceptance criterion IDs are scoped per requirement. Each requirement may start
at `AC-0001`; `RUN-FR-0001.AC-0001` and `GEN-FR-0001.AC-0001` are different
criteria. Acceptance criterion IDs must not be renumbered or reused within
their parent requirement.

The composite reference to an acceptance criterion is:

```text
<FR-ID>.<AC-ID>
```

Example:

```text
FLAGS-FR-0001.AC-0002
```

## Initial Requirement Set

The initial registry must preserve the current executable behavior coverage
from the v2 docs examples, grouped into these functional requirements:

| Requirement ID | Product promise | Initial e2e case slug(s) |
| --- | --- | --- |
| `RUN-FR-0001` | Run renders a skill template | `basic-run` |
| `INPUT-FR-0001` | String input interpolates into output | `string-interpolation` |
| `INPUT-FR-0002` | Missing optional input interpolates as empty | `missing-optional-interpolation` |
| `INPUT-FR-0003` | Missing required input is rejected | `missing-required-input` |
| `INPUT-FR-0004` | Value outside an enum input's allowed set is rejected | `invalid-enum-value` |
| `FLAGS-FR-0001` | Boolean inputs accept bare, `=true`, and `=false` forms | `boolean-bare`, `boolean-true`, `boolean-false` |
| `FLAGS-FR-0002` | Unknown flag is rejected | `unknown-flag` |
| `FLAGS-FR-0003` | Duplicate flag is rejected | `duplicate-flag` |
| `COND-FR-0001` | Enum input selects an `if`/`else-if`/`else` branch | `enum-branch` |
| `COND-FR-0002` | Missing optional input takes the false branch | `missing-optional-condition` |
| `INCLUDE-FR-0001` | `include` renders a fragment | `include-fragment` |
| `INCLUDE-FR-0002` | `include-raw` emits verbatim text | `include-raw` |
| `INCLUDE-FR-0003` | Unsafe includes are rejected | `include-escape-rejected`, `include-env-rejected` |
| `GEN-FR-0001` | `generate` writes a router skill | `generate-router` |
| `GEN-FR-0002` | `generate` requires `--out` | `generate-missing-out` |
| `GEN-FR-0003` | `generate` refuses overwrite without `--force` | `generate-overwrite-refused` |
| `GEN-FR-0004` | `generate --force` overwrites | `generate-force` |
| `HELP-FR-0001` | CLI prints help for the program and commands | `help-root`, `help-run` |
| `VERSION-FR-0001` | Version prints package version and build SHA marker | `version` |

For requirements that group multiple observable forms, split those forms into
separate acceptance criteria. For example:

- `FLAGS-FR-0001.AC-0001`: bare boolean flag sets the input to true;
- `FLAGS-FR-0001.AC-0002`: `=true` sets the input to true;
- `FLAGS-FR-0001.AC-0003`: `=false` sets the input to false.

This avoids the false-positive where one case covers a broad requirement while
leaving part of the intended behavior untested.

## E2E Case Layout

Move executable cases under:

```text
test/e2e/cases/
```

Each case directory must contain:

```text
test/e2e/cases/<case-id>/
  case.yml
  project/
  expected/
```

The `project/` directory is copied into a temporary directory before execution.
The e2e runner must execute the real CLI entry point from the temp project, not
call internal rendering functions directly.

Case IDs are descriptive lowercase kebab-case slugs. They do not embed
functional requirement IDs.

The case ID regex is:

```text
^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$
```

Examples:

```text
basic-run
string-interpolation
generate-force-overwrite
include-env-rejected
```

Traceability lives solely in the case manifest's `covers` field. Because a case
may cover multiple acceptance criteria, there is no "primary requirement" in
the case ID.

## E2E Case Manifest

Each `case.yml` must contain only behavior and traceability information. It
must not contain site rendering metadata.

Required shape:

```yaml
id: string-interpolation
covers:
  - INPUT-FR-0001.AC-0001
title: Interpolate a string input
description: Expands a declared string input in rendered template output.
cwd: project
command: ["run", "demo", "--target-file=src/index.ts"]
expect:
  exitCode: 0
  stdout: "Analyze `src/index.ts`.\n"
  stderr: ""
```

Required fields:

- `id`: unique e2e case ID.
- `covers`: non-empty list of acceptance-criterion references.
- `title`: short human-readable case name.
- `description`: concise explanation of the behavior under test.
- `cwd`: safe relative path inside the copied temp project.
- `command`: non-empty array of Skillrouter CLI arguments, excluding the
  executable name.
- `expect.exitCode`: expected process exit code.
- `expect.stdout` or `expect.stdoutFile`.
- `expect.stderr` or `expect.stderrFile`.

Optional `expect` fields:

- `files`: mapping of generated file path to expected file path.
- `fileContains`: mapping of generated file path to required substrings.
- `fileNotContains`: mapping of generated file path to forbidden substrings.

The manifest schema must reject:

- unknown top-level fields;
- unknown `expect` fields;
- any `render`, `hidden`, docs label, page reference, or documentation-specific
  field;
- empty IDs, titles, descriptions, commands, or covers;
- invalid lowercase kebab case IDs;
- duplicate case IDs;
- duplicate entries inside one case's `covers`;
- bare FR IDs inside `covers`;
- malformed acceptance-criterion refs inside `covers`;
- absolute paths;
- `..` path segments;
- backslash path separators;
- non-string command items;
- setting both `stdout` and `stdoutFile`;
- setting both `stderr` and `stderrFile`;
- missing stdout expectation;
- missing stderr expectation.

Case failure diagnostics must include the case ID and the case's `covers` refs
so a failed e2e run points back to the affected acceptance criteria.

## Traceability Behavior

Add validation that reads both `requirements/functional-requirements.yml` and
all `test/e2e/cases/*/case.yml` files.

The validation must fail when:

- a requirement ID is duplicated;
- a requirement has an invalid ID format;
- a requirement has an invalid status;
- a requirement is `removed` and lacks `removedReason`;
- a requirement is `deferred` and lacks a `coverage` explanation;
- an acceptance criterion has an invalid ID format;
- an acceptance criterion ID is duplicated within its parent requirement;
- an acceptance criterion has `status: removed` and lacks `removedReason`;
- an `active` requirement has a non-removed acceptance criterion with no
  covering e2e case;
- an e2e case references an FR ID that does not exist;
- an e2e case references an AC ID that does not exist on the referenced FR;
- an e2e case references a `removed` requirement;
- an e2e case references a removed acceptance criterion;
- an e2e case's `covers` list contains duplicate refs.

The validation may allow an e2e case to cover multiple acceptance criteria when
one CLI invocation genuinely proves several observable promises. It should not
require one case per acceptance criterion.

The validation should produce error messages that name the affected requirement
ID, acceptance criterion ref, or case ID so AI agents can repair the right
record.

## E2E Execution Behavior

Add a Vitest e2e suite under `test/e2e/`.

The runner must:

1. Load and validate the functional requirements registry.
2. Load and validate every case manifest.
3. Validate acceptance-criterion traceability.
4. For each case, create an isolated temp directory.
5. Copy the case's `project/` fixture into that temp directory.
6. Resolve `cwd` inside the temp project, creating it when needed.
7. Execute the real CLI entry point with Bun:

   ```text
   bun <repoRoot>/src/cli/index.ts ...case.command
   ```

8. Compare actual exit code, stdout, stderr, generated files, required
   substrings, and forbidden substrings against the manifest.
9. Clean up the temp directory after execution.

The runner must support these test-harness placeholders in expected stdout,
stderr, expected files, `fileContains`, and `fileNotContains` assertions:

```text
{{projectRoot}}
{{cwd}}
{{version}}
```

`{{projectRoot}}` expands to the real temp project root. `{{cwd}}` expands to
the resolved case working directory. `{{version}}` expands to the repo's current
`package.json` version.

These placeholders are unrelated to Skillrouter template interpolation. They
must not be expanded in template fixtures, CLI arguments, requirement records,
or case metadata.

The migrated version case must assert exact stdout with:

```yaml
stdout: "{{version}} (dev)\n"
```

Do not add a general regex matcher or substring matcher just for version output.

## Project Scripts And Verification

Add or update package scripts:

```json
{
  "test": "vitest run",
  "test:e2e": "vitest run test/e2e"
}
```

Do not configure Vitest to exclude `test/e2e` from the default test run. The
default `bun run test` must run the whole test tree, including e2e. `test:e2e`
is a convenience subset for focused work.

Remove package scripts that exist only for the v2 docs site:

```json
{
  "docs:dev": "...",
  "docs:check": "...",
  "docs:build": "..."
}
```

Remove `vitepress` from `devDependencies` if it is no longer used anywhere.

The normal full verification set after this change should be:

```bash
bun run check
bun run typecheck
bun run test
bun run test:e2e
```

`bun run test:e2e` is expected to repeat the e2e portion that already ran during
`bun run test`. That redundancy is accepted for clarity and focused debugging.

## Validator Test Requirements

Unit-test the validator functions directly for rejection behavior. Do not add a
parallel `cases-bad/` or negative fixture directory tree.

The unit tests must cover the schema and traceability rejection rules listed in
this spec, including:

- invalid requirement IDs;
- duplicate requirement IDs;
- invalid requirement status;
- removed requirement without `removedReason`;
- deferred requirement without `coverage`;
- invalid AC IDs;
- duplicate AC IDs within a requirement;
- removed AC without `removedReason`;
- active AC with no covering case;
- case `covers` pointing to a missing FR;
- case `covers` pointing to a missing AC;
- case `covers` pointing to a removed FR or AC;
- bare FR refs in `covers`;
- duplicate cover refs in a case;
- unsafe paths;
- malformed case IDs;
- mutually exclusive stdout/stderr fields.

Also add one happy-path integration test that loads the real
`requirements/functional-requirements.yml` and `test/e2e/cases/` tree and
asserts that it validates cleanly.

The e2e execution test should then run every real case. A reviewer should be
able to intentionally break one expected output and see the e2e suite fail with
the case ID and cover refs.

## Constraints

Use the repo's existing stack:

- Bun for package scripts and CLI execution.
- Vitest for tests.
- TypeScript for harness code.
- YAML for requirement and case manifests, using the existing `yaml`
  dependency if suitable.

Keep this change focused on boundaries, naming, and traceability. Do not
introduce a docs generator, Gherkin, Cucumber, a database, a web UI, or a
requirements-management service.

The implementation must preserve Skillrouter's current CLI contract unless an
e2e migration reveals a real mismatch with the existing v1 spec. In that case,
the implementer should fix the bug or stop and surface the contradiction rather
than silently changing expected outputs.

The e2e cases must be deterministic and isolated. They must not depend on the
developer's current working directory, global configuration, network access, or
existing files outside the repo and temp directories.

The registry and case files are intended to be edited by AI agents. Prefer
simple schemas, clear error messages, stable IDs, and file names that make
failure output actionable.

## Unresolved Questions

The first implementation can proceed without resolving these:

- Whether future final-user documentation should consume the requirements and
  e2e cases directly or use a separate curated docs source.
- Whether the project should later add `NFR` or `BR` records to the same
  registry format.
- Whether removed requirements and removed acceptance criteria should remain
  forever or move to an archive after a period of time.
- Whether future non-executable acceptance criteria need a dedicated lifecycle
  beyond the current `deferred` requirement status and removed-AC history
  support.

For v4, use root `requirements/functional-requirements.yml`, require e2e
coverage for every non-removed AC under every `active` requirement, and keep
removed requirements and removed ACs in the registry.

## Acceptance Guidance

A reviewer should consider the implementation complete when all of the
following are true:

- `requirements/functional-requirements.yml` exists and contains the initial
  functional requirement records listed in this spec.
- Requirement IDs use `<AREA>-FR-<NNNN>` and pass
  `^[A-Z]+-FR-\d{4}$`.
- Acceptance criteria are records, not strings, and use `AC-NNNN` IDs scoped to
  their parent requirement.
- E2E cases live under `test/e2e/cases/`, not `docs/examples/`.
- E2E case manifests use `case.yml`.
- E2E case IDs are lowercase kebab slugs and do not embed FR IDs.
- E2E case manifests use `covers`, and every `covers` entry is a composite
  `<FR-ID>.AC-NNNN` ref.
- E2E case manifests contain no `requirementIds`, `render`, `hidden`, or
  docs-site fields.
- The current executable behavior coverage from v2 has been migrated into e2e
  cases.
- The e2e runner executes cases through the real CLI entry point from isolated
  temp project fixtures.
- Expected output placeholder expansion supports `{{projectRoot}}`, `{{cwd}}`,
  and `{{version}}`.
- The version case asserts exact stdout as `"{{version}} (dev)\n"`.
- Validator unit tests cover schema and traceability rejection rules without
  negative fixture directories.
- One happy-path integration test validates the real registry and case tree.
- Traceability validation fails for uncovered active ACs, invalid refs, bare FR
  refs, duplicate IDs, duplicate cover refs, removed FR refs, removed AC refs,
  and invalid manifest schemas.
- `docs/site/`, `docs/examples/`, and `test/docs/` are removed unless a
  concrete non-site dependency remains and is documented.
- `package.json` no longer exposes `docs:dev`, `docs:check`, or `docs:build`
  for the removed site.
- `vitepress` is removed from `devDependencies` if unused.
- `AGENTS.md` accurately describes the functional requirements and e2e case
  workflow.
- `README.md` does not advertise removed docs-site commands or directories.
- These commands exit 0:

  ```bash
  bun run check
  bun run typecheck
  bun run test
  bun run test:e2e
  ```

- A reviewer can intentionally break a case expectation and see the e2e suite
  fail with the case ID and `covers` refs.
- A reviewer can intentionally add an active acceptance criterion with no case
  and see the traceability suite fail with the composite AC ref.
