**Why:** Reviewers flagged three latent robustness gaps that are acceptable for v1 but worth revisiting; capturing them so the conscious "fine for now" decisions are recoverable.

1. **Frontmatter `name`/`description` are not sanitized for YAML-significant characters.** `src/compiler/schema.ts` only enforces non-empty trimmed strings — it does not reject newlines, a leading `:` / `-` / `#`, etc. A `description` containing a newline or YAML-significant leading char would produce structurally broken frontmatter both in `generate`'s emitted router `SKILL.md` (`buildRouterSkillContent` interpolates the raw string) and in any `run` output that echoes it. This is a schema-layer concern, not a `generate`-layer one. Consider whether the schema should constrain these, or whether the generator should quote/escape them.

2. **`readIncludeFile` reports every read error as "not found."** `src/compiler/includes.ts` wraps the `readFile` in a catch-all that always throws `include_not_found`. A permission error (EACCES), a directory at the path (EISDIR), or a symlink loop (ELOOP) would all surface as "was not found," which can misdirect debugging. The v1 spec's error contract is intentionally concise (only an "include not found" category), so this is acceptable for now; distinguishing errno codes would mean defining new error categories.

3. **Include containment is lexical, not symlink-resolved.** `resolveIncludePath` checks project-root containment with `path.relative` on the lexically-normalized path; it does not resolve symlinks before the boundary check, so a symlink inside the project can point outside the root. This is already noted as an explicit Unresolved Question in the v1 spec ("Whether include containment should resolve symlinks before checking the project-root boundary"); recorded here for visibility alongside the other gaps.
