# Skillrouter V1 Spec

## Intended Outcome

Skillrouter v1 is a CLI for deterministic AI-agent skill specialization. When a
user invokes a tiny agent-facing router skill, the agent runs Skillrouter with
explicit arguments and receives only the final Markdown instructions relevant to
that invocation.

V1 produces two user-facing capabilities:

- `skillrouter run <skill> [input flags...]` reads
  `.skillrouter/<skill>/SKILL.template.md`, validates the provided input flags,
  evaluates the supported template syntax, resolves selected includes,
  interpolates declared inputs, and prints clean Markdown to stdout.
- `skillrouter generate <skill> --out <path> [--force]` writes a minimal
  agent-facing `SKILL.md` router file at an explicit destination, so users do not
  have to hand-write the small wrapper skill.

Successful `run` output is intended for an AI agent to follow directly. Failed
commands print a concise human-readable error that the agent can relay to the
user and then stop.

skill locations, such as `.claude/skills/` or `.agents/skills/`, remain outside
the core rendering contract. V1 includes a narrow `generate` command only because
creating the tiny router file is useful onboarding, and it remains explicit and
agent-agnostic rather than auto-detecting an ecosystem-specific destination (per

## Template Authoring Contract

A skill source lives at `.skillrouter/<skill>/SKILL.template.md`. The root
template must have YAML frontmatter containing at least `name`, `description`,
and, when the skill accepts inputs, an `inputs` object.

Example frontmatter shape:

```yaml
---
name: analyze-code
description: Analyze code for bugs, security issues, and code quality problems
inputs:
  language:
    type: enum
    values: [typescript, python]
    required: true
  target-file:
    type: string
    required: false
  dry-run:
    type: boolean
    required: false
---
```

Every input name must be a lowercase ASCII kebab-case name matching
`^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$`. The same spelling is used as the
frontmatter key, the `run` input flag name, the condition input reference name,
and the direct interpolation name. For example, an input declared as
`target-file` is passed as `--target-file=value`, referenced in conditions as
`${target-file}`, and interpolated in Markdown as `{{target-file}}`.

Every input must use a uniform `type` field. Enum inputs must provide a
non-empty `values` array. Every input must explicitly declare `required: true`
or `required: false`; there is no implicit requiredness default (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P26 and P27, and
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P1).

The template body is Markdown plus Skillrouter directives. The renderer outputs
ordinary Markdown and erases directive syntax from the final output (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P32).

Conditional blocks are container directives. They use the standard `:::`
container fence by default, and may use three or more colons following
`remark-directive` syntax. When container directives are nested, an outer
container must use a longer colon fence than any nested container it contains;
for example, use `::::` outside and `:::` inside. This prevents an inner closing
fence from closing the outer directive early (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P5).

Non-nested conditional blocks use sibling block directives with strict
adjacency:

```md
:::if{condition="${language} == 'typescript'"}
Read the TypeScript rules.
:::

:::else-if{condition="${language} == 'python'"}
Read the Python rules.
:::

:::else
Ask the user for a supported language.
:::
```

`else-if` and `else` must immediately follow the previous branch except for
blank lines. Any non-blank content between branches ends the conditional group,
and a later `else-if` or `else` is a template error (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P7).

`if` and `else-if` require a `condition` attribute. Include directives do not
accept `condition` or `when`; conditional includes must be wrapped in an `if`
block so there is only one conditional mechanism (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P8).

Includes are leaf directives and use an explicit `path` attribute:

```md
::include{path="fragments/typescript.md"}

::include-raw{path="examples/raw-prompt.md"}
```

`include` reads the target and processes Skillrouter directives and direct input
interpolation inside that file with the current input context. Included files
are fragments in v1: they do not declare additional input schemas, and
frontmatter-like content at the start of an included file is rendered as
ordinary Markdown content rather than parsed as metadata. `include-raw` reads
the target and inserts its contents literally, without evaluating Skillrouter
directives or interpolation inside that file (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P6 and P9, and
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-spec-findings-review-discussion.md`
P3 and P5).

Direct input interpolation uses `{{input-name}}` only:

```md
Analyze `{{target-file}}` with language `{{language}}`.
```

Interpolation may reference declared inputs only. It does not support
expressions, helpers, nested paths, function calls, or formatting filters. If a
template interpolates an optional input that was not provided, rendering fails
instead of silently inserting empty text (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P16 and P17, and
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P1).

## Command Behavior

`skillrouter run <skill> [input flags...]` performs this observable sequence:

1. Discover the project root from `cwd`.
2. Validate that `<skill>` matches `^[a-z0-9][a-z0-9-]*$`.
3. Load `.skillrouter/<skill>/SKILL.template.md`.
4. Parse frontmatter, Markdown, directives, include directives, and
   interpolation references.
5. Parse generic command syntax and preserve raw flag shape for schema-aware
   validation.
6. Validate all skill-specific inputs and template semantics in the compiler.
7. Resolve selected includes, detect include cycles, evaluate conditions,
   interpolate declared inputs, and render clean Markdown.
8. Print rendered Markdown to stdout and exit `0`.

The CLI parser owns only generic command syntax, such as recognizing the command
shape, malformed flags, duplicate raw flag appearances, and whether a flag is
written as `--key`, `--key=value`, or another unsupported form. The compiler owns
all skill-specific validation, including input-name grammar, required inputs,
enum membership, declared input names, condition input references,
interpolation references, and template errors. This keeps the CLI a thin shell
around testable compiler logic (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P12, and
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P1 and P3).

Input flag rules:

- Unknown input flags are validation errors (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P28).
- Duplicate input flags are rejected for all types (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P29).
- Boolean inputs accept `--flag`, `--flag=true`, and `--flag=false`; duplicate
  or conflicting boolean values are rejected, and `--no-flag` is not a v1 form
  (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P14).
- String and enum inputs require `--name=value`. Empty values are invalid unless
  a future version explicitly defines empty-string semantics.
- Missing required inputs are validation errors. Missing optional inputs are
  falsey in conditions and invalid when interpolated directly.

Condition expressions use a small Skillrouter-owned expression grammar. Declared
inputs are referenced only as `${input-name}`; bare identifiers are not part of
the v1 condition language. The `input-name` inside `${...}` must match a
declared input and the input-name grammar above.

Condition expressions support:

- input references: `${target-file}`
- boolean literals: `true` and `false`
- string literals: single-quoted or double-quoted
- number literals: base-10 integers or decimals
- prefix `!`
- equality operators: `==` and `!=`
- boolean operators: `&&` and `||`
- parentheses

String literals may use a backslash to escape the matching quote delimiter and
backslash itself, such as `'it\'s'`, `"say \"hi\""`, and `'a\\b'`. Other
backslash escapes are condition parse errors. Number literals match
`-?(?:0|[1-9][0-9]*)(?:\.[0-9]+)?`; exponents, `NaN`, `Infinity`, numeric
separators, and hexadecimal or other non-decimal forms are not supported.

Operator precedence from highest to lowest is prefix `!`, `==` and `!=`, `&&`,
then `||`. Binary operators are left-associative, so `${a} || ${b} && ${c}`
means `${a} || (${b} && ${c})`.

Equality uses Skillrouter's typed semantics, not JavaScript coercion. Values are
equal only when they have the same runtime type and value; for example,
`${dry-run} == true` can be true for a boolean input, but `${dry-run} == 'true'`
is false. Conditions do not support `in`, comparison operators, property access,
function calls, file reads, bare JavaScript identifiers, or JavaScript/TypeScript
evaluation (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P10, and
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P1 and P2).

Truthiness for v1 conditions is:

- Missing optional input: false.
- Boolean input: its boolean value.
- String or enum input: true when provided with a non-empty value.

Include path rules:

- Paths are resolved relative to the file containing the directive.
- `../` is allowed only if the lexically normalized path remains inside the
  discovered project root.
- Absolute paths and `~` are rejected.
- `.env` and `.env.*` files are rejected even when they are inside the project
  root.
- `include` tracks the include stack and errors with the include chain when a
  recursive include cycle is detected (per
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
  P18 and P19, and
  `/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-spec-findings-review-discussion.md`
  P4).

V1 include containment does not resolve symlinks before checking the project-root
boundary. A symlink inside the project can therefore point outside the project
root. This is accepted for v1 because symlinked content is often intentional by
the user, but symlink-aware containment remains an explicit follow-up question.

`skillrouter generate <skill> --out <path> [--force]` performs this observable
sequence:

1. Discover the project root from `cwd`.
2. Load `.skillrouter/<skill>/SKILL.template.md` and run full static template
   validation through the shared compiler validation pipeline.
3. Read `name` and `description` from template frontmatter.
4. Refuse to write if `--out` is missing.
5. Resolve relative `--out` paths from the invoking `cwd`; absolute `--out`
   paths are allowed, including destinations outside the discovered project
   root.
6. Create missing parent directories for the output path.
7. Refuse to overwrite an existing file unless `--force` is provided.
8. Write a minimal router skill file to the explicit output path.

Full static validation for `generate` includes frontmatter, schema, input-name
grammar, body directives, condition syntax, declared condition input references,
interpolation references, statically checkable include paths, missing includes,
and include cycles. It does not evaluate branch truth because no invocation
inputs are provided. `generate` and `run` must share compiler validation modules
for these checks; the behavior must not be duplicated independently in separate
command handlers (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P3).

`generate` never guesses the correct agent-specific destination. The output path
is mandatory because v1 stays agent-agnostic and avoids surprising writes into
`.claude/`, `.agents/`, or any other ecosystem-specific folder (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P2). Existing files are protected by default and can only be replaced with
`--force` (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P22). Relative `--out` paths behave like ordinary filesystem CLI arguments and
are resolved from the invoking `cwd`; users running from subdirectories should
use `..` or an absolute path when targeting a project-root or external
destination. Missing parent directories are created automatically (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-spec-findings-review-discussion.md`
P1 and P2).

The generated router skill content must contain only frontmatter `name` and
`description`, the command, and failure handling:

````md
---
name: <template name>
description: <template description>
---

Run this command and follow its output exactly:

```bash
skillrouter run <skill> $ARGUMENTS
```

If the command exits non-zero, report the exact error output to the user and
stop.
````

This minimal content keeps the agent-facing file short while making failure
reporting actionable (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P21 and P31).

## Error And Output Contract

Successful `run` output goes to stdout as Markdown only. V1 does not expose
JSON, verbose traces, provenance comments, or generated wrappers in rendered
Markdown (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P20 and P32).

The root template frontmatter is metadata and is never emitted by `run`.
Directive syntax and unselected directive content are also omitted. After those
removals and selected includes/interpolations, v1 output is source-preserving and
byte-stable: Skillrouter preserves the leftover authored Markdown bytes and does
not apply additional whitespace normalization to blank lines created by removed
directives. If skipped branches leave extra blank lines, those blank lines remain
part of the expected stdout contract (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P4).

Failures print a single-line `Error: ...` message to stderr and exit non-zero.
Stdout must remain empty on failure so router skills can safely treat stdout as
the rendered instruction channel. Error categories include invalid command
syntax, missing project root, invalid skill name, missing skill, invalid
frontmatter, malformed schema, invalid input name, missing required input,
unknown input flag, duplicate input flag, invalid enum value, condition parse
error, condition reference to an undeclared input, invalid interpolation
reference, absent optional interpolation, include not found, include outside
project root, rejected absolute or home-relative include, rejected `.env*`
include, include cycle, missing output path, generate static validation failure,
and existing output file without `--force` (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P30 and
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-spec-findings-review-discussion.md`
P1, P2, and P6).

Diagnostic text should be concise and safe for an AI agent to quote to the user.
Detailed tracing is a future feature, not part of the v1 public output contract.

## Implementation Constraints

Use Bun as the runtime, package manager, and bundler for the project, but keep
runtime code portable by avoiding Bun-specific runtime APIs and Bun's test
runner. Tests should not depend on Bun-only test semantics so the project can
move to Node or another runtime later.

The CLI must remain a thin shell around separately testable modules. At minimum,
the implementation should isolate these concerns:

- command parsing and raw flag collection
- project root discovery
- skill name validation and template path resolution
- frontmatter parsing and schema validation
- Markdown/directive parsing
- condition expression parsing and evaluation
- include resolution and cycle detection
- input validation and coercion
- interpolation
- Markdown rendering
- router skill generation
- structured errors mapped to CLI stderr messages

The Markdown parser foundation should use the `unified` / `remark` ecosystem,
including `remark-directive` or equivalent remark-compatible directive support,
behind internal parser/rendering modules. The user-facing contract is
Skillrouter's documented syntax, not remark internals (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-agent-skill-router-cli-design-discussion.md`
P4).

Template evaluation must be deterministic and side-effect free except for
explicit include file reads. Conditions and interpolation must not execute
JavaScript, TypeScript, shell commands, user-defined functions, or arbitrary file
access. Condition strings must be parsed into Skillrouter's own validated AST or
an equivalent safe AST and evaluated by Skillrouter; they must not be delegated
to `eval`, `new Function`, the TypeScript compiler, Node's `vm`, or any other
general-purpose JavaScript execution mechanism (per
`/Users/jacopo/Developer/projects/personal/tools/skillrouter/docs/threads/260526113604Z-agent-skill-router-cli/discussions/2026-05-26-v1-input-condition-ambiguity-review-discussion.md`
P2).

## Unresolved Questions

These questions do not block v1 because the spec either excludes them or defines
a conservative first behavior:

- Whether later versions need `path`, array, object, number, or default input
  semantics once real skill templates show pressure for them.
- Whether root-relative include syntax, include aliases, or a `--root` option
  are needed for larger repositories.
- Whether include containment should resolve symlinks before checking the
  project-root boundary, or whether v1's lexical containment should remain the
  long-term behavior.
- Whether authoring ergonomics require a future preprocessor or alternate
  directive syntax if remark directive markers prove awkward.
- Whether JSON, verbose traces, or other debug output formats should be added
  after the Markdown stdout contract is stable.
- Whether nested skill groups or lifecycle commands for install, update, sync,
  and uninstall are useful enough to justify broadening the v1 boundaries.

## Acceptance Guidance

An implementation satisfies this spec when reviewers can verify the behavior
through focused unit tests and CLI-level integration tests:

- `skillrouter run <skill>` discovers the nearest `.skillrouter/` project root
  from nested directories, accepts only skill names matching
  `^[a-z0-9][a-z0-9-]*$`, reads `.skillrouter/<skill>/SKILL.template.md`, and
  prints rendered Markdown to stdout on success.
- Templates with valid kebab-case `string`, `boolean`, and `enum` frontmatter
  inputs render selected `if` / `else-if` / `else` branches, erase directives
  from output, recursively process leaf `::include{path="..."}`, insert literal
  leaf `::include-raw{path="..."}`, render frontmatter-like content in included
  fragments as ordinary Markdown content, and interpolate direct input
  placeholders such as `{{target-file}}`.
- Directive syntax tests cover non-nested conditional containers using `:::`,
  nested conditional containers using a longer outer fence such as `::::` around
  inner `:::` containers, and rejection or diagnostic coverage for malformed
  nesting where equal-length fences close the wrong container.
- Condition tests cover `${input-name}` references, rejection of bare
  identifiers, single-quoted and double-quoted strings with the supported escape
  forms, integer and decimal number literals, typed `==` and `!=`, prefix `!`,
  parentheses, and precedence/associativity for `!`, equality, `&&`, and `||`.
- Rendered Markdown tests compare stdout byte-for-byte, including source
  preservation after root frontmatter removal, directive removal, skipped
  branches, selected includes, `include-raw`, and interpolation. V1 tests should
  not normalize blank lines introduced by directive removal.
- Validation fails with stderr `Error: ...`, non-zero exit, and empty stdout for
  missing required inputs, unknown flags, duplicate flags, invalid enum values,
  malformed schemas, invalid input names, unsupported input types, unsupported
  condition operators, undeclared condition input references, undeclared
  interpolation placeholders, and absent optional interpolation.
- Boolean flags accept `--flag`, `--flag=true`, and `--flag=false`; duplicate
  boolean flags and `--no-flag` are rejected.
- Include tests cover file-relative resolution, lexical normalization of `../`
  within the project root, rejection of lexically outside-project paths,
  rejection of absolute and `~` paths, rejection of `.env` and `.env.*`,
  include-not-found errors, and include cycle diagnostics that include the
  chain.
- `skillrouter generate <skill> --out <path>` writes the minimal router skill
  content with template `name` and `description`, the exact
  `skillrouter run <skill> $ARGUMENTS` command, and the exact non-zero error
  instruction. It resolves relative output paths from the invoking `cwd`, allows
  output outside the discovered project root, creates missing parent
  directories, refuses existing files without `--force`, and overwrites them with
  `--force`. It uses the shared compiler validation pipeline to reject
  statically invalid template bodies before writing output.
- The implementation exposes independently testable modules for parsing,
  validation, condition evaluation, include resolution, interpolation, rendering,
  and generation rather than embedding those behaviors only in CLI command
  handlers.
- Project verification passes with `bun run check`, `bun run typecheck`, and
  `bun test`.
